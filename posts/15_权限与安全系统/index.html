权限与安全系统 — Claude Code 源码深度分析

一句话理解

Claude 每次要执行一个操作（运行命令、修改文件、访问网络），都要先过一道”安检”。安检有七道关卡，从”你被明确禁止”到”你需要用户点头”，任何一关拦住就不执行。用户可以设置”永远允许”、”永远拒绝”、”每次问我”三种规则，还可以开启 AI 分类器让它自动判断安不安全。

白话版运行流程

整体分两个维度

权限系统有两个维度：权限模式（全局开关）和权限规则（细粒度控制）。

用户执行操作 → 权限检查
                    │
        ┌───────────┴───────────┐
        ▼                       ▼
   【权限规则】              【权限模式】
   七道关卡逐一检查          全局行为开关
   deny/ask/allow            default/plan/acceptEdits
                             bypassPermissions/dontAsk/bubble/auto

七种权限模式 — “全局开关”

类比：就像汽车的驾驶模式，不同模式下同样的操作会有不同的处理方式。

模式 通俗理解 适用场景
default 正常模式，危险操作弹确认框 日常使用
plan 只规划不执行，所有写操作都要确认 审查 Claude 的计划
acceptEdits 自动接受文件编辑，其他操作还是问 信任 Claude 改代码
bypassPermissions 跳过所有权限检查（危险！） 自动化脚本、CI/CD
dontAsk 不弹框，未预批准的操作直接拒绝 无头环境、后台任务
bubble 权限提示冒泡到父级终端 子 Agent 继承父级权限确认
auto AI 分类器自动判断，安全的自动放行 高级用户，减少打扰

七道安检关卡 — “细粒度控制”

类比：机场安检，每道关卡检查不同的东西，任何一关拦住就不让过。

操作请求进来
    │
    ▼
关卡1a: 整体拒绝规则？ ──→ 有 → 直接拒绝（最高优先级）
    │
    ▼
关卡1b: 整体询问规则？ ──→ 有 → 弹确认框
    │
    ▼
关卡1c: 工具自定义检查？ ──→ Bash分析命令/文件工具检查路径...
    │
    ▼
关卡1d: 工具返回拒绝？ ──→ 是 → 直接拒绝
    │
    ▼
关卡1e: 必须用户交互？ ──→ 是 → 弹确认框（bypass也不能跳过）
    │
    ▼
关卡1f: 内容级询问规则？ ──→ 有 → 弹确认框（bypass也不能跳过）
    │
    ▼
关卡1g: 安全路径检查？ ──→ .git/.claude/shell配置 → 弹确认框（bypass免疫）
    │
    ▼
关卡2a: bypassPermissions模式？ ──→ 是 → 直接放行
    │
    ▼
关卡2b: 整体允许规则？ ──→ 有 → 直接放行
    │
    ▼
关卡3: passthrough → ask，进入模式处理
    │
    ├─ dontAsk模式: ask → 直接拒绝
    ├─ auto模式: AI分类器判断
    ├─ shouldAvoidPermissionPrompts: 先跑Hook，没决策则拒绝
    └─ 其他: 弹确认框给用户

权限规则语法 — “怎么写规则”

类比：就像写防火墙规则，可以精确到”允许 npm test，但不允许 npm publish”。

规则格式: "工具名" 或 "工具名(内容)"

例子:
  Bash              → 允许/拒绝所有 Bash 命令
  Bash(npm test)    → 允许/拒绝 npm test 命令
  Bash(prefix:git*) → 允许/拒绝所有 git 开头的命令
  mcp__server1      → 允许/拒绝某个 MCP 服务器的所有工具
  Agent(Explore)    → 允许/拒绝 Explore 类型的子 Agent

规则来源优先级 — “谁说了算”

规则可以来自多个地方，优先级从高到低：

policySettings（企业管理员）
    ↓
flagSettings（功能开关）
    ↓
userSettings（~/.claude/settings.json）
    ↓
projectSettings（项目根目录 .claude/settings.json）
    ↓
localSettings（本地覆盖）
    ↓
cliArg（--allowed-tools 命令行参数）
    ↓
session（本次会话临时规则）
    ↓
command（/permissions 命令添加的规则）

注意：SETTING_SOURCES 数组中的顺序是 userSettings → projectSettings → localSettings → flagSettings → policySettings（后者覆盖前者）。cliArg、session、command 是仅存在于内存中的权限规则来源，不属于 settings 文件系统来源。policySettings 和 flagSettings 始终被加载（getEnabledSettingSources() 强制包含），其他来源可通过 --setting-sources 控制。

Auto 模式 — “AI 分类器自动判断”

类比：雇了一个安全顾问，每次有操作请求，顾问快速看一眼对话历史，判断这个操作是不是合理的。

运行过程：

  1. 操作请求到达，前七道关卡都没拦住（passthrough）
  2. 进入 auto 模式处理
  3. 先检查”快速路径”：
    • 这个操作在 acceptEdits 模式下会被允许吗？→ 直接放行（省去 API 调用）
    • 这个工具在安全白名单里吗？→ 直接放行
  4. 调用 classifyYoloAction() — 把对话历史 + 当前操作发给 AI 分类器
  5. 分类器返回 shouldBlock: true/false
  6. 如果不可用（API 错误）：根据 tengu_iron_gate_closed 开关决定 fail-closed（拒绝）还是 fail-open（弹框）
  7. 连续拒绝超过阈值 → 自动降级为弹框让用户决定

多 Agent 权限冒泡 — “后台 Agent 遇到权限怎么办”

类比：你的下属在后台干活，遇到需要你签字的事情，通过”收件箱”把请求发给你，你在自己的终端上确认，结果写回下属的收件箱。

Worker Agent 遇到需要权限的操作
    │
    ▼
shouldAvoidPermissionPrompts = true（后台代理不能弹框）
    │
    ▼
先跑 PermissionRequest hooks（给自动化脚本机会处理）
    │
    ├─ Hook 返回 allow → 放行
    ├─ Hook 返回 deny → 拒绝
    └─ 没有 Hook 决策 → 通过邮箱发权限请求给 Leader
                              │
                              ▼
                    Leader 的收件箱轮询器收到请求
                              │
                              ▼
                    Leader 终端弹出确认框
                              │
                              ▼
                    用户确认/拒绝 → 结果写回 Worker 邮箱
                              │
                              ▼
                    Worker 的 useSwarmPermissionPoller 每 **500ms** 轮询到响应
                              │
                              ▼
                    processMailboxPermissionResponse() 调用 onAllow/onReject 回调 → 继续执行

核心函数详解 — “它到底做了什么”

1. hasPermissionsToUseTool() — 权限决策总入口

文件: src/utils/permissions/permissions.ts

这是整个权限系统的”总闸门”。每次工具调用前，toolExecution.ts 都会调用这个函数，它负责把请求路由到正确的决策路径。

具体做了什么（逐步）:

  1. 调用内部核心逻辑 hasPermissionsToUseToolInner()，得到初步决策。

  2. 成功路径: 如果结果是 allow，在 auto 模式下重置连续拒绝计数（recordSuccess()），然后直接返回。

  3. dontAsk 模式转换: 如果结果是 ask 且当前是 dontAsk 模式，把 ask 转为 deny（无头环境不能弹框）。

  4. auto 模式处理（最复杂的路径）:

    • 非分类器可批准的 safetyCheck（如 Windows 路径绕过）→ 直接弹框或拒绝
    • requiresUserInteraction 工具 → 直接弹框
    • acceptEdits 快速路径: 模拟 acceptEdits 模式重新调用 tool.checkPermissions()，如果会被允许 → 直接放行（省去 API 调用）
    • 安全工具白名单: isAutoModeAllowlistedTool() → 直接放行
    • AI 分类器: classifyYoloAction() → 根据结果决定放行/拒绝/降级弹框
    • 拒绝限制: 连续拒绝超过阈值 → handleDenialLimitExceeded() 降级为弹框

  5. 无头代理路径 (shouldAvoidPermissionPrompts): 先跑 runPermissionRequestHooksForHeadlessAgent()，给 Hook 机会处理；没有 Hook 决策 → 自动拒绝。

2. hasPermissionsToUseToolInner() — 七道关卡核心

文件: src/utils/permissions/permissions.ts

这是真正执行七道安检关卡的函数。

具体做了什么（逐步）:

第一层：规则检查（bypassPermissions 也无法跳过 1d/1e/1f/1g）

  • 1a. 整体拒绝: getDenyRuleForTool() — 在 alwaysDenyRules 中查找整体拒绝规则（无 ruleContent 的规则，如 "Bash" 拒绝所有 Bash 调用）。支持 MCP 服务器级别拒绝（规则 "mcp__server1" 匹配该服务器所有工具）。

  • 1b. 整体询问: getAskRuleForTool() — 在 alwaysAskRules 中查找整体询问规则。特例：如果沙箱启用且 autoAllowBashIfSandboxed=true，沙箱化的 Bash 命令可以跳过这一关。

  • 1c. 工具自定义检查: tool.checkPermissions(parsedInput, context) — 每个工具实现自己的权限逻辑：

    • BashTool: 解析命令，拆解为子命令，逐个检查前缀规则
    • FileEditTool: 检查路径是否在工作目录内
    • WebFetchTool: 检查 URL 是否在允许列表

  • 1d. 工具返回拒绝: 工具的 checkPermissions 返回 deny → 直接拒绝。

  • 1e. 必须用户交互: tool.requiresUserInteraction?.() 返回 true 且结果是 ask → 弹框（即使在 bypassPermissions 模式下也不能跳过）。

  • 1f. 内容级询问规则: 工具返回 ask 且 decisionReason.type === 'rule' 且 ruleBehavior === 'ask' → 弹框（用户明确配置的 ask 规则，bypass 也不能跳过）。

  • 1g. 安全路径检查: 工具返回 ask 且 decisionReason.type === 'safetyCheck' → 弹框（.git/、.claude/、shell 配置文件等，bypass 免疫）。

第二层：模式快速放行

  • 2a. bypassPermissions 模式: 直接放行（包括 plan 模式 + isBypassPermissionsModeAvailable 的组合）。

  • 2b. 整体允许规则: toolAlwaysAllowedRule() — 工具在 alwaysAllowRules 中 → 直接放行。

第三层：转换 passthrough

  • 把 passthrough 结果转为 ask，附上权限请求消息，返回给调用方处理。

3. checkRuleBasedPermissions() — 规则检查子集

文件: src/utils/permissions/permissions.ts

这是 hasPermissionsToUseToolInner 的一个子集，只执行规则检查部分（1a/1b/1c/1d/1f/1g），不执行模式检查和 auto 分类器。

使用场景: Hook 系统中，当 Hook 返回 allow 时，仍然需要检查用户配置的 deny/ask 规则是否有冲突——Hook 的 allow 不能绕过用户明确配置的规则。

关键不变量: deny 规则覆盖 Hook 的 allow。这是整个权限系统最重要的安全保证之一。

4. permissionRuleValueFromString() — 规则语法解析

文件: src/utils/permissions/permissionRuleParser.ts

把规则字符串解析为结构化对象。

格式: "ToolName" 或 "ToolName(content)"

具体做了什么:

  1. 查找第一个未转义的 ( 字符（转义：\( 不算）。
  2. 如果没有 (，整个字符串就是工具名。
  3. 找到最后一个未转义的 ) 字符。
  4. 提取工具名和内容，对内容进行反转义（\( → (，\\ → \）。
  5. 空内容或 * 内容 → 视为整体工具规则（无 ruleContent）。
  6. 调用 normalizeLegacyToolName() 处理旧名称（如 Task → Agent）。

转义规则（防止规则内容中的括号破坏解析）:

escapeRuleContent('psycopg2.connect()') → 'psycopg2.connect\(\)'
permissionRuleValueFromString('Bash(psycopg2.connect\(\))') → { toolName: 'Bash', ruleContent: 'psycopg2.connect()' }

5. initializeToolPermissionContext() — 权限上下文初始化

文件: src/utils/permissions/permissionSetup.ts

在 Claude Code 启动时，这个函数构建初始的 ToolPermissionContext。

具体做了什么:

  1. 解析 CLI 参数: parseToolListFromCLI(allowedToolsCli) — 支持逗号分隔、空格分隔、括号内容（Bash(npm test), Edit）。

  2. 处理 base tools: 如果指定了 --base-tools，自动把不在列表里的工具加入 deny 列表。

  3. 检测 bypassPermissions 可用性: 查询 GrowthBook 门控（tengu_disable_bypass_permissions_mode）和 settings 配置，决定 isBypassPermissionsModeAvailable。

  4. 加载磁盘规则: loadAllPermissionRulesFromDisk() — 从所有启用的设置来源加载规则。如果 allowManagedPermissionRulesOnly=true，只加载 policySettings 的规则。

  5. 检测危险规则（仅 ant 用户）:

    • findOverlyBroadBashPermissions() — 检测 Bash(*) 等等价于 YOLO 模式的规则
    • findDangerousClassifierPermissions() — 检测会绕过 auto 模式分类器的规则（Bash(python:*)、Agent(*)、Tmux 等）

  6. 构建初始上下文: 合并 CLI 规则和磁盘规则，设置 additionalWorkingDirectories（处理 symlink 的 $PWD）。

  7. 处理 –add-dir: 并行验证所有额外目录，串行应用到上下文。

6. transitionPermissionMode() — 模式切换

文件: src/utils/permissions/permissionSetup.ts

所有权限模式切换都必须经过这个函数，它集中处理所有副作用。

具体做了什么:

  1. plan 模式进入: prepareContextForPlanMode() — 把当前模式存入 prePlanMode，如果用户开启了 auto+plan 联动，激活分类器并剥离危险规则。

  2. auto 模式进入: setAutoModeActive(true) + stripDangerousPermissionsForAutoMode() — 把危险的允许规则（Bash(python:*)、Agent(*)）从内存中移除，存入 strippedDangerousRules 备份。

  3. auto 模式退出: setAutoModeActive(false) + restoreDangerousPermissions() — 从 strippedDangerousRules 恢复被移除的规则，设置 needsAutoModeExitAttachment（通知 UI 显示退出提示）。

  4. plan 模式退出: setHasExitedPlanMode(true) — 标记已退出 plan 模式，用于 UI 状态管理。

关键设计: 模式切换是纯函数（返回新的 context），调用方负责把结果写入 AppState。这样可以在 setAppState(prev => ...) 内部安全地应用，避免异步竞争。

7. stripDangerousPermissionsForAutoMode() — 危险规则剥离

文件: src/utils/permissions/permissionSetup.ts

进入 auto 模式时，把会绕过分类器的危险允许规则从内存中移除。

什么是危险规则:

  • Bash(*) / Bash(python:*) / Bash(node:*) — 允许任意代码执行，分类器看不到
  • PowerShell(iex:*) / PowerShell(Start-Process:*) — PowerShell 代码执行
  • Agent(*) / Agent — 允许任意子 Agent，绕过委托攻击防护
  • Tmux（仅 ant 用户）— tmux send-keys 等价于任意 shell 执行

具体做了什么:

  1. 遍历 alwaysAllowRules 中所有规则。
  2. 调用 isDangerousClassifierPermission() 判断是否危险。
  3. 把危险规则从 alwaysAllowRules 中移除（applyPermissionUpdate + removeRules）。
  4. 把移除的规则存入 strippedDangerousRules（按来源分组）。
  5. 退出 auto 模式时，restoreDangerousPermissions() 从 strippedDangerousRules 恢复。

8. createPermissionContext() — 用户交互上下文

文件: src/hooks/toolPermission/PermissionContext.ts

当权限决策结果是 ask 时，这个上下文对象负责协调用户交互的整个流程。

具体做了什么:

  1. runHooks(): 先跑 PermissionRequest hooks，给自动化脚本机会处理。Hook 返回 allow → 调用 handleHookAllow()；返回 deny → 直接拒绝。

  2. **tryClassifier()**（仅 Bash + BASH_CLASSIFIER 特性）: 等待投机性分类器检查结果（在 PreToolUse hooks 执行期间并行运行）。

  3. pushToQueue(): 把 ToolUseConfirm 对象推入 toolUseConfirmQueue，React 渲染 <PermissionRequest> 组件。

  4. handleUserAllow(): 用户点击”允许”后：

    • 调用 persistPermissions() 持久化”永远允许”规则（写入 settings.json）
    • 记录分析事件
    • 返回 PermissionAllowDecision

  5. cancelAndAbort(): 用户点击”拒绝”或按 ESC：

    • 如果是 abort（无反馈），调用 abortController.abort() 中止整个查询
    • 如果有反馈，只返回拒绝决策，不中止（让 LLM 看到反馈后自行调整）

  6. createResolveOnce(): 原子性的 claim-and-resolve 机制，防止多个异步路径（Hook、分类器、用户、桥接）同时 resolve 同一个 Promise。

9. sendPermissionRequestViaMailbox() — 多 Agent 权限请求

文件: src/utils/swarm/permissionSync.ts

Worker Agent 遇到需要权限的操作时，通过邮箱把请求发给 Leader。

具体做了什么:

  1. getLeaderName() — 读取 team file，找到 lead 成员的名字。
  2. 构造 permission_request JSON 消息（包含 request_id、工具名、输入、权限建议等）。
  3. writeToMailbox(leaderName, message, teamName) — 写入 Leader 的收件箱文件（带文件锁）。
  4. Worker 调用 registerPermissionCallback(requestId) 注册 { onAllow, onReject } 回调，由 useSwarmPermissionPoller 每 500ms 轮询等待响应。

请求文件结构（~/.claude/teams/{team}/permissions/pending/{id}.json）:

{
  "id": "perm-1234567890-abc123",
  "workerId": "worker@team",
  "workerName": "coder",
  "teamName": "refactor-team",
  "toolName": "Bash",
  "toolUseId": "toolu_xxx",
  "description": "Run: npm publish",
  "input": { "command": "npm publish" },
  "permissionSuggestions": [],
  "status": "pending",
  "createdAt": 1234567890
}

10. resolvePermission() — Leader 处理权限请求

文件: src/utils/swarm/permissionSync.ts

Leader 在 UI 上确认/拒绝后，调用这个函数写入结果。

具体做了什么:

  1. 获取 pending 目录的文件锁。
  2. 读取 pending 请求文件。
  3. 构造 resolved 请求（加入 status、resolvedBy、resolvedAt、updatedInput、permissionUpdates）。
  4. 写入 resolved 目录（~/.claude/teams/{team}/permissions/resolved/{id}.json）。
  5. 删除 pending 目录中的请求文件。
  6. 释放文件锁。

11. useSwarmPermissionPoller — Worker 轮询权限响应

文件: src/hooks/useSwarmPermissionPoller.ts

Worker 端的轮询机制，等待 Leader 的权限响应。

具体做了什么:

  1. 维护一个 pendingCallbacks Map（requestId → { onAllow, onReject } 回调对象）。
  2. Worker 发出权限请求后，调用 registerPermissionCallback(requestId) 注册 { onAllow, onReject } 回调。
  3. 收件箱轮询器（useInboxPoller）收到 permission_response 消息后，调用 processMailboxPermissionResponse()。
  4. 根据 decision: 'approved'/'rejected' 调用对应的 onAllow 或 onReject 回调。
  5. useSwarmPermissionPoller 每 500ms 轮询 pendingCallbacks，通过 pollForResponse() 检查文件系统中的 resolved 目录，找到响应后调用回调并清理。

安全沙箱（Sandbox）

权限系统 vs 沙箱：权限系统是应用层控制（”是否执行”），沙箱是 OS 层隔离（”执行时能做什么”）。两者互补，缺一不可。

白话理解

类比：权限系统是门卫（决定谁能进门），沙箱是牢笼（进门后能活动的范围）。即使门卫放行了，牢笼仍然限制了能做的事情。

Bash 命令执行流程（沙箱启用时）

用户请求执行命令
    │
    ▼
权限系统检查（应用层）
    │ 通过
    ▼
shouldUseSandbox() 判断
    ├─ 沙箱未启用 → 直接执行（无隔离）
    ├─ dangerouslyDisableSandbox=true 且策略允许 → 直接执行
    ├─ 命令在 excludedCommands 列表 → 直接执行（用户豁免）
    └─ 其他 → wrapWithSandbox() 包装命令
                    │
                    ▼
            OS 级沙箱执行
            ├─ macOS: Seatbelt (sandbox-exec)
            └─ Linux/WSL2: bubblewrap (bwrap)
                    │
                    ▼
            受限环境中运行命令
            ├─ 文件系统: 只能读写允许的路径
            ├─ 网络: 只能访问允许的域名
            └─ 违规操作: 被 OS 拦截，记录到 ViolationStore

沙箱配置来源

沙箱配置由 convertToSandboxRuntimeConfig() 从权限规则自动派生，无需单独配置：

权限规则 自动转换为沙箱配置
FileEdit(allow:/path/*) filesystem.allowWrite: ["/path/*"]
FileEdit(deny:/secret/*) filesystem.denyWrite: ["/secret/*"]
FileRead(deny:/private/*) filesystem.denyRead: ["/private/*"]
WebFetch(allow:domain:github.com) network.allowedDomains: ["github.com"]
WebFetch(deny:domain:evil.com) network.deniedDomains: ["evil.com"]

也可以通过 settings.json 的 sandbox.filesystem.* 和 sandbox.network.* 直接配置。

沙箱启用条件

isSandboxingEnabled() 需要同时满足：

1. isSupportedPlatform()
   ├─ macOS ✓
   ├─ Linux ✓
   ├─ WSL2 ✓
   └─ WSL1 ✗（不支持）

2. checkDependencies().errors.length === 0
   ├─ macOS: sandbox-exec（系统自带）
   └─ Linux: bubblewrap (bwrap) + socat

3. isPlatformInEnabledList()
   └─ settings.sandbox.enabledPlatforms（未设置则全部允许）

4. settings.sandbox.enabled === true
   └─ 默认 false，需要用户显式开启

防逃逸设计

沙箱内强制保护以下路径，防止提权或逃逸：

// 永远加入 denyWrite（沙箱内不可修改）：
denyWrite.push(...settingsPaths)          // 所有 settings.json 文件
denyWrite.push(getManagedSettingsDropInDir())  // 企业管理设置目录
denyWrite.push(resolve(cwd, '.claude', 'skills'))  // 技能目录（防植入恶意技能）

// Git bare repo 逃逸防护（#29316）：
// 攻击者可在 cwd 植入 HEAD + objects/ + refs/ 伪造 bare repo，
// 触发 git 的 core.fsmonitor 钩子在沙箱外执行任意代码。
// 防护策略：
//   - 文件已存在 → denyWrite（只读挂载，无法修改）
//   - 文件不存在 → 命令执行后 scrubBareGitRepoFiles() 清除植入文件

关键配置项

配置项 默认值 说明
sandbox.enabled false 是否启用沙箱
sandbox.autoAllowBashIfSandboxed true 沙箱启用时自动允许 Bash（减少弹框）
sandbox.allowUnsandboxedCommands true 是否允许 dangerouslyDisableSandbox 绕过
sandbox.failIfUnavailable false 沙箱不可用时是否拒绝执行
sandbox.excludedCommands [] 豁免沙箱的命令列表（非安全边界，仅便利功能）
sandbox.network.allowManagedDomainsOnly false 企业策略：只允许管理员配置的域名
sandbox.filesystem.allowManagedReadPathsOnly false 企业策略：只允许管理员配置的读取路径

沙箱与 autoAllowBashIfSandboxed

当沙箱启用且 autoAllowBashIfSandboxed=true 时，Bash 命令的权限检查流程发生变化：

Bash 命令 + 沙箱启用 + autoAllowBashIfSandboxed=true
    │
    ▼
关卡1b: 有整体 ask 规则？
    ├─ 有，但 canSandboxAutoAllow=true → 跳过 ask 规则，继续检查
    └─ 有，但 canSandboxAutoAllow=false（excluded/dangerouslyDisable）→ 弹确认框
    │
    ▼
关卡1c: bashPermissions 检查命令级规则
    └─ 沙箱已限制危害范围，可以更宽松地自动允许

启用失败可见性（#34044 修复）

用户显式开启沙箱（sandbox.enabled: true）但依赖缺失时，getSandboxUnavailableReason() 返回明确错误信息，在启动时展示给用户：

"sandbox.enabled is set but dependencies are missing: bwrap not found
 · install missing tools (e.g. apt install bubblewrap socat) or run /sandbox for details"

为什么重要：静默失败是安全隐患。用户配置了 allowedDomains 期望网络访问受限，但沙箱实际上没有运行，所有网络请求都能通过。

设计模式总结

模式 实现细节 为什么这样设计
Fail-closed 默认 isConcurrencySafe=false、isReadOnly=false。新工具忘记声明属性时走最保守路径。passthrough 最终转为 ask 而非 allow。 安全系统的核心原则：不确定时拒绝，而非放行。
分层防御 七道关卡逐一检查，任一层拦截就短路。规则检查层（1a-1g）即使在 bypass 模式下也部分生效。 单一防线被绕过时，其他防线仍然有效。
规则不变量 deny 规则覆盖 Hook 的 allow。checkRuleBasedPermissions() 在 Hook allow 后仍然运行。 用户明确配置的规则不能被自动化脚本绕过。
bypass 免疫 requiresUserInteraction、内容级 ask 规则、safetyCheck 在 bypass 模式下仍然生效。 某些操作（如修改 shell 配置）无论如何都需要用户确认。
危险规则剥离 进入 auto 模式时，把会绕过分类器的规则移入 strippedDangerousRules；退出时恢复。 防止用户的 Bash(python:*) 规则在 auto 模式下绕过 AI 安全检查。
投机性分类器 Bash 工具在权限检查前就 startSpeculativeClassifierCheck()，与 PreToolUse hooks 并行运行。 分类器延迟被 hooks 执行时间掩盖，用户等待时间缩短。
原子性 claim createResolveOnce() 的 claim() 方法原子性地标记”已被认领”，防止多个异步路径（Hook、分类器、用户、桥接）同时 resolve 同一个 Promise。 权限确认是一次性的，多个并发路径竞争时只有一个赢家。
邮箱权限冒泡 Worker 通过文件系统邮箱把权限请求发给 Leader，Leader 在自己的 UI 上确认，结果写回 Worker 邮箱。 后台 Agent 没有 UI，但权限决策必须由人类做出。邮箱机制把决策路由到有 UI 的 Leader。
熔断器模式 bypassPermissionsKillswitch.ts 的运行时门控，可以通过 GrowthBook 远程禁用危险模式。 紧急情况下可以远程关闭危险功能，无需发布新版本。
模式切换纯函数 transitionPermissionMode() 返回新的 context，不直接修改 AppState。调用方在 setAppState(prev => ...) 内部应用。 避免异步竞争：模式切换期间如果用户又切换了模式，不会被覆盖。
沙箱配置自动派生 convertToSandboxRuntimeConfig() 直接从权限规则生成沙箱配置：FileEdit(allow:path) → allowWrite，WebFetch(deny:domain:xxx) → deniedDomains。 用户只需配置权限规则，沙箱自动跟进，两套系统不会出现不一致。
沙箱防逃逸 settings.json 和 .claude/skills/ 强制加入 denyWrite；git bare repo 文件已存在则只读挂载，不存在则命令后 scrubBareGitRepoFiles() 清除。 防止沙箱内的代码通过修改配置提权，或通过植入 git 文件触发沙箱外 git 钩子逃逸。
沙箱动态更新 订阅 settingsChangeDetector，设置变更时立即调用 BaseSandboxManager.updateConfig()；refreshConfig() 提供同步刷新。 权限规则更新后沙箱立即生效，避免竞态条件导致新规则在下一次命令前未生效。

对 Agent 开发的参考价值

1. 权限系统设计

  • 分层防御: 把权限检查分为”规则层”（用户配置，最高优先级）和”模式层”（全局行为），两层独立，互不干扰。
  • Fail-closed 默认: 任何不确定的情况都倾向于询问用户，而非自动放行。新工具的默认权限应该是最保守的。
  • 规则不变量: 用户明确配置的规则（deny/ask）不能被任何自动化机制绕过，包括 Hook 和 AI 分类器。

2. 多 Agent 安全

  • 权限冒泡: 后台 Agent 不能弹框，但权限决策必须由人类做出。通过邮箱把请求路由到有 UI 的 Leader，是一个优雅的解决方案。
  • shouldAvoidPermissionPrompts: 后台代理的标志位，让权限系统知道不能弹框，自动走 Hook → 拒绝路径。
  • PermissionRequest hooks: 给自动化脚本机会处理权限请求，是无头环境中唯一的自动批准机制。

3. Auto 模式安全

  • 危险规则剥离: 进入 AI 自动模式时，必须移除会绕过 AI 检查的规则。否则用户的 Bash(python:*) 规则会让 AI 在没有安全检查的情况下执行任意 Python 代码。
  • 拒绝计数: 连续拒绝超过阈值时降级为人工确认，防止 AI 分类器在某些场景下无限拒绝导致任务卡死。
  • fail-closed/open 开关: 分类器不可用时的行为应该是可配置的，紧急情况下可以远程切换。

4. 规则系统设计

  • 多来源合并: 规则可以来自多个来源（企业策略、用户配置、项目配置、CLI 参数、会话临时），按优先级合并。企业管理员可以通过 allowManagedPermissionRulesOnly 锁定规则来源。
  • 规则语法转义: 规则内容中的括号需要转义（\( \)），防止解析歧义。这是一个容易被忽视但重要的细节。
  • 遗留名称兼容: 工具改名时，通过 LEGACY_TOOL_NAME_ALIASES 映射保持向后兼容，旧规则不会失效。

5. 沙箱安全

  • 沙箱 ≠ 权限系统: 沙箱是 OS 级别的隔离（macOS Seatbelt / Linux bubblewrap），权限系统是应用级别的控制。两者互补：权限系统控制”是否执行”，沙箱控制”执行时能做什么”。
  • 配置自动派生: 沙箱配置直接从权限规则派生，FileEdit(allow:path) 自动变成 allowWrite，WebFetch(deny:domain:xxx) 自动变成 deniedDomains。用户只需配置权限规则，沙箱自动跟进。
  • autoAllowBashIfSandboxed: 沙箱启用时，可以自动允许 Bash 命令（因为沙箱已经限制了危害范围），减少用户打扰。
  • 防逃逸设计: 沙箱内不能修改 settings.json（防止提权）、不能写入 .claude/skills/（防止植入恶意技能）、专项防护 git bare repo 逃逸攻击（scrubBareGitRepoFiles()）。
  • 启用失败可见性: 用户显式开启沙箱但依赖缺失时，必须给出明确提示，而非静默跳过。静默失败是安全隐患——用户以为有保护，实际上没有（修复 #34044）。
  • 动态配置更新: 权限规则变更后立即同步到沙箱（refreshConfig()），避免竞态条件导致新规则在下一次命令前未生效。
