记忆系统 — Claude Code 源码深度分析

一句话理解

Claude Code 的记忆系统分四层：项目规则文件（CLAUDE.md）在每次对话开始时注入 System Prompt，自动记忆（MEMORY.md）在对话结束后由后台代理提取并写入，会话记忆在 compact 时提供上下文连续性，团队记忆通过服务端同步在多人之间共享。

白话版运行流程

整体一条线

对话开始

"fire-and-forget：把这轮对话的重要信息存下来"

四层记忆 — “各司其职”

类比：一个公司的知识管理体系。

层级 类比 存在哪 谁来写
CLAUDE.md / rules 公司规章制度 项目文件系统 人工维护
AutoMem (MEMORY.md) 个人工作笔记 ~/.claude/projects/<slug>/memory/ 后台代理自动提取
TeamMem 团队共享知识库 同上 /team/ + 服务端 后台代理 + 服务端同步
SessionMemory 会议纪要 ~/.claude/session-memory/<id>.md 每轮对话后更新

异步预取 — “不阻塞主流程的智能检索”

类比：你在开会时，助理悄悄去查资料，等你需要的时候刚好递过来。

怎么运作：

  1. 用户刚输入，系统立刻启动 startRelevantMemoryPrefetch()（异步，不等结果）
  2. 主流程继续：LLM 流式响应、工具执行……
  3. 在 tool_result 收集完后，消费预取结果：
    • 过滤掉已经在上下文里的记忆（去重）
    • 剩余的作为附件消息注入对话流
  4. 用 using 语法绑定生命周期，任何退出路径都自动清理

后台提取 — “对话结束后自动整理笔记”

类比：开完会自动生成会议纪要，存到知识库里。

关键设计：

  • 互斥：主代理刚写过记忆 → 后台跳过（避免重复）
  • 节流：距上次提取不足 N 轮 → 跳过（避免频繁）
  • Forked Agent：共享主对话的 prompt cache，有独立的文件读写状态
  • fire-and-forget：不阻塞主循环，shutdown 前用 drainPendingExtraction 等待完成

团队记忆同步 — “多人共享知识库”

类比：团队共享的 Notion 文档，本地编辑后自动同步到云端。

怎么运作：

  1. 会话开始时 pullTeamMemory（服务端内容覆盖本地）
  2. 本地文件变化 → fs.watch 监听 → 2秒去抖 → pushTeamMemory
  3. 只上传内容 hash 变化的文件（delta upload）
  4. 上传前用 gitleaks 规则扫描密钥，含密钥的文件永不离开本机
  5. 冲突时用 ETag 乐观锁，412 冲突 → 轻量探测 → 重算 delta → 重试

以下是源码级别的详细分析，如果你只想了解整体思路，上面的部分已经足够。

核心函数详解 — “它到底做了什么”

1. getMemoryFiles() — 记忆文件发现入口

文件: src/utils/claudemd.ts

具体做了什么:

  1. Memoized 缓存: 使用 lodash memoize 包裹，首次调用后结果缓存，后续调用直接返回。/compact 时会 clear() 缓存并触发 InstructionsLoaded hook 重新加载。

  2. 按优先级收集文件（后加载的优先级更高）:

    • Managed CLAUDE.md（/etc/claude-code/CLAUDE.md）— 最低优先级，企业级策略
    • User CLAUDE.md（~/.claude/CLAUDE.md）— 用户全局配置
    • 从根目录到 CWD 逐层遍历：CLAUDE.md、.claude/CLAUDE.md、.claude/rules/*.md、CLAUDE.local.md
    • AutoMem MEMORY.md — 自动记忆索引
    • TeamMem MEMORY.md — 团队记忆索引

  3. 为什么从根到 CWD 遍历: 父目录的规则先加载，子目录的规则后加载（更高优先级），实现”越具体越优先”的覆盖语义。

2. parseMemoryFileContent() — 记忆文件解析

文件: src/utils/claudemd.ts

具体做了什么:

  1. 解析 frontmatter: parseFrontmatterPaths() 提取 paths 字段（glob 模式），用于条件规则——只有当前操作的文件路径匹配时，该规则才生效。

  2. 词法分析: Lexer().lex() 扫描文件内容：

    • stripHtmlComments：移除 HTML 注释（<!-- ... -->）
    • extractIncludePaths：提取 @include path/to/file.md 指令

  3. 递归处理 include: processMemoryFile() 递归展开 @include，最大深度 MAX_INCLUDE_DEPTH=5，防止循环引用。外部路径（项目目录外）需要用户明确授权。

  4. 截断保护: truncateEntrypoint(200行/25KB) — 单个入口文件超过限制时截断，防止单个 CLAUDE.md 撑爆 System Prompt。

3. startRelevantMemoryPrefetch() — 异步记忆预取

文件: src/utils/attachments.ts

具体做了什么:

  1. 立即启动，不等结果: 在用户输入处理的最开始就启动，返回一个 Promise。主流程继续执行，不阻塞。

  2. using 语法绑定: TC39 Explicit Resource Management 语法，确保任何退出路径（正常返回、异常、abort）都会调用 dispose() 清理资源，防止内存泄漏。

  3. findRelevantMemories():

    • scanMemoryFiles()：扫描 ~/.claude/projects/<slug>/memory/ 下的所有 .md 文件，解析 frontmatter 获取标题和 hook
    • Sonnet sideQuery：把记忆文件列表和用户查询发给 Sonnet，让它选出最相关的最多 5 条
    • 返回选中的记忆文件路径列表

  4. readMemoriesForSurfacing(): 读取选中文件的实际内容，准备注入。

  5. 消费时机: 在 tool_result 收集完后（settledAt !== null && consumedOnIteration === -1）才消费预取结果，确保注入时机正确。

4. filterDuplicateMemoryAttachments() — 记忆去重

文件: src/utils/attachments.ts

这是记忆注入的关键安全机制，防止同一条记忆被重复注入上下文。

具体做了什么:

  1. 过滤阶段: 检查每条预取到的记忆，如果其文件路径已经在 readFileState（已读文件状态）中 → 过滤掉（这条记忆已经在上下文里了）。

  2. Mark 阶段（过滤之后）: 把通过过滤的记忆路径写入 readFileState。

  3. 为什么是 mark-after-filter 而不是 mark-before-filter:

    • 如果先 mark 再 filter：预取阶段写入 readFileState → 过滤时看到自己 → 全部丢弃 → 记忆永远注入不进来
    • 先 filter 再 mark：过滤时 readFileState 里还没有这些记忆 → 正常通过 → 注入后再标记

5. executeExtractMemories() — 后台记忆提取

文件: src/services/extractMemories/extractMemories.ts

具体做了什么:

  1. 互斥检查: hasMemoryWritesSince(lastExtractionTime) — 如果主代理在上次提取后写过记忆，跳过本次提取（主代理已经处理了，后台不重复）。

  2. 节流检查: turnsSinceLastExtraction < N — 距上次提取不足 N 轮，跳过（防止每轮都提取）。还检查 token 增长量，增长不够多说明没有值得提取的新信息。

  3. 构建记忆清单: scanMemoryFiles() + formatMemoryManifest() — 把现有记忆文件整理成清单，告诉提取代理”已经有哪些记忆了，不要重复”。

  4. 启动 Forked Agent: runForkedAgent(maxTurns=5) — 启动一个子代理：

    • 工具权限：只能用 Read/Grep/Glob + 记忆目录内的 Edit（不能乱改其他文件）
    • 任务：分析本轮对话，提取值得长期保存的信息，写入 MEMORY.md
    • 完成后发送 memory_saved 系统消息通知主对话

  5. fire-and-forget: 不等待结果，主循环继续。但 drainPendingExtraction() 在 shutdown 前会等待完成，确保数据不丢失。

6. pushTeamMemory() — 团队记忆推送

文件: src/services/teamMemorySync/index.ts

具体做了什么:

  1. 密钥扫描: 调用 secretScanner 用 gitleaks 规则子集扫描每个文件，含密钥的文件加入 suppressedFiles 列表，永不上传。

  2. 计算 delta: 对比本地文件的内容 hash 与 serverChecksums（上次同步时记录的服务端 hash），只上传 hash 不同的文件。删除操作不上传（server wins 策略）。

  3. 分批上传: 如果 delta 超过 200KB，batchDeltaByBytes() 分批 PUT，避免单次请求过大。

  4. 乐观锁处理:

    • 每次 PUT 带 If-Match: <ETag> 请求头
    • 服务端返回 412（冲突）→ 调用 fetchTeamMemoryHashes() 轻量探测（只拉 hash，不拉内容）→ 刷新 serverChecksums → 重算 delta → 重试（最多 2 次）

  5. 永久失败抑制: 认证失败（401/403）→ 设置 pushSuppressedReason → 后续推送直接跳过，不无限重试。清除 pushSuppressedReason 的方式是重启会话。

记忆提取与注入循环

这是整个记忆系统最核心的运行时循环。每轮对话都经历：预取 → 注入 → 提取三个阶段。

第一阶段：预取（用户输入时立即启动，异步不阻塞）

startRelevantMemoryPrefetch() 启动后，主流程继续处理 LLM 响应和工具调用。预取在后台用 Sonnet sideQuery 选出最相关的记忆。

第二阶段：注入（tool_result 收集完后消费）

预取结果经过 filterDuplicateMemoryAttachments() 去重后，作为附件消息注入对话流。mark-after-filter 确保不会自引用。

第三阶段：提取（对话结束后 fire-and-forget）

executeExtractMemories() 启动 Forked Agent，分析本轮对话，把值得保存的信息写入 MEMORY.md。

设计模式总结

模式 实现细节 为什么这样设计
Memoized + Cache-clear getMemoryFiles 使用 lodash memoize，/compact 时 clear 记忆文件在一次会话中几乎不变，缓存避免重复 IO。compact 是唯一需要重新加载的时机，精确清除而非全量失效。
Forked Agent 模式 Session Memory 和 Extract Memories 都使用 runForkedAgent 子代理共享主对话的 prompt cache（节省 token），同时有独立的 readFileState（不污染主对话的文件读取状态）。
闭包状态隔离 initExtractMemories 使用闭包而非模块级变量 模块级变量在测试中会跨用例污染。闭包让每个测试用例在 beforeEach 中调用 init 获得完全干净的状态。
Fire-and-forget + Drain 后台提取不阻塞主循环，drainPendingExtraction 在 shutdown 前等待 用户不需要等待记忆提取完成才能继续对话。但 shutdown 时必须等待，否则最后一轮对话的记忆会丢失。
Mark-after-filter 记忆去重中先过滤再标记 readFileState 先 mark 再 filter 会导致预取的记忆把自己过滤掉（自引用循环）。先 filter 再 mark 确保首次注入正常通过，后续重复注入被拦截。
互斥提取 hasMemoryWritesSince 检测主代理写入时间 主代理刚写过记忆，说明它已经处理了这些信息。后台代理重复提取会产生冗余记忆，互斥避免浪费。
分层安全 路径穿越检测 + realpath 验证 + 密钥扫描 团队记忆涉及多用户共享，需要多层防护：防止恶意路径（../../../etc/passwd）、防止密钥泄露到服务端。
乐观锁 + Delta Upload ETag 条件请求 + 内容 hash delta 乐观锁避免了悲观锁的性能开销（不需要先获取锁再操作）。Delta upload 只传变化的文件，最小化网络传输。
四类记忆分类法 user/feedback/project/reference 明确的分类约束了记忆范围，防止存储可推导信息（如代码逻辑）。只存储”下次对话时 LLM 不会自然知道”的信息。
永久失败抑制 pushSuppressedReason + 重启清除 认证失败是持久性错误，重试没有意义。抑制后续推送避免无效网络请求，重启会话时自然重新尝试认证。

对 Agent 开发的参考价值

1. 记忆系统设计

  • 分层优先级是关键：Managed → User → Project → Local，模型更关注后加载的内容。设计自己的记忆系统时，越具体的规则应该越晚注入。
  • MEMORY.md 是索引而非记忆本身，每条一行 - [Title](file.md) — hook。索引和内容分离，让 System Prompt 保持轻量，按需加载详细内容。
  • 四类分类法 (user/feedback/project/reference) 有效约束了记忆范围，避免把可推导的信息也存进去（浪费空间且可能过时）。

2. 自动提取模式

  • Forked Agent 共享 prompt cache 是高效的后台任务模式。子代理不需要重新发送整个上下文，直接复用主对话的 cache。
  • 互斥设计：主代理写了记忆 → 后台跳过该范围，避免冲突。设计多写者系统时，互斥检查比加锁更轻量。
  • 节流策略：turnsSinceLastExtraction + token 增长阈值，防止过度提取。两个维度的节流比单一维度更精准。

3. 检索与注入

  • 异步预取：在用户输入时启动，不阻塞主循环。Promise.race 模式让预取结果”刚好在需要时准备好”。
  • Mark-after-filter：记忆去重的关键技巧，防止自引用循环。任何”生产者-消费者”模式中，标记时机都很重要。
  • using 语法：TC39 Explicit Resource Management 确保任何退出路径都清理资源，比 try/finally 更简洁可靠。

4. 团队同步

  • Delta Upload：只上传变化的文件，不传删除操作（server wins 策略简化了冲突处理）。
  • 乐观锁：412 冲突时用轻量 ?view=hashes 探测，无需全量拉取。适合写冲突概率低的场景。
  • 密钥扫描：上传前用 gitleaks 规则子集检测，含密钥的文件永不离开本机。安全检查应该在数据离开本地之前完成。

5. 安全考量

  • 外部 include 审批：CLAUDE.md 引用外部文件需用户明确授权，防止恶意 CLAUDE.md 引用系统文件。
  • 路径穿越防护：validateTeamMemWritePath 使用 resolve + realpath 双重验证，resolve 处理 ..，realpath 处理符号链接。
  • 条件规则：.claude/rules/*.md 支持 frontmatter paths 字段做 glob 匹配，只在操作特定文件时生效，减少无关规则的干扰。
