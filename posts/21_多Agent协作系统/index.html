多Agent协作系统 — Claude Code 源码深度分析

一句话理解

用户跟”主 Agent”对话，主 Agent 可以派活给子 Agent（像领导叫下属干活，干完汇报），也可以拉一个团队（像建一个群聊，大家各干各的，通过”收件箱”互通消息）。

白话版运行流程

整体分两条路

当 LLM 决定需要帮手时，会调用 AgentTool。这个工具内部有一个路由开关，根据参数走两条完全不同的路：

用户提问 → 主Agent思考 → "我需要帮手" → 调用 AgentTool
                                              │
                    ┌─────────────────────────┴─────────────────────────┐
                    ▼                                                   ▼
            【子Agent模式】                                       【团队模式】
           传了 prompt                                      传了 prompt + name + team
           没传 name/team                                    
                    │                                                   │
                    ▼                                                   ▼
        在当前进程内启动一个"下属"                              开一个新终端/新线程，
        下属干完活，结果直接                                   创建一个独立的"队友"
        通过函数返回值交给主Agent                               队友们通过"收件箱文件"互发消息

子Agent模式 — “叫下属干活”

类比：你是经理，叫一个下属去查资料。下属在你旁边干活，干完直接口头汇报给你。

运行过程：

  1. 主Agent 说”帮我搜一下这个文件的实现”
  2. 系统创建一个子Agent（比如 Explore 类型），给它一套精简的工具
  3. 子Agent 独立运行一个”思考-调工具-思考”的循环
  4. 每条消息通过 yield 实时传回给主Agent
  5. 子Agent 干完了，主Agent 拿到结果继续思考

三种子Agent类型：

类型 什么时候用 通俗理解
指定类型（如 Explore、Plan） 明确知道要干什么 “叫专门搜文件的人去搜”
Fork 需要父级的完整对话上下文 “克隆一个我自己，带着全部记忆去干”
general-purpose 没指定，走默认 “叫一个通用助手去干”

同步 vs 异步：

  • 同步：主Agent 等着子Agent 干完（2分钟内大概能完成的任务）
  • 异步：主Agent 不等，立刻继续做自己的事，子Agent 干完后会弹通知
  • 中途转换：同步任务跑超 2 分钟，自动变成异步（用户也可以手动点”放到后台”）

团队模式 — “建群协作”

类比：你是项目经理，拉了一个群，分配几个人各做一部分。大家各自在自己的终端里干活，干完了在群里汇报”我好了”。

运行过程：

  1. 主Agent（Leader）创建一个”团队”，本质是在 ~/.claude/teams/ 下建了个文件夹
  2. Leader 派出多个 Teammate，每个可能是：
    • 一个新的 tmux 终端面板（独立进程）
    • 一个 iTerm2 分屏（独立进程）
    • 同进程内的一个隔离线程（轻量级）
  3. Leader 创建”任务”（写到磁盘 JSON 文件），分配给不同 Teammate
  4. Teammate 从”收件箱”里收到任务 → 干活 → 完成后往 Leader 的收件箱写”我好了”
  5. Leader 每 1 秒看一下收件箱，发现有人汇报了就处理

“收件箱”是什么：就是一个 JSON 文件（~/.claude/teams/{团队名}/inboxes/{人名}.json），写入时加文件锁防冲突。简单但有效。

两种模式的关键区别

子Agent模式 团队模式
关系 父子（上下级） 平级（Leader + Workers）
通信 函数调用（yield） 文件系统收件箱（JSON 文件）
生命期 跟随父Agent 独立运行，可以活得更久
使用场景 搜索、分析、单次任务 多人并行开发、复杂协作
开销 轻（同进程） 重（可能开新进程/终端）

权限和安全怎么处理

  • 子Agent：默认不能弹权限对话框（因为可能在后台），遇到需要权限的操作会自动拒绝
  • Team 的 Worker：遇到需要权限的操作，会通过收件箱把”权限请求”发给 Leader，Leader 的终端上弹出确认框，用户确认后结果写回 Worker 的收件箱
  • 关闭队友：Leader 发 shutdown_request → Worker 回 shutdown_approved → Leader 关掉 Worker 的终端面板

一个完整的团队协作例子

1. 用户说："帮我重构这个模块，加上测试"
2. 主Agent 创建团队 "refactor-team"
3. 主Agent 创建两个任务：①重构代码 ②写测试
4. 主Agent 派出两个 Teammate：
   - "coder"（负责重构）→ 开一个 tmux 面板
   - "tester"（负责测试）→ 开另一个 tmux 面板
5. coder 收到任务 → 修改文件 → 提交代码 → 标记"完成" → 通知 Leader
6. tester 收到任务 → 等 coder 完成 → 写测试 → 标记"完成" → 通知 Leader
7. Leader 收到两个"完成"通知 → 汇总结果 → 回复用户
8. Leader 发 shutdown 给 coder 和 tester → 关闭团队

核心函数详解 — “它到底做了什么”

1. AgentTool.call() — 总入口路由

文件: src/tools/AgentTool/AgentTool.tsx

当 LLM 决定调用 Agent 工具时，call() 被触发。它是整个多Agent系统的”交通枢纽”，负责把请求路由到正确的执行路径。

具体做了什么（逐步）:

  1. 权限检查: 从 AppState 读取 permissionMode，如果传了 team_name 但 swarms 未启用，直接报错；如果当前已经是 teammate 且传了 name，也报错（禁止嵌套 teammate）。

  2. 多代理 spawn 路径: 如果 teamName && name 同时存在 → 调用 spawnTeammate() 在 tmux/iTerm2/进程内创建一个独立的 teammate，立即返回 { status: 'teammate_spawned' }，不等待结果。

  3. 选择代理定义:

    • Fork 路径: 如果 fork 实验开关打开且 subagent_type 没有传，选择内置的 FORK_AGENT，同时做递归防护——主要检查 toolUseContext.options.querySource === 'agent:builtin:fork'（compaction-resistant，能抵抗 autocompact 的消息重写），fallback 扫描消息历史中是否已有 <fork-boilerplate> 标签，有就拒绝（防止 fork 子代理再 fork）。
    • 普通路径: 从 agentDefinitions.activeAgents 列表中按 subagent_type 查找对应的 AgentDefinition，找不到就抛错。

  4. MCP 服务器等待: 如果 agent 定义了 requiredMcpServers，会轮询等待最多 30 秒（每 500ms 检查一次），直到这些 MCP 服务器连接就绪且有 tools 可用。

  5. 构建 System Prompt 和消息:

    • Fork 路径: 直接继承父级已渲染好的 system prompt（toolUseContext.renderedSystemPrompt），调用 buildForkedMessages() 构建消息——这样所有 fork 子代理产生字节相同的 API 请求前缀，最大化 prompt cache 命中。
    • 普通路径: 调用 agent 自己的 getSystemPrompt() 再用 enhanceSystemPromptWithEnvDetails() 增强（加入 OS 信息、CWD 等环境细节）。

  6. 同步 vs 异步决策（shouldRunAsync）: 满足以下任一条件就走异步——run_in_background=true、agent 定义中 background=true、coordinator 模式、fork 实验开启（所有 fork 都强制异步）、Kairos 助手模式、proactive 模式。但如果环境变量禁用了后台任务则全部降级为同步。

  7. 异步路径: 调用 registerAsyncAgent() 注册后台任务，然后用 void（fire-and-forget）启动 runAsyncAgentLifecycle()，立即返回 { status: 'async_launched', agentId, outputFile } 给父级。

  8. 同步路径: 调用 registerAgentForeground() 注册前台任务，进入 while(true) 循环。每次循环用 Promise.race([下一条消息, 后台化信号]) 竞争——如果后台化信号先 resolve（用户点击了后台按钮或 2 分钟自动超时），就把迭代器清理掉，重新以异步模式启动 runAgent()；如果消息先到，继续处理。循环结束后调用 finalizeAgentTool() 提取最终结果。

2. registerAsyncAgent() — 注册后台任务

文件: src/tasks/LocalAgentTask/LocalAgentTask.tsx

具体做了什么:

  1. 调用 initTaskOutputAsSymlink(agentId, transcriptPath) — 在磁盘上创建一个符号链接，指向 agent 的 transcript 文件，方便后续通过 TaskOutput 查看进度。
  2. 创建 AbortController：如果传入了 parentAbortController，则创建链接到父级的子控制器（createChildAbortController）；否则创建独立的控制器（createAbortController()）。这意味着后台代理通常不会因为用户按 ESC 而被中断，但可以通过父级控制器级联中断。
  3. 构造 LocalAgentTaskState 对象：type: 'local_agent'、status: 'running'、isBackgrounded: true（一开始就是后台）。
  4. 注册 cleanup handler（用于 gracefulShutdown 时自动 kill）。
  5. 调用 registerTask(taskState, setAppState) 写入全局 AppState，这样 UI 上就能看到这个任务在运行。

3. registerAgentForeground() — 注册前台任务

文件: src/tasks/LocalAgentTask/LocalAgentTask.tsx

与异步版本的关键区别: isBackgrounded: false（初始在前台），且会创建一个 backgroundSignal Promise。

具体做了什么:

  1. 创建 AbortController 和任务状态，但 isBackgrounded: false。
  2. 创建一个 Promise<void> 及对应的 resolve 函数，存入 backgroundSignalResolvers Map（key 是 agentId）。
  3. 注册任务到 AppState。
  4. 如果配置了 autoBackgroundMs（通过 CLAUDE_AUTO_BACKGROUND_TASKS 环境变量或 GrowthBook 特性开关启用，启用后默认 120000ms = 2分钟），设置 setTimeout：到时间后自动把任务标记为 isBackgrounded: true 并 resolve 那个 Promise。
  5. 返回 { taskId, backgroundSignal, cancelAutoBackground } — 调用方通过 Promise.race 监听 backgroundSignal。

4. runAgent() — 子代理核心执行循环

文件: src/tools/AgentTool/runAgent.ts

这是一个 async function*（异步生成器），是所有子代理执行的核心。

具体做了什么（逐步）:

  1. 解析模型: getAgentModel() 从 agent 定义、主循环模型、用户指定模型中选出最终使用的模型。

  2. 处理 Fork 消息: 如果有 forkContextMessages（从父级继承的消息历史），先用 filterIncompleteToolCalls() 过滤掉没有 tool_result 配对的 tool_use（否则 API 会报错）。

  3. 构建文件状态缓存: fork 上下文 → 克隆父级的 readFileState（防止子代理的文件缓存污染父级）；否则 → 创建新的带大小限制的缓存。

  4. 获取 User/System Context: 并行获取用户上下文和系统上下文。对于 Explore/Plan 代理，省略 claudeMd（节省约 5-15 Gtok/周）和 gitStatus（如果需要，agent 自己执行 git status 获取新鲜数据）。

  5. 覆盖权限模式: 构建 agentGetAppState() 包装函数——如果 agent 定义了自己的 permissionMode（如 bubble）且父级不是 bypassPermissions/acceptEdits/auto，就覆盖为 agent 自己的模式。对于异步代理，还会设置 shouldAvoidPermissionPrompts: true（它们没有终端 UI，不能弹权限对话框）——但 bubble 模式例外，即使是异步代理也允许弹窗（权限会冒泡到父级终端显示）。

  6. 解析工具集: fork 路径直接用父级的工具池（保证 API 请求字节相同）；普通路径用 resolveAgentTools() 根据 agent 定义过滤工具。

  7. 执行 SubagentStart hooks: 遍历所有注册的钩子，收集 additionalContexts（额外上下文注入到初始消息中）。

  8. 预加载 Skills: 如果 agent frontmatter 指定了 skills，逐一加载技能内容并添加为初始消息。

  9. 初始化 Agent MCP 服务器: 连接 agent 专属的 MCP 服务器，获取工具，与已有工具合并（按名称去重）。

  10. 创建子代理上下文 createSubagentContext():

    • 克隆 readFileState
    • 创建新的空 Set（nestedMemoryAttachmentTriggers、loadedNestedMemoryPaths、dynamicSkillDirTriggers、discoveredSkillNames 等多个集合）
    • 根据共享选项决定 setAppState 是父级的还是空函数（异步代理的 setAppState 是 no-op，防止后台代理修改 UI 状态）
    • 所有 UI 回调设为 undefined（子代理不能控制父级 UI）
    • 新的 queryTracking.chainId + depth + 1

  11. 主查询循环: for await (const message of query({...})) — 这里调用了核心的 query() 函数，它会不断循环调用 LLM → 执行工具 → 返回结果，直到 LLM 停止调用工具或达到 maxTurns。每条消息都会:

    • stream_event 中的 message_start → 记录 TTFT 指标
    • attachment 类型的 max_turns_reached → 跳出循环
    • 可记录的消息 → recordSidechainTranscript() 持久化到磁盘 + yield 给父级

  12. finally 清理链: MCP 连接 → session hooks → prompt cache 追踪 → 文件状态缓存 → 消息数组 → perfetto → transcript 子目录 → agent 的 todos 条目 → 后台 bash 任务 → Monitor MCP 任务，全部依次清理。

5. buildForkedMessages() — 构建 Fork 子代理的消息

文件: src/tools/AgentTool/forkSubagent.ts

目标: 让所有 fork 子代理产生字节相同的 API 请求前缀，最大化 prompt cache 命中。

具体做了什么:

  1. 克隆 assistant 消息: 复制父级的完整 assistant 消息（包含所有 content blocks：thinking、text、每个 tool_use），赋新 UUID。

  2. 收集所有 tool_use blocks: 从 assistant 消息中过滤出 type === 'tool_use' 的块。

  3. 构建 tool_result 占位符: 为每个 tool_use 创建 tool_result，内容全部是完全相同的 "Fork started — processing in background" — 这是为了所有 fork 子代理产生字节相同的 API 请求前缀。

  4. 构建 user 消息: 一条消息包含所有 tool_result 块 + 一个 text 块。text 块内容是 <fork-boilerplate> 包裹的 10 条不可协商规则（”你是一个 fork worker process”、”不要再 fork”、”直接执行”、”保持简洁”、”报告格式：Scope/Result/Key files/Files changed/Issues”等）+ 具体的 per-child 指令。

  5. 返回: [fullAssistantMessage, toolResultMessage] — 只有最后的 text block（per-child 指令）不同，前缀全部相同。

isInForkChild() 递归防护: 扫描所有 user 消息，检查是否存在 <fork-boilerplate> 标签。如果发现，说明当前已经在 fork 子代理里了，拒绝再次 fork。

6. createSubagentContext() — 创建隔离的子代理上下文

文件: src/utils/forkedAgent.ts

核心理念: 默认所有可变状态都隔离，防止子代理和父代理互相干扰。调用方可以显式 opt-in 共享特定回调。

具体做了什么:

返回一个新的 ToolUseContext 对象:
├─ readFileState: 克隆父级的文件缓存（防止子代理的文件读取污染父级缓存）
├─ nestedMemoryAttachmentTriggers: 全新的空 Set
├─ loadedNestedMemoryPaths: 全新的空 Set
├─ contentReplacementState: 克隆的（保证 fork 子代理做出与父级相同的替换决策 → 相同的 wire prefix → cache hit）
├─ abortController: override > 共享父级的 > 创建链接到父级的子 controller
├─ getAppState: override > shareAbortController ? 父级的 : 包装为设置 shouldAvoidPermissionPrompts: true（交互式子代理共享父级 AbortController 时，说明它能显示 UI，不需要避免权限弹窗）
├─ setAppState: 如果 shareSetAppState → 父级的；否则 → () => {}（空函数）
├─ setAppStateForTasks: 始终指向根 store（即使 setAppState 是空函数，任务注册/kill 必须到达根 store）
├─ localDenialTracking: 如果不共享 setAppState → 创建新的拒绝追踪状态
├─ setInProgressToolUseIDs: () => {}（空函数）
├─ setResponseLength: 根据 shareSetResponseLength 决定
├─ UI 回调全部 undefined（子代理不能控制父级 UI）
├─ agentId: override > 新建
└─ queryTracking: 新的 chainId + depth + 1

7. writeToMailbox() — 写消息到邮箱

文件: src/utils/teammateMailbox.ts

这是整个多代理通信的基础设施。每个 teammate 在磁盘上有一个 JSON 文件作为”收件箱”。

具体做了什么:

  1. ensureInboxDir() — 确保 ~/.claude/teams/{team}/inboxes/ 目录存在（mkdir -p）。
  2. 获取收件箱路径: ~/.claude/teams/{team}/inboxes/{agent_name}.json。
  3. 尝试用 flag: 'wx'（不存在才创建）创建空文件 []。如果文件已存在，EEXIST 错误被忽略。
  4. 获取文件锁: 使用 proper-lockfile 库，锁文件路径为 ${inboxPath}.lock，重试配置为最多 10 次、最小超时 5ms、最大超时 100ms。
  5. 获取锁后重新读取: 读取最新状态（在获取锁的等待期间，其他代理可能已经写入了新消息）。
  6. 追加新消息: 构造 { ...message, read: false } 并 push 到数组。
  7. 写回完整数组: JSON.stringify(messages, null, 2) 写入文件。
  8. 释放锁。

邮箱文件格式:

[
  {
    "from": "researcher",
    "text": "消息内容或 JSON 结构化消息",
    "timestamp": "2024-01-01T00:00:00.000Z",
    "read": false,
    "color": "blue",
    "summary": "5-10 word summary"
  }
]

8. useInboxPoller() — 收件箱轮询 Hook

文件: src/hooks/useInboxPoller.ts

一个 React Hook，每 1 秒轮询一次磁盘上的收件箱文件。

具体做了什么:

  1. 决定谁需要轮询: getAgentNameToPoll() 根据当前身份返回:

    • in-process teammate → undefined（它们有自己的轮询机制）
    • process-based teammate → 返回自己的 agentName
    • team lead → 从 teamContext 查找 lead 名称
    • 普通会话 → undefined

  2. 每 1 秒: 调用 readUnreadMessages() 读取未读消息。

  3. 分类所有消息: 逐条尝试解析为结构化协议消息:

    • permission_request → 权限请求队列
    • permission_response → 权限响应队列
    • sandbox_permission_request/response → 沙箱网络权限
    • shutdown_request → 关机请求
    • shutdown_approved → 关机批准
    • team_permission_update → 团队权限更新
    • mode_set_request → 模式设置
    • plan_approval_request → 计划审批
    • 其他 → 普通消息

  4. 处理权限请求（Leader 侧）: 创建完整的 ToolUseConfirm 条目（包含 onAllow/onReject/onAbort 回调），回调通过 sendPermissionResponseViaMailbox 写回 worker 的收件箱。路由到标准的权限确认 UI。

  5. 处理权限响应（Worker 侧）: 调用 processMailboxPermissionResponse() 回调已注册的 Promise（worker 在请求权限时会挂起等待这个 Promise resolve）。

  6. 处理关机批准（Leader 侧）: kill 对应的 tmux pane → 从 teamContext.teammates 移除 → unassign 相关任务 → 更新 AppState。

  7. 投递普通消息: 格式化为 XML → 如果空闲（!isLoading），立即通过 onSubmitTeammateMessage() 提交为新 turn；如果忙碌，加入 AppState.inbox.messages 队列，标记为 pending。当会话变空闲时，useEffect 自动投递 pending 消息。

  8. 最后: 调用 markMessagesAsRead() 标记为已读（获取锁 → 所有消息 read = true → 写回）。

10. SendMessageTool.call() — 智能体间消息路由

文件: src/tools/SendMessageTool/SendMessageTool.ts

**这是多代理通信的”路由器”**，根据收件人地址决定走哪条通信通道。

具体路由逻辑:

  1. Bridge/UDS 地址: parseAddress(input.to) 解析地址 scheme:

    • "bridge:session-id" → 通过 postInterClaudeMessage() 发送到远程 REPL 会话（跨机器）
    • "uds:/path/to/socket" → 通过 sendToUdsSocket() 发送到本地 Unix 域套接字（跨会话）

  2. 进程内子代理路由: 如果 input.to 不是广播且是字符串消息:

    • 从 AppState.agentNameRegistry 查找名称 → agentId
    • 找到 running task → queuePendingMessage(agentId, message) 排队等待下次工具轮
    • 找到 stopped task → resumeAgentBackground() 自动恢复已暂停的代理
    • 已清理的 task → 尝试从磁盘 transcript 恢复

  3. 广播 (input.to === '*'): 读取 team file，对每个非自身 member 调用 writeToMailbox()。

  4. 普通消息: writeToMailbox(recipientName, { from, text, summary, timestamp, color }, teamName)。

  5. 结构化消息处理:

    • shutdown_request → 构造关机请求 JSON → 写入目标 teammate 的 inbox
    • shutdown_response + approve → 写入 team-lead 的 inbox shutdown_approved JSON → 对 in-process teammate 调用 abortController.abort()，对 tmux teammate 调用 gracefulShutdown(0)（进程退出）
    • plan_approval_response → 写入对应 teammate 的 inbox 审批响应

核心流程图

AgentTool.call() 路由决策

call() 是整个多Agent系统的”交通枢纽”。LLM 每次想派活，都要经过这里的层层判断，最终落到某一条执行路径上。下面用文字走一遍完整的决策链路：

第一关：你想建团队，还是想派子Agent？

  • 如果同时传了 team_name（团队名）和 name（队友名）→ 走团队模式，调用 spawnTeammate() 开一个新终端/新线程，立刻返回”已派出”，不等结果。
  • 否则 → 走子Agent模式，继续往下判断。

第二关：派什么类型的子Agent？

  • 如果明确传了 subagent_type（比如 "Explore"、"Plan"）→ 从已注册的 agent 列表里查找对应定义，找不到就报错。
  • 如果没传 subagent_type：
    • Fork 实验开关打开 → 走 Fork 路径（克隆父级全部上下文，共享 prompt cache）。但要先做递归防护：如果当前已经在一个 fork 子代理里了，拒绝再 fork。
    • Fork 没开 → 默认用 general-purpose（通用子代理）。

第三关：前台跑还是后台跑？

满足以下任一条件就走异步（后台）：

  • 用户显式说 run_in_background=true
  • agent 定义里写了 background: true
  • coordinator 模式 / fork 实验 / Kairos 助手模式 / proactive 模式

不满足 → 走同步（前台等待），但如果超过 2 分钟还没完成，会自动转为异步。

异步路径：注册一个后台任务 → fire-and-forget 启动 agent → 立即返回”已启动”。agent 干完后会弹桌面通知。

同步路径：注册一个前台任务 → 进入 while 循环，每次用 Promise.race 同时等两件事——agent 的下一条消息 vs 后台化信号。如果消息先到，处理后继续循环；如果后台化信号先到（用户点了”放后台”或 2 分钟超时），就把当前的前台迭代器关掉，重新以后台模式启动，然后立即返回。

设计模式总结

模式 实现细节 为什么这样设计
单进程多上下文 AsyncLocalStorage 为每个 in-process teammate 创建独立 TeammateContext。getAgentName()/getTeamName() 全局函数通过 ALS 返回正确的身份。 无需启动新进程，内存高效，一个 Node.js 进程内可运行多个 teammate，且彼此身份完全隔离。
文件系统即消息总线 JSON 文件 + proper-lockfile (10次重试, 5-100ms退避)。每次写操作: 获取锁 → 读最新 → 追加 → 写回 → 释放锁。 解耦、天然持久化、跨进程可用，无需额外消息中间件。崩溃重启后消息不丢失（未标已读的会被重新消费）。
同步→异步透明迁移 registerAgentForeground() 创建 backgroundSignal Promise + 2分钟自动超时。Promise.race 监听。触发后 agentIterator.return() 清理前台 → 重新以 isAsync=true 启动 runAgent()。 前台任务可中途转后台，用户体验连续。不需要提前预判任务时长——短任务同步等待结果，长任务自动转后台。
Fork 共享缓存 fork 子代理继承父的 system prompt + tool pool + 消息前缀。所有 tool_result 使用相同占位符文本。只有最后的 per-child 指令不同。 所有 fork 子代理产生字节相同的 API 请求前缀 → 最大化 Anthropic API 的 prompt cache 命中率，节省大量 token 费用。
结构化协议消息 JSON 消息类型鉴别 (type 字段) 叠加在纯文本邮箱之上。isPermissionRequest()、isShutdownRequest() 等函数逐一尝试 JSON 解析 + schema 校验。 在简单的文件邮箱上实现复杂交互（权限请求/响应、关机协商、计划审批），无需引入 RPC 框架。
Fail-closed 默认 isConcurrencySafe = false、isReadOnly = false。异步代理的 setAppState 默认是 no-op。 新工具忘记声明属性时走最保守路径。后台代理不能意外修改 UI 状态。
三层后端抽象 detectAndGetBackend() 运行时自动检测: iTerm2 with it2 CLI → tmux → in-process 回退。统一 TeammateExecutor 接口 (spawn/kill/listPanes)。 一套代码适配多种终端环境，用户无需手动配置。
递归防护 isInForkChild() 检查消息中的 <fork-boilerplate> 标签。主检查是 querySource（compaction-resistant），消息扫描是 fallback。 防止 fork 子代理再 fork（fork 子代理保留 Agent 工具是为了 cache-identical tool defs，但实际调用时要拦截）。

对 Agent 开发的参考价值

1. 子代理编排模式

  • 同步 vs 异步: 简单任务用同步等结果（调用方被阻塞直到完成），长任务用异步 fire-and-forget + 完成通知。两者可以中途转换——同步任务超过 2 分钟自动变异步。
  • Fork 模式: 需要父级完整对话上下文的调查任务用 fork（共享 prompt cache，极大节省 token），独立任务用新代理（干净的上下文）。
  • 工具池隔离: 子代理的工具池由 assembleToolPool() 独立构建，使用 worker 自己的 permissionMode，不受父级的工具限制影响。

2. 通信架构

  • 文件系统邮箱 是极简但有效的 IPC 方案：JSON 文件 + 文件锁。适合低频消息场景（每秒轮询一次），实现简单、天然持久化。
  • proper-lockfile 的重试配置（10次 / 5-100ms退避）平衡了并发安全和性能。
  • 结构化协议消息 在纯文本层上叠加类型安全的 JSON 消息格式——一个 type 字段就区分了 10+ 种协议消息（权限、关机、审批等），简洁高效。

3. 团队协调

  • 任务看板模式（Team = TaskList）: 创建团队自动创建对应任务目录，多代理共享一个任务板。任务分配自动触发邮箱通知。
  • idle_notification + Stop hook: 队友完成任务后自动通知 leader “我空了”，leader 可以分配新任务或关闭队友。
  • 权限冒泡（bubble mode）: 后台代理遇到需要用户确认的操作时，权限请求通过邮箱冒泡到 leader 终端显示。

4. 资源管理

  • finally 清理链: runAgent() 的 finally 块依次清理 11 项资源（MCP 连接 → hooks → cache → 文件状态 → 消息 → perfetto → transcript → todos → bash 任务 → Monitor 任务），确保任何退出路径都不会泄漏。
  • abort 级联: 同步代理共享父级的 AbortController（按 ESC 一起停），异步代理有独立的（不受 ESC 影响，只能显式 kill）。In-process teammate 也是独立的（不随 leader 中断）。

5. 状态隔离

  • AsyncLocalStorage: 零成本的上下文隔离。多个 in-process teammate 运行在同一进程内，但 getAgentName() 始终返回正确的身份。
  • readFileState 独立克隆: 每个子代理有自己的文件缓存副本，防止子代理读取的文件内容影响父级或其他代理。
  • setAppState 隔离: 异步代理的 setAppState 是空函数——防止后台代理意外修改 UI 状态。但 setAppStateForTasks 始终指向根 store，确保任务注册/kill 能正确传递。
