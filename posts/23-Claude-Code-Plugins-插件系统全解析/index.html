Claude Code Plugins：插件系统全解析

发表于2026|更新于2026|AI相关
|字数总计:3.5k|阅读时长:13分钟

前言：前面几篇文章分别讲了 MCP、Skills、Hooks、Agent 定义系统。但 Claude Code 还有一个把这些能力”打包分发”的机制——Plugins（插件）。如果说 Skills 是”操作手册”，MCP 是”接口协议”，那 Plugins 就是”把手册、协议、工具箱打包成一个可安装的 App”。

一句话理解 Plugins

Plugin = Skills + Agents + Hooks + MCP + 设置，打包成一个可分发的扩展包。

类比现实世界：

  • Skills = 单个操作手册
  • MCP = 单个外部接口
  • Hooks = 单个自动化规则
  • Plugin = 把上面这些打包成一个”工具箱”，可以一键安装、一键卸载
Plugin 和独立配置的关系：

独立配置（散装）：
  .claude/skills/deploy/SKILL.md     ← 一个 Skill
  .claude/agents/reviewer.json       ← 一个 Agent
  .claude/settings.json → hooks      ← 一些 Hooks
  .claude/settings.json → mcpServers ← 一些 MCP Server

Plugin（打包）：
  my-plugin/
  ├── .claude-plugin/plugin.json     ← 插件元数据
  ├── skills/deploy/SKILL.md         ← 打包的 Skill
  ├── agents/reviewer.json           ← 打包的 Agent
  ├── hooks/hooks.json               ← 打包的 Hooks
  └── .mcp.json                      ← 打包的 MCP Server

什么时候用 Plugin，什么时候用独立配置

这是很多人搞不清楚的问题。官方文档给了一个清晰的判断标准：

方式 Skill 命名 适用场景
独立配置（.claude/ 目录） /hello 个人工作流、项目定制、快速实验
Plugin（带 plugin.json） /plugin-name:hello 团队共享、社区分发、跨项目复用

选择独立配置：

  • 只在一个项目里用
  • 个人习惯，不需要分享
  • 正在实验阶段，还没定型
  • 想要短命令名（/deploy 而不是 /my-plugin:deploy）

选择 Plugin：

  • 需要分享给团队或社区
  • 同一套能力要在多个项目中使用
  • 需要版本控制和更新机制
  • 可以接受命名空间前缀（/my-plugin:deploy）

我的建议是：先用独立配置快速迭代，稳定后再转成 Plugin 分发。

Plugin 的目录结构

一个完整的 Plugin 长这样：

my-plugin/
├── .claude-plugin/
│   └── plugin.json          # 插件元数据（必需）
├── skills/                  # Skills（可选）
│   └── deploy/
│       └── SKILL.md
├── agents/                  # 自定义 Agent（可选）
│   └── reviewer.json
├── hooks/                   # Hooks 配置（可选）
│   └── hooks.json
├── .mcp.json                # MCP Server 配置（可选）
├── .lsp.json                # LSP Server 配置（可选）
├── bin/                     # 可执行文件（可选，会加入 PATH）
├── settings.json            # 默认设置（可选）
└── README.md                # 文档

注意一个常见错误：skills/、agents/、hooks/ 这些目录要放在插件根目录，不要放在 .claude-plugin/ 里面。.claude-plugin/ 里面只放 plugin.json。

关于 Skills 的存放位置：Plugin 中的 Skills 放在插件自己的目录下（如 my-plugin/skills/deploy/SKILL.md），而不是项目的 .claude/skills/ 下。当你通过 claude --plugin-dir ./my-plugin 加载或从市场安装插件时，Claude Code 会自动扫描插件目录下的 skills/ 文件夹，并注册到系统中，同时加上命名空间前缀（plugin-name:skill-name）。你不需要手动把插件的 Skills 复制到 .claude/skills/ 目录——插件加载机制会自动处理这一切。

plugin.json：插件的身份证

{
  "name": "my-plugin",
  "description": "一句话描述插件功能",
  "version": "1.0.0",
  "author": {
    "name": "Your Name"
  },
  "homepage": "https://github.com/you/my-plugin",
  "repository": "https://github.com/you/my-plugin",
  "license": "MIT"
}

name 字段特别重要——它决定了插件的命名空间。插件里的 Skill 都会以 name 为前缀：如果 name 是 my-plugin，那么 skills/deploy/SKILL.md 对应的命令就是 /my-plugin:deploy。

实战：从零创建一个 Plugin

以一个”Git 工作流”插件为例，包含一个 Skill（提交代码）和一个 Hook（写文件后自动格式化）。

第一步：创建目录结构

mkdir -p my-git-workflow/.claude-plugin
mkdir -p my-git-workflow/skills/commit
mkdir -p my-git-workflow/hooks

第二步：写 plugin.json

// my-git-workflow/.claude-plugin/plugin.json
{
  "name": "git-workflow",
  "description": "Git 工作流自动化：规范化提交、自动格式化",
  "version": "1.0.0",
  "author": {
    "name": "Your Name"
  }
}

第三步：写 Skill

<!-- my-git-workflow/skills/commit/SKILL.md -->
---
name: commit
description: 规范化提交代码，自动生成 commit message
disable-model-invocation: true
allowed-tools:
  - Bash(git *)
  - Read
---

# 规范化提交

## 流程
1. 运行 `git diff --staged --stat` 查看暂存区变更
2. 如果暂存区为空，运行 `git add -A` 暂存所有变更
3. 分析变更内容，生成符合 Conventional Commits 规范的 commit message
4. 格式：`<type>(<scope>): <description>`
5. type 可选：feat / fix / docs / style / refactor / test / chore
6. 执行 `git commit -m "<message>"`

## 规则
- commit message 用中文描述
- scope 用英文（如 auth、api、ui）
- 一次提交只做一件事，如果变更太多，建议拆分

第四步：写 Hooks

// my-git-workflow/hooks/hooks.json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          {
            "type": "command",
            "command": "npx prettier --write $(echo $HOOK_INPUT | jq -r '.tool_input.file_path // .tool_input.filePath // empty')",
            "timeout": 10
          }
        ]
      }
    ]
  }
}

第五步：本地测试

# 用 --plugin-dir 加载本地插件
claude --plugin-dir ./my-git-workflow

# 在 Claude Code 中测试
/git-workflow:commit

开发过程中修改了插件文件后，在 Claude Code 中运行 /reload-plugins 即可热加载更新，不需要重启。

Plugin 的各个组件详解

Skills 组件

Plugin 中的 Skills 和独立 Skills 写法完全一样，唯一区别是命名空间。

独立 Skill：  /deploy
Plugin Skill：/my-plugin:deploy

命名空间的好处是避免冲突——如果你安装了两个插件都有 deploy Skill，它们不会互相覆盖。

Agents 组件

Plugin 可以包含自定义 Agent 定义，放在 agents/ 目录下：

// my-plugin/agents/security-reviewer.json
{
  "description": "安全审查专员，检查代码中的安全隐患",
  "tools": ["Read", "Glob", "Grep"],
  "disallowedTools": ["Bash", "Write", "Edit"],
  "prompt": "你是一个严格的安全审查员。检查代码中的安全隐患，包括：注入攻击、XSS、敏感信息泄露、不安全的加密、权限绕过等。只报告问题，不修改代码。",
  "model": "sonnet",
  "permissionMode": "dontAsk"
}

安装插件后，这个 Agent 会出现在 /agents 列表中。

Hooks 组件

Plugin 的 Hooks 通过 hooks/hooks.json 文件定义，格式和 settings.json 中的 hooks 字段完全一致。简单来说，Hooks 就是在 Claude Code 执行到某个生命周期阶段时，自动执行对应的 command 命令：

  • PreToolUse：在 Claude 调用某个工具之前触发（适合做安全检查、权限校验）
  • PostToolUse：在 Claude 调用某个工具之后触发（适合做格式化、日志记录）
  • **SessionStart**：会话开始时触发
  • **matcher**：用来匹配具体是哪个工具（如 Bash、Write|Edit）
// my-plugin/hooks/hooks.json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "echo $HOOK_INPUT | jq -r '.tool_input.command' | grep -qE 'rm -rf|sudo|chmod 777' && echo '{\"decision\":\"block\",\"reason\":\"危险命令被阻止\"}' && exit 2 || exit 0",
            "timeout": 5
          }
        ]
      }
    ]
  }
}

这个 Hook 会在每次执行 Bash 命令前检查是否包含危险命令（rm -rf、sudo、chmod 777），如果匹配则阻止执行。

注意：这里的 Hook 示例（PreToolUse + Bash 前置安全拦截）和前面实战部分第四步的 Hook 示例（PostToolUse + Write|Edit 后置格式化）内容不同，这是为了分别展示了 Hooks 的两种典型用法：前置拦截（执行前检查并阻止危险操作）和后置处理（执行后自动格式化代码）。

MCP Server 组件

Plugin 可以通过 .mcp.json 文件配置 MCP Server：

// my-plugin/.mcp.json
{
  "mcpServers": {
    "my-tool": {
      "command": "node",
      "args": ["${PLUGIN_DIR}/mcp-server/index.js"],
      "env": {
        "API_KEY": "${API_KEY}"
      }
    }
  }
}

${PLUGIN_DIR} 会自动替换为插件的根目录路径。

LSP Server 组件

LSP（Language Server Protocol）插件给 Claude 提供实时的代码智能能力。对于常见语言（TypeScript、Python、Rust 等），官方市场已经有预置的 LSP 插件。只有需要支持不常见语言时才需要自己写：

// my-plugin/.lsp.json
{
  "go": {
    "command": "gopls",
    "args": ["serve"],
    "extensionToLanguage": {
      ".go": "go"
    }
  }
}

bin 目录

bin/ 目录下的可执行文件会在插件启用时自动加入 Bash 工具的 PATH。这意味着 Claude 可以直接调用这些命令，不需要写完整路径。

举个例子，假设你的插件有一个自定义安全检查脚本：

my-plugin/
└── bin/
    └── check-security    # 自定义安全检查脚本

启用插件后，Claude 可以直接执行 check-security，而不需要写 /path/to/my-plugin/bin/check-security。这对于插件自带辅助命令行工具的场景很有用——把工具脚本放在 bin/ 下就能被 Claude 无缝调用。

settings.json

Plugin 可以通过根目录的 settings.json 提供默认设置。目前只支持 agent 字段——指定一个插件内的 Agent 作为主线程的默认 Agent：

// my-plugin/settings.json
{
  "agent": "security-reviewer"
}

启用这个插件后，Claude Code 会默认使用 security-reviewer Agent 的系统提示词、工具限制和模型设置。

注意：settings.json 不是对 Agent 的”限制”，更准确地说是”默认配置”——它改变的是 Claude Code 的默认行为模式（包括系统提示词、可用工具列表、模型选择、权限模式等）。用户仍然可以手动切换覆盖，它只是提供一个默认值。

官方插件一览

Claude Code 官方仓库提供了一批高质量的示例插件，值得参考：

插件名 功能 包含组件
code-review 自动化 PR 代码审查，5 个并行 Agent 分别检查不同维度 Skill + 5 个 Agent
commit-commands Git 工作流自动化（commit、push、PR） 3 个 Skill
feature-dev 7 阶段结构化功能开发流程 Skill + 3 个 Agent
hookify 通过对话分析自动创建 Hooks 4 个 Skill + Agent
security-guidance 安全提醒，监控 9 种安全模式 Hook（PreToolUse）
plugin-dev 插件开发工具包，7 个专家 Skill 7 个 Skill + 3 个 Agent
ralph-wiggum 自引用迭代循环，Claude 反复执行直到完成 2 个 Skill + Hook
frontend-design 前端设计指导，避免”AI 审美” Skill（自动触发）
explanatory-output-style 教育性输出风格，解释实现选择 Hook（SessionStart）

其中 code-review 插件的设计特别值得学习——它并行启动 5 个 Sonnet Agent，分别负责 CLAUDE.md 合规检查、Bug 检测、历史上下文分析、PR 历史分析和代码注释审查，最后汇总结果。这是 Skills + Agents 组合的典型范例。

插件的安装与管理

安装方式

# 在 Claude Code 中使用 /plugin 命令安装
/plugin

# 从本地目录加载（开发测试用）
claude --plugin-dir ./my-plugin

# 同时加载多个插件
claude --plugin-dir ./plugin-one --plugin-dir ./plugin-two

管理命令

命令 说明
/plugin 打开插件管理器，浏览和安装插件
/reload-plugins 热加载所有插件的更新
/help 查看所有已安装插件的 Skill 列表
/agents 查看所有已安装插件的 Agent 列表

优先级规则

当本地 --plugin-dir 加载的插件和已安装的市场插件同名时，本地版本优先。这方便你在不卸载市场版本的情况下测试本地修改。唯一的例外是通过企业管理策略强制启用的插件，它们不能被覆盖。

从独立配置迁移到 Plugin

如果你已经在 .claude/ 目录下有了一套好用的 Skills、Agents 和 Hooks，想打包成 Plugin 分享给团队，迁移步骤很简单：

第一步：创建插件结构

mkdir -p my-plugin/.claude-plugin

第二步：创建 plugin.json

{
  "name": "my-plugin",
  "description": "从项目配置迁移而来",
  "version": "1.0.0"
}

第三步：复制现有配置

# 复制 Skills
cp -r .claude/skills my-plugin/

# 复制 Agents
cp -r .claude/agents my-plugin/

# 复制 Commands（如果有）
cp -r .claude/commands my-plugin/

第四步：迁移 Hooks

如果你在 settings.json 中配置了 Hooks，需要把它们提取到 hooks/hooks.json 中：

mkdir my-plugin/hooks

把 settings.json 中的 hooks 对象复制到 hooks/hooks.json，格式完全一样。

迁移后的变化

维度 迁移前 迁移后
Skill 命令 /deploy /my-plugin:deploy
配置位置 .claude/ 目录 独立的插件目录
分发方式 复制文件 插件市场安装
版本管理 随项目代码 独立版本号

Plugin vs Skills vs MCP：完整对比

维度 Plugin Skills MCP
本质 可分发的扩展包 操作手册 接口协议
粒度 粗（包含多个组件） 中（单个工作流） 细（单个工具）
分发 插件市场 / Git 仓库 文件夹复制 Server 部署
命名空间 有（plugin:skill） 无 无
包含关系 可包含 Skills + Agents + Hooks + MCP 独立 独立
适用场景 团队共享、社区分发 项目定制、个人工作流 外部系统连接
热更新 /reload-plugins 自动检测 需重启 
层次关系：

┌──────────────────────────────────────┐
│           Plugin（分发层）            │  ← 打包 + 命名空间 + 版本管理

│  │  Skills + Agents + Hooks       │  │  ← 能力层
│  │  ┌──────────────────────────┐  │  │
│  │  │  MCP Server              │  │  │  ← 连接层

发布到官方市场

当你的插件开发完成并经过充分测试后，可以提交到 Anthropic 官方市场：

  • Claude.ai：claude.ai/settings/plugins/submit
  • Console：platform.claude.com/plugins/submit

提交前的检查清单：

  • 插件结构正确（plugin.json 在 .claude-plugin/ 中，其他目录在根级别）
  • 包含 README.md 说明安装和使用方法
  • 使用语义化版本号
  • 所有 Skill 都有 description 字段
  • 在多个项目中测试过

小结

Claude Code Plugins 核心要点：

  1. Plugin 是”打包分发”机制——把 Skills、Agents、Hooks、MCP 打包成可安装的扩展
  2. 命名空间隔离——/plugin-name:skill-name 避免多插件冲突
  3. 先独立配置，再转 Plugin——在 .claude/ 中快速迭代，稳定后打包分发
  4. plugin.json 是唯一必需文件——放在 .claude-plugin/ 目录中，其他组件都可选
  5. /reload-plugins 支持热更新——开发时不需要反复重启
  6. 官方插件是最好的学习材料——code-review、feature-dev、hookify 等插件展示了 Skills + Agents + Hooks 的最佳组合实践
