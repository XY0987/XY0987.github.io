Function Calling深度解析：让AI拥有双手

发表于2025|更新于2026|AI相关
|字数总计:2.4k|阅读时长:9分钟

前言：正常情况下的 AI 只能”说”不能”做”。用户问天气，它只能编一个；用户想查数据库，它无能为力。Function Calling 是让 AI 获得”双手”的关键技术—— LLM 不再只是一个聊天机器人。

Function Calling 是整个 AI 应用工程化的分水岭——它让 LLM 从”知识问答”进化为”任务执行”。理解 FC 的工作原理，是理解后续MCP协议工具调用的基础。

Function Calling 是什么

一句话：Function Calling 让 LLM 告诉你”应该调用什么函数、传什么参数”，但实际的函数执行由你的代码完成。

用户: "北京今天天气怎么样？"

传统LLM: "北京今天大概是晴天吧"  ← 瞎猜的

Function Calling:
  LLM决策 → "需要调用 get_weather 函数，参数 city=北京"
  你的代码 → 调用天气API → 获取真实数据
  LLM生成 → "北京今天晴，25℃，适合出行"  ← 基于真实数据

关键理解：LLM 不直接执行函数！它只负责”决策”——判断该不该调函数、调哪个、传什么参数。真正的执行在代码里。这个”决策+执行”分离的设计非常重要。

完整工作流程

                    ┌──────────────┐
                    │   你的代码    │
                    └──────┬───────┘
                           │
        ①定义工具+用户问题   │
                           ▼
                    ┌──────────────┐
                    │   LLM API    │
                    └──────┬───────┘
                           │
              ②返回工具调用决策（JSON结构）
                           │
                           ▼
                  ┌──────────────┐
        ③解析参数  │   你的代码    │  ④执行实际函数
        并调用     └──────┬───────┘  （调API/查DB等）
                         │
      ⑤把函数结果返回给LLM │
                         ▼
                  ┌──────────────┐
                  │   LLM API    │  ⑥生成最终自然语言回答
                  └──────────────┘

实战代码：完整的 Function Calling 实现

// fc-demo.mjs —— Function Calling 完整示例
// 需要安装：npm install openai

import OpenAI from 'openai';

const openai = new OpenAI({
  apiKey: process.env.OPENAI_API_KEY
});

// ========== 步骤1：定义可用的工具 ==========
const tools = [
  {
    type: 'function',
    function: {
      name: 'get_weather',
      description: '获取指定城市的实时天气信息。当用户询问天气相关问题时调用此函数。',
      parameters: {
        type: 'object',
        properties: {
          city: {
            type: 'string',
            description: '城市名称，如：北京、上海、广州'
          },
          unit: {
            type: 'string',
            enum: ['celsius', 'fahrenheit'],
            description: '温度单位，默认celsius'
          }
        },
        required: ['city']
      }
    }
  },
  {
    type: 'function',
    function: {
      name: 'search_product',
      description: '搜索商品信息。当用户想查找或购买商品时调用此函数。',
      parameters: {
        type: 'object',
        properties: {
          keyword: {
            type: 'string',
            description: '搜索关键词'
          },
          max_price: {
            type: 'number',
            description: '最高价格（可选）'
          }
        },
        required: ['keyword']
      }
    }
  }
];

// ========== 步骤2：实现实际的函数 ==========
// 这些是你的业务逻辑，LLM 不知道也不关心具体实现
const functions = {
  get_weather: async ({ city, unit = 'celsius' }) => {
    // 实际项目中这里会调用真实天气API
    // 这里用模拟数据演示
    const mockData = {
      '北京': { temp: 22, condition: '晴', humidity: 45 },
      '上海': { temp: 26, condition: '多云', humidity: 72 },
      '广州': { temp: 30, condition: '雷阵雨', humidity: 85 }
    };
    const data = mockData[city] || { temp: 20, condition: '未知', humidity: 50 };
    return JSON.stringify({
      city,
      temperature: data.temp,
      unit,
      condition: data.condition,
      humidity: `${data.humidity}%`
    });
  },

  search_product: async ({ keyword, max_price }) => {
    // 实际项目中这里会查询数据库或调用商品API
    const mockProducts = [
      { name: `${keyword} A款`, price: 99, rating: 4.5 },
      { name: `${keyword} B款`, price: 199, rating: 4.8 },
      { name: `${keyword} C款 旗舰版`, price: 399, rating: 4.9 }
    ];
    const filtered = max_price
      ? mockProducts.filter(p => p.price <= max_price)
      : mockProducts;
    return JSON.stringify(filtered);
  }
};

// ========== 步骤3：核心调用流程 ==========
async function chat(userMessage) {
  console.log(`\n👤 用户: ${userMessage}`);

  // 第一轮：发送用户消息和工具定义
  const messages = [
    {
      role: 'system',
      content: '你是一个智能助手，可以查询天气和搜索商品。请根据用户需求选择合适的工具。'
    },
    { role: 'user', content: userMessage }
  ];

  const firstResponse = await openai.chat.completions.create({
    model: 'gpt-4',
    messages,
    tools,
    tool_choice: 'auto'  // 让模型自动决定是否调用工具
  });

  const firstMessage = firstResponse.choices[0].message;

  // 判断模型是否决定调用工具
  if (!firstMessage.tool_calls) {
    // 不需要工具，直接返回文本回答
    console.log(`🤖 助手: ${firstMessage.content}`);
    return firstMessage.content;
  }

  // 有工具调用请求
  console.log(`🔧 工具调用:`);
  messages.push(firstMessage);  // 保留模型的决策消息

  // 执行每个工具调用（可能有多个并行调用）
  for (const toolCall of firstMessage.tool_calls) {
    const funcName = toolCall.function.name;
    const funcArgs = JSON.parse(toolCall.function.arguments);

    console.log(`   → ${funcName}(${JSON.stringify(funcArgs)})`);

    // 执行实际函数
    const funcResult = await functions[funcName](funcArgs);

    // 将结果放回消息列表
    messages.push({
      role: 'tool',
      tool_call_id: toolCall.id,  // 必须匹配
      content: funcResult
    });
  }

  // 第二轮：带上函数结果，让模型生成最终回答
  const secondResponse = await openai.chat.completions.create({
    model: 'gpt-4',
    messages
  });

  const finalAnswer = secondResponse.choices[0].message.content;
  console.log(`🤖 助手: ${finalAnswer}`);
  return finalAnswer;
}

// ========== 测试 ==========
await chat('北京今天天气怎么样？');
// 🔧 工具调用: → get_weather({"city":"北京"})
// 🤖 助手: 北京今天天气晴朗，气温22℃，湿度45%，非常适合出行！

await chat('帮我找一下200块以内的蓝牙耳机');
// 🔧 工具调用: → search_product({"keyword":"蓝牙耳机","max_price":200})
// 🤖 助手: 为您找到以下200元以内的蓝牙耳机：
// 1. 蓝牙耳机 A款 - ¥99，评分4.5
// 2. 蓝牙耳机 B款 - ¥199，评分4.8
// 推荐B款，评分更高，性价比也不错。

await chat('你好，今天心情不错');
// （不触发工具调用，直接文本回复）
// 🤖 助手: 很高兴听到您心情不错！有什么我可以帮您的吗？

Parallel Function Calling（并行调用）

2023年11月，OpenAI 支持了并行工具调用——模型可以一次返回多个函数调用请求：

// 用户: "北京和上海今天天气怎么样？"
// 模型会返回两个并行的工具调用：

const response = await openai.chat.completions.create({
  model: 'gpt-4',
  messages: [{ role: 'user', content: '北京和上海今天天气怎么样？' }],
  tools
});

// response.choices[0].message.tool_calls 包含两个调用：
// [
//   { id: "call_1", function: { name: "get_weather", arguments: '{"city":"北京"}' } },
//   { id: "call_2", function: { name: "get_weather", arguments: '{"city":"上海"}' } }
// ]

// 你可以并行执行它们！
const results = await Promise.all(
  response.choices[0].message.tool_calls.map(async (toolCall) => {
    const args = JSON.parse(toolCall.function.arguments);
    const result = await functions[toolCall.function.name](args);
    return { tool_call_id: toolCall.id, role: 'tool', content: result };
  })
);

并行调用在”对比类”需求中效率提升巨大。但要注意：如果函数之间有依赖关系（比如先查用户再查订单），模型通常会自动串行处理，只需要按顺序返回结果。

我的踩坑实录

坑1：函数描述直接影响调用准确率

// ❌ 模糊的描述 —— 模型容易误判
{
  name: 'get_data',
  description: '获取数据'  // 太模糊了！
}

// ✅ 精确的描述
{
  name: 'get_weather',
  description: '获取指定城市的实时天气信息，包括温度、天气状况和湿度。当用户询问天气、温度、是否需要带伞等问题时调用此函数。'
}

坑2：参数类型和枚举值要写清楚

// ❌ 不清晰的参数定义
parameters: {
  properties: {
    status: { type: 'string' }  // 模型可能传任何字符串
  }
}

// ✅ 使用 enum 约束取值范围
parameters: {
  properties: {
    status: {
      type: 'string',
      enum: ['pending', 'completed', 'cancelled'],
      description: '订单状态'
    }
  }
}

坑3：处理模型”幻觉”调用

// 模型有时会编造不存在的函数名或参数
// 必须在代码层面做防御

async function executeToolCall(toolCall) {
  const funcName = toolCall.function.name;

  // 检查函数是否存在
  if (!functions[funcName]) {
    console.warn(`⚠️ 未知函数: ${funcName}`);
    return JSON.stringify({ error: `函数 ${funcName} 不存在` });
  }

  // 安全解析参数
  let args;
  try {
    args = JSON.parse(toolCall.function.arguments);
  } catch (e) {
    console.warn(`⚠️ 参数解析失败: ${toolCall.function.arguments}`);
    return JSON.stringify({ error: '参数格式错误' });
  }

  // 执行函数
  try {
    return await functions[funcName](args);
  } catch (e) {
    return JSON.stringify({ error: e.message });
  }
}

Function Calling 的局限性

理解局限性同样重要：

局限 具体表现 应对方案
平台绑定 OpenAI 格式和 Claude 格式不同 MCP 协议解决了这个问题
每次传递 工具定义需要每次请求都传 消耗 token，成本高
决策不可靠 复杂场景下可能选错工具 细化 description，增加示例
无状态 两次调用之间没有”记忆” 需要自己维护上下文

这些局限性，正是后来 MCP 协议 要解决的问题。

独家技术思考

Function Calling 的本质是什么？

我的理解是：Function Calling 本质上是”结构化输出”的一种特殊形式。

模型并没有真正”理解”函数的含义——它只是被训练成：当看到某些模式的输入时，输出特定 JSON 格式的内容。这就是为什么 description 写得越精确，调用准确率越高——你在帮模型做模式匹配。

我们在应用过程中，不要指望 LLM”理解”你的函数逻辑，你要做的是让 LLM 能够精确地做出”调用哪个函数”的决策，然后把执行交给确定性的代码。

小结

Function Calling 核心要点：

  1. LLM 负责决策，代码负责执行——这是核心设计哲学
  2. 函数描述的质量直接决定调用准确率
  3. 必须在代码层面做好防御——处理未知函数、参数错误等情况
  4. 并行调用可以大幅提升效率
