TCP与UDP协议详解：传输层的两大核心协议

发表于2025|更新于2026|基础知识
|字数总计:2.4k|阅读时长:7分钟

前言

在计算机网络中，传输层负责为应用层提供端到端的通信服务。TCP 和 UDP 是传输层最重要的两个协议，它们各有特点，适用于不同的场景。本文将深入介绍这两个协议的工作原理、区别及应用场景。

一、TCP 协议

1.1 TCP 简介

TCP（Transmission Control Protocol，传输控制协议）是一种面向连接的、可靠的传输层协议。

1.2 TCP 的特点

特性 说明
面向连接 通信前需要建立连接
可靠传输 保证数据无差错、不丢失、不重复、按序到达
面向字节流 将数据看作字节流，不保留应用层的边界
全双工通信 双方可以同时发送和接收数据
流量控制 使用滑动窗口控制发送速率
拥塞控制 避免网络拥塞

1.3 TCP 报文格式

┌─────────────────────────────────────────────────────────────────────────────┐
│                              TCP 报文格式                                    │
├──────────┬──────────┬─────────────────────────────────────────────────────────┤
│ 源端口    │ 目的端口 │  各 16 位                                               │
├──────────┴──────────┴─────────────────────────────────────────────────────────┤
│                          序号（Sequence Number）32 位                         │
├─────────────────────────────────────────────────────────────────────────────┤
│                          确认号（Acknowledgment Number）32 位                 │
├─────────┬──────┬─────┬────────────────────────────────────────────────────────┤
│数据偏移  │保留  │标志位│  窗口大小 16 位                                        │
├─────────┴──────┴─────┴────────────────────────────────────────────────────────┤
│   校验和 16 位        │          紧急指针 16 位                                │
├─────────────────────────────────────────────────────────────────────────────┤
│                              选项（可选）                                     │
├─────────────────────────────────────────────────────────────────────────────┤
│                              数据部分                                        │
└─────────────────────────────────────────────────────────────────────────────┘

标志位说明：
- SYN (Synchronize): 建立连接
- ACK (Acknowledgment): 确认
- FIN (Finish): 释放连接
- RST (Reset): 重置连接
- PSH (Push): 接收方应尽快将数据交给应用层
- URG (Urgent): 紧急指针有效

1.4 TCP 三次握手（建立连接）

客户端                                              服务端
   │                                                   │
   │ ──────── SYN=1, seq=x ─────────────────────────→ │
   │           (请求建立连接)                           │
   │                                                   │
   │ ←─────── SYN=1, ACK=1, seq=y, ack=x+1 ───────── │
   │           (确认并同意建立连接)                     │
   │                                                   │
   │ ──────── ACK=1, seq=x+1, ack=y+1 ──────────────→ │
   │           (确认服务端的确认)                       │
   │                                                   │
   │            连接建立成功                            │

// 为什么需要三次握手？
// 1. 防止已失效的连接请求报文段突然到达服务端
// 2. 同步双方的序列号
// 3. 防止服务端一直等待，浪费资源

// 示例：如果两次握手
// 客户端发送 SYN，但网络拥塞，客户端超时重发 SYN
// 服务端收到第二个 SYN，建立连接
// 第一个 SYN 到达服务端，服务端误以为是新连接（错误！）

1.5 TCP 四次挥手（释放连接）

客户端                                              服务端
   │                                                   │
   │ ──────── FIN=1, ACK=1, seq=u ─────────────────→ │
   │           (请求释放连接)                           │
   │                                                   │
   │ ←─────── ACK=1, seq=v, ack=u+1 ──────────────── │
   │           (确认客户端的释放请求)                   │
   │                                                   │
   │         客户端→服务端方向连接关闭                   │
   │         服务端可能还有数据要发送                    │
   │                                                   │
   │ ←─────── FIN=1, ACK=1, seq=w, ack=u+1 ───────── │
   │           (服务端请求释放连接)                     │
   │                                                   │
   │ ──────── ACK=1, seq=u+1, ack=w+1 ─────────────→ │
   │           (确认服务端的释放请求)                   │
   │                                                   │
   │            连接完全关闭                            │

// 为什么需要四次挥手？
// TCP 是全双工通信，每个方向都需要单独关闭
// 服务端收到 FIN 后，可能还有数据需要发送给客户端
// 所以 ACK 和 FIN 分开发送（不像三次握手可以合并）

1.6 TCP 可靠传输机制

1.6.1 序列号与确认应答

// 发送方发送数据包，每个包有序列号
// 接收方收到后返回 ACK，包含下一个期望的序列号

// 示例
// 发送方: 发送 seq=1, len=100 的数据
// 接收方: 返回 ack=101（期望下一个从 101 开始）

// 超时重传
// 如果发送方在一定时间内没收到 ACK，则重传数据

1.6.2 滑动窗口

// 滑动窗口实现流量控制
// 接收方通过窗口大小告知发送方自己的接收能力

// 发送方维护的窗口
// ┌───────────────────────────────────────────┐
// │  已发送确认  │  已发送未确认  │  可发送  │ 不可发送 │
// └───────────────────────────────────────────┘
//               ↑                           ↑
//            窗口左边界                   窗口右边界

// 收到 ACK 后，窗口向右滑动

1.6.3 拥塞控制

// 拥塞控制的四个阶段

// 1. 慢启动（Slow Start）
// 初始 cwnd = 1，每收到一个 ACK，cwnd 加倍
// cwnd: 1 → 2 → 4 → 8 → 16 → ...

// 2. 拥塞避免（Congestion Avoidance）
// 当 cwnd 达到 ssthresh（慢启动阈值），进入拥塞避免
// 每个 RTT，cwnd 加 1（线性增长）

// 3. 快速重传（Fast Retransmit）
// 收到 3 个重复 ACK，立即重传丢失的数据包
// 不等待超时

// 4. 快速恢复（Fast Recovery）
// 快速重传后，cwnd 减半，进入拥塞避免
// 不重新慢启动

二、UDP 协议

2.1 UDP 简介

UDP（User Datagram Protocol，用户数据报协议）是一种无连接的、不可靠的传输层协议。

2.2 UDP 的特点

特性 说明
无连接 不需要建立连接，直接发送数据
不可靠 不保证数据到达，不保证顺序
面向报文 保留应用层的边界，应用层给什么就发什么
没有拥塞控制 网络拥塞时不会降低发送速率
支持多播 支持一对一、一对多、多对多通信
开销小 报文头部只有 8 字节

2.3 UDP 报文格式

┌─────────────────────────────────────────────────────────────────────────────┐
│                              UDP 报文格式                                    │
├─────────────────────────────┬───────────────────────────────────────────────┤
│       源端口 16 位           │          目的端口 16 位                       │
├─────────────────────────────┼───────────────────────────────────────────────┤
│       长度 16 位             │          校验和 16 位                         │
├─────────────────────────────┴───────────────────────────────────────────────┤
│                              数据部分                                        │
└─────────────────────────────────────────────────────────────────────────────┘

// UDP 头部只有 8 字节，TCP 头部至少 20 字节

2.4 UDP 的应用场景

// 1. 实时音视频通信
// - 视频会议、直播
// - 丢帧比延迟更可接受

// 2. 在线游戏
// - 游戏状态同步
// - 快速响应比数据完整性更重要

// 3. DNS 查询
// - 查询请求小，响应快
// - 失败可以重试

// 4. IoT 设备通信
// - 资源受限设备
// - 简单高效

// 5. 流媒体
// - 实时视频/音频传输
// - 允许少量丢包

三、TCP 与 UDP 对比

3.1 核心区别

对比项 TCP UDP
连接性 面向连接 无连接
可靠性 可靠传输 不可靠传输
顺序 有序 无序
速度 较慢 较快
头部开销 20-60 字节 8 字节
流量控制 有（滑动窗口） 无
拥塞控制 有 无
传输方式 字节流 报文
多播 不支持 支持
应用场景 文件传输、网页、邮件 视频通话、游戏、DNS

3.2 形象比喻

TCP = 打电话
- 需要先拨号建立连接
- 对话双方确认对方在听
- 说话顺序有序
- 确保每句话都传达

UDP = 寄信/发短信
- 直接发送，不确认对方是否收到
- 可能丢失
- 可能乱序
- 发完就不管了

四、TCP 如何实现可靠性

4.1 可靠性的实现机制

// TCP 实现可靠性的核心机制

// 1. 校验和
// 检测数据在传输过程中是否发生变化

// 2. 序列号
// 对数据包进行编号，接收方按序重组

// 3. 确认应答
// 接收方收到数据后发送 ACK 确认

// 4. 超时重传
// 发送方在一定时间内未收到 ACK，重传数据

// 5. 滑动窗口
// 控制发送速率，避免接收方缓冲区溢出

// 6. 拥塞控制
// 避免网络拥塞，动态调整发送速率

4.2 超时重传时间（RTO）计算

// RTO (Retransmission Timeout) 的计算

// RTT (Round Trip Time): 往返时间

// 加权平均往返时间
RTTs = (1 - α) × RTTs + α × RTT  // α 通常取 0.125

// RTT 偏差加权平均
RTTd = (1 - β) × RTTd + β × |RTT - RTTs|  // β 通常取 0.25

// 超时重传时间
RTO = RTTs + 4 × RTTd

五、应用层协议的选择

5.1 什么时候选择 TCP

// 1. 数据完整性要求高
// - 网页浏览（HTTP/HTTPS）
// - 文件传输（FTP）
// - 电子邮件（SMTP、IMAP）

// 2. 需要可靠交付
// - 数据库连接
// - 远程登录（SSH、Telnet）

// 3. 数据量大
// - 大文件下载
// - 软件更新

5.2 什么时候选择 UDP

// 1. 实时性要求高
// - 视频会议
// - 在线直播
// - 在线游戏

// 2. 简单快速
// - DNS 查询
// - DHCP

// 3. 多播/广播
// - 局域网发现
// - 视频流分发

// 4. 应用层自己实现可靠性
// - QUIC 协议（HTTP/3）
// - 自定义协议

5.3 QUIC 协议

// QUIC = UDP + TCP 的可靠性 + TLS 加密

// 特点：
// 1. 基于 UDP，避免 TCP 的队头阻塞
// 2. 内置 TLS 1.3，减少握手延迟
// 3. 连接迁移：网络切换不断开
// 4. 快速建立连接：0-RTT 或 1-RTT

// HTTP/3 基于 QUIC 实现
// Chrome、Firefox 等浏览器已支持

六、面试常见问题

6.1 为什么 TCP 需要三次握手？

// 核心原因：防止已失效的连接请求报文段突然到达服务端

// 假设两次握手：
// 1. 客户端发送 SYN（网络拥塞，滞留）
// 2. 客户端超时重发 SYN
// 3. 服务端收到第二个 SYN，建立连接
// 4. 连接关闭后，第一个 SYN 到达服务端
// 5. 服务端误以为是新连接，一直等待客户端
// 6. 浪费服务端资源

// 三次握手可以避免这种情况

6.2 为什么 TCP 需要四次挥手？

// 核心原因：TCP 是全双工通信

// 1. 客户端发送 FIN，表示没有数据要发送了
// 2. 服务端收到 FIN，返回 ACK
// 3. 但服务端可能还有数据要发送给客户端
// 4. 服务端发送完毕后，发送自己的 FIN
// 5. 客户端返回 ACK
// 6. 连接完全关闭

// 不能合并 ACK 和 FIN，因为服务端可能还有数据要发
