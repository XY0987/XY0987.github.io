OpenSpec：规范驱动的AI编程

发表于2025|更新于2026|AI相关
|字数总计:1k|阅读时长:3分钟

前言：Vibe Coding 的自由度也带来了失控风险——AI 经常跑偏、返工不断、决策无据可查。于是出现了 OpenSpec，一个”规范驱动”的 AI 编程框架。

背景：Vibe Coding 的”后遗症”

上一篇我们讲了 Vibe Coding 的威力，但也提到了它的问题：

Vibe Coding 的理想：
  你：做一个博客系统
  AI：完整实现一个博客系统

Vibe Coding 的现实：
  你：做一个博客系统
  AI：做了一个博客系统（用了你不想用的技术栈）
  你：不对，我要用Next.js
  AI：重写了（但改了你之前确认过的功能）
  你：刚才的XXX功能怎么没了？
  AI：加回来了（又引入了新bug）
  ... 循环30分钟 ...

核心痛点：AI编程缺乏”记忆”和”规范”——每次对话都是从零开始，没有持久化的决策记录。

OpenSpec 是什么

OpenSpec 是一个规范驱动（Spec-Driven）的AI编程框架，核心理念是：

先写规范（Spec），再写代码（Code）。AI 编程也要有”设计文档”。

传统编程流程：
  需求 → 设计文档 → 编码 → 测试 → 发布

Vibe Coding 流程（无规范）：
  需求 → AI直接编码 → 发现问题 → 重新描述 → AI重新编码 → ...

OpenSpec 流程（规范驱动）：
  需求 → AI生成规范（Proposal）→ 人审核规范 → AI按规范编码 → 归档

核心工作流

openspec init          → 初始化项目规范体系
  ↓
/openspec:proposal     → AI 生成变更提案（包含：背景、方案、任务清单）
  ↓
人工审核 & 批准         → 确认规范无误
  ↓
/openspec:apply        → AI 严格按规范编码实现
  ↓
/openspec:archive      → 归档已完成的变更记录

规范的价值

<!-- 一个 OpenSpec 提案示例 -->
# Proposal: 添加用户认证功能

## 背景
当前系统缺少用户认证，所有API都是公开的。

## 设计决策
- 认证方案：JWT Token（理由：无状态、适合API场景）
- 存储方案：PostgreSQL（理由：已有实例，无需新增）
- 密码哈希：bcrypt（理由：业界标准，抗暴力破解）

## 任务清单
- [ ] 创建用户表（users）
- [ ] 实现注册接口 POST /api/auth/register
- [ ] 实现登录接口 POST /api/auth/login
- [ ] 实现 JWT 中间件
- [ ] 保护需要认证的路由

## 数据结构
```sql
CREATE TABLE users (
  id SERIAL PRIMARY KEY,
  email VARCHAR(255) UNIQUE NOT NULL,
  password_hash VARCHAR(255) NOT NULL,
  created_at TIMESTAMP DEFAULT NOW()
);

影响范围

  • 新增文件：auth.ts, middleware/jwt.ts
  • 修改文件：routes/index.ts, package.json

**有了这个规范**：
1. AI 编码时有明确的"需求文档"，不会偏离方向
2. 每个决策都有记录，后续维护有据可查
3. 人工只需要审核规范，不需要逐行审核代码
4. 规范可以跨AI工具使用（Cursor、Claude Code、CodeBuddy都支持）

## 我的实战感受

在实际项目中试用 OpenSpec 后，我的体感是：

| 指标 | 无规范的Vibe Coding | 用OpenSpec |
|------|-------------------|-----------|
| 需求偏离率 | 高（经常跑偏） | 低（有规范约束） |
| 返工次数 | 5-10次 | 1-2次 |
| 决策可追溯性 | 无 | 完整记录 |
| 初始准备时间 | 0 | 多5-10分钟写规范 |
| 总完成时间 | 反而更长（因为返工） | 更短 |

## OpenSpec 对 AI 编程的启示

OpenSpec 代表了 AI 编程成熟化的方向——**不是让 AI 自由发挥，而是在规范约束下高效执行**。这和传统软件工程中"设计先行"的理念一脉相承。

更深层来看，它回答了一个关键问题：**AI 编程和传统编程的最佳关系是什么？**

不是替代，而是增强：

传统软件工程的智慧 AI 的执行力
┌──────────┐ ┌──────────┐
│ 需求分析 │ │ 快速编码 │
│ 架构设计 │ + │ 模式匹配 │
│ 规范约束 │ │ 批量生成 │
│ 质量把控 │ │ 持续迭代 │
└──────────┘ └──────────┘
│ │
└────────┬───────────┘
▼
高质量、可控的
AI辅助编程


**我的看法**：无论你用哪个 AI 编程工具，"先想清楚再动手"这个原则永远不会错。OpenSpec 只是把这个原则工具化了——但即使不用它，你也应该在让 AI 写代码之前，先写清楚你的需求和约束。
