HTTP与HTTPS深度解析：原理、区别与安全性

发表于2024|更新于2026|基础知识
|字数总计:4.9k|阅读时长:18分钟

HTTP 和 HTTPS 是 Web 通信的基础协议。HTTPS 作为 HTTP 的安全版本，已经成为现代 Web 的标配。本文将深入解析两者的区别以及 HTTPS 的安全机制。

一、HTTP 简介

1.1 什么是 HTTP

HTTP（HyperText Transfer Protocol，超文本传输协议）是互联网上应用最广泛的网络协议，用于客户端与服务器之间的通信。

客户端                    服务器
   │                        │
   │───── HTTP 请求 ───────>│
   │                        │
   │<──── HTTP 响应 ────────│
   │                        │

1.2 HTTP 特点

// HTTP 默认端口
http://example.com:80

// HTTP 特点
const HTTPFeatures = {
  simple: '简单快速，基于请求-响应模型',
  flexible: '灵活可扩展，支持任意数据类型',
  stateless: '无状态，每个请求相互独立',
  connection: '支持持久连接（HTTP/1.1）'
}

1.3 HTTP 工作流程

1. 建立 TCP 连接（三次握手）
   客户端 ──SYN──> 服务器
   客户端 <──SYN+ACK── 服务器
   客户端 ──ACK──> 服务器

2. 发送 HTTP 请求

3. 服务器处理请求

4. 返回 HTTP 响应

5. 断开连接（四次挥手）

1.4 HTTP 的问题

┌─────────────────────────────────────┐
│         HTTP 安全问题                │
├─────────────────────────────────────┤
│ ❌ 明文传输，数据可被窃听            │
│ ❌ 不验证身份，可被伪装              │
│ ❌ 数据可被篡改，无法验证完整性      │
└─────────────────────────────────────┘

具体风险：

  1. 窃听风险：通信内容可被第三方监听
  2. 篡改风险：数据在传输过程中可被修改
  3. 冒充风险：服务器身份可被伪造

二、HTTPS 简介

2.1 什么是 HTTPS

HTTPS（HyperText Transfer Protocol Secure）是 HTTP 的安全版本，在 HTTP 和 TCP 之间加入了 SSL/TLS 安全协议。

HTTP 协议栈：
┌───────────┐
│   HTTP    │  应用层
├───────────┤
│   TCP     │  传输层
├───────────┤
│    IP     │  网络层
└───────────┘

HTTPS 协议栈：
┌───────────┐
│   HTTP    │  应用层
├───────────┤
│ SSL/TLS   │  安全层
├───────────┤
│   TCP     │  传输层
├───────────┤
│    IP     │  网络层
└───────────┘

三、HTTPS 为什么安全

HTTPS 通过以下机制解决了 HTTP 的三大安全问题：

3.1 加密：防止窃听

HTTPS 使用混合加密机制：

混合加密 = 对称加密 + 非对称加密

对称加密

// 加密和解密使用同一把密钥
const key = 'secret-key'

function encrypt(data, key) {
  // 使用密钥加密数据
  return encryptedData
}

function decrypt(encryptedData, key) {
  // 使用相同密钥解密
  return data
}

// 优点：速度快
// 缺点：密钥传输不安全

非对称加密

// 使用一对密钥：公钥和私钥
const publicKey = 'public-key'   // 公开
const privateKey = 'private-key' // 保密

// 公钥加密，私钥解密
function encrypt(data, publicKey) {
  return encryptedData
}

function decrypt(encryptedData, privateKey) {
  return data
}

// 优点：密钥传输安全
// 缺点：速度慢

HTTPS 的混合加密流程

客户端                                    服务器
   │                                        │
   │──── 1. 发起 HTTPS 请求 ──────────────>│
   │                                        │
   │<─── 2. 返回证书（包含公钥） ───────────│
   │                                        │
   │ 3. 生成随机对称密钥                     │
   │    用公钥加密对称密钥                   │
   │                                        │
   │──── 4. 发送加密的对称密钥 ─────────────>│
   │                                        │
   │                          5. 私钥解密获取对称密钥
   │                                        │
   │<═══════ 6. 使用对称密钥加密通信 ═══════>│
   │                                        │

3.2 身份验证：防止冒充

HTTPS 使用数字证书验证服务器身份。这是 HTTPS 安全的核心机制之一。

数字证书

数字证书包含：
├── 版本号（Version）
├── 序列号（Serial Number）
├── 签名算法（Signature Algorithm）
├── 颁发者（Issuer）
├── 有效期（Validity）
│   ├── 生效时间（Not Before）
│   └── 失效时间（Not After）
├── 使用者（Subject）
├── 公钥信息（Subject Public Key Info）
│   ├── 公钥算法
│   └── 公钥数据
├── 扩展信息（Extensions）
│   ├── 密钥用途（Key Usage）
│   ├── 增强密钥用途（Extended Key Usage）
│   ├── 主题备用名称（SAN）
│   └── 基本约束（Basic Constraints）
└── 数字签名（Signature）

数字签名原理：为什么不可伪造

数字签名是证书安全的核心。理解它是如何工作的，就能理解为什么中间人无法伪造证书。

┌─────────────────────────────────────────────────────────────┐
│                    数字签名生成过程                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   证书内容（明文）                                            │
│        │                                                    │
│        ▼                                                    │
│   ┌─────────────┐                                           │
│   │  Hash 算法  │  SHA-256                                   │
│   └─────────────┘                                           │
│        │                                                    │
│        ▼                                                    │
│   证书摘要（Hash 值）                                         │
│        │                                                    │
│        ▼                                                    │
│   ┌─────────────┐                                           │
│   │ CA 私钥加密 │  RSA/ECDSA                                 │
│   └─────────────┘                                           │
│        │                                                    │
│        ▼                                                    │
│   数字签名                                                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│                    数字签名验证过程                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   客户端收到证书后，两条路径并行计算：                         │
│（首先使用公钥把签名解析出来，然后对传入内容使用hash算法计算一下） │
│                                                              │
│   路径1：解密签名（签名里存的是加密后的摘要）                   │
│   数字签名 ──[签发者 CA 公钥解密]──→ 摘要 A                    │
│   （验证服务器证书用中间 CA 公钥；验证中间证书用根 CA 公钥）     │
│                                                             │
│   路径2：计算摘要（对明文内容自己算一遍）                       │
│   证书内容 ──[Hash 算法]──→ 摘要 B                            │
│                                                             │
│   比对：摘要 A == 摘要 B ?                                    │
│   ✅ 相等：证书未被篡改，确实由该 CA 签发                      │
│   ❌ 不等：证书被篡改或伪造                                   │
│                                                             │
│   【常见误解澄清】                                            │
│   Q: 公钥是公开的，中间人能看到，不就泄密了吗？                 │
│   A: 公钥是用来"加密"的，不是用来"解密"的。                    │
│      中间人能看到公钥，但没有私钥，无法解密用户发出去的数据。    │
│                                                             │
└─────────────────────────────────────────────────────────────┘

// 数字签名的工作原理（简化示意）
const crypto = require('crypto')

// ====== CA 签发证书 ======
const caPrivateKey = 'ca-private-key'

// 证书内容（不包含签名）
const certificateContent = {
  subject: 'example.com',
  publicKey: 'server-public-key',
  issuer: 'DigiCert',
  validFrom: '2025-01-01',
  validTo: '2026-01-01'
}

// 1. 计算证书内容的摘要
const contentHash = crypto
  .createHash('sha256')
  .update(JSON.stringify(certificateContent))
  .digest()

// 2. 用 CA 私钥对摘要进行签名
const signature = crypto
  .createSign('RSA-SHA256')
  .update(JSON.stringify(certificateContent))
  .sign(caPrivateKey)

// 完整证书 = 证书内容 + 签名
const certificate = {
  ...certificateContent,
  signature: signature
}

// ====== 客户端验证证书 ======
const caPublicKey = 'ca-public-key' // 客户端预置的 CA 公钥

// 1. 用 CA 公钥验证签名
const isValid = crypto
  .createVerify('RSA-SHA256')
  .update(JSON.stringify(certificateContent))
  .verify(caPublicKey, signature)

console.log(isValid ? '证书验证通过' : '证书无效')

// 2. 为什么不可伪造？
// - 没有CA私钥，无法生成有效的签名
// - 篡改证书内容会导致摘要不匹配
// - CA私钥是保密的，只有CA机构持有

关键点：只有拥有 CA 私钥才能生成有效的签名。攻击者没有 CA 私钥，无法伪造签名。

CA（Certificate Authority）信任体系

根 CA 证书是如何校验的

关键问题：既然每个证书都需要上级 CA 的签名来验证，那根 CA 的证书谁来验证？

答案：根 CA 证书是自签名的，并且预置在浏览器/操作系统中

┌─────────────────────────────────────────────────────────────┐
│                    根 CA 证书的特殊性                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 自签名（Self-signed）                                    │
│     根证书：签发者 = 使用者 = 根CA本身                        │
│     签名：用自己的私钥给自己签名                              │
│                                                             │
│  2. 预置信任（Pre-installed）                                │
│     - 浏览器出厂时已内置根证书                                │
│     - 操作系统也预装了信任的根证书                            │
│     - 这些根证书的公钥是"信任锚点"                            │
│                                                             │
│  3. 严格审核（Strict Auditing）                              │
│     - CA/Browser Forum 制定严格标准                          │
│     - 根 CA 需要通过 WebTrust 等审计                         │
│     - 私钥存储在 HSM（硬件安全模块）中                        │
│     - 离线存储，物理隔离                                     │
│                                                             │
└─────────────────────────────────────────────────────────────┘

// 根证书验证流程（客户端）

const rootCAValidation = {
  step1: '收到证书链',
  step2: '找到签发者（Issuer）',
  step3: '在本地证书库中查找对应的根证书',
  step4: '用根证书的公钥验证签名',
  step5: '签名有效 → 信任该证书链',
  
  // 核心原理：
  principle: `
    根证书本身不需要"验证"，因为：
    1. 它已经预置在系统中，默认被信任
    2. 它的公钥已知，可以验证它签发的证书
    3. 攻击者无法替换系统中的根证书（需要系统权限）
  `
}

验证过程示例：

1. 访问 example.com，收到证书链：
   ┌─────────────────────┐
   │ 服务器证书           │ Issuer: DigiCert TLS RSA SHA256 2020 CA1
   │ Subject: example.com│
   └─────────────────────┘
            ↓ 签发
   ┌─────────────────────┐
   │ 中间证书             │ Issuer: DigiCert Global Root CA
   │ Subject: DigiCert...│
   └─────────────────────┘
            ↓ 签发
   ┌─────────────────────┐
   │ 根证书（本地已有）    │
   │ Subject: DigiCert   │  ← 预置在浏览器/系统中
   │ Global Root CA      │
   └─────────────────────┘

2. 验证链：
   - 用中间证书的公钥 → 验证服务器证书的签名 ✅
   - 用根证书的公钥   → 验证中间证书的签名 ✅
   - 根证书已在本地信任库 → 直接信任 ✅

证书链完整验证流程

┌─────────────────────────────────────────────────────────────┐
│                    证书验证完整流程                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 检查证书基本属性                                         │
│     ├── 当前时间是否在有效期内（Not Before ~ Not After）     │
│     ├── 证书域名是否匹配（CN 或 SAN）                        │
│     ├── 密钥用途是否正确（Key Usage）                        │
│     └── 证书是否被吊销（CRL / OCSP）                         │
│                                                             │
│  2. 构建证书链                                               │
│     ├── 从服务器证书开始                                     │
│     ├── 根据 Issuer 字段找到签发者证书                       │
│     └── 直到找到根证书                                       │
│                                                             │
│  3. 验证签名链                                               │
│     ├── 用中间证书公钥验证服务器证书签名                      │
│     ├── 用根证书公钥验证中间证书签名                          │
│     └── 根证书在本地信任库中                                 │
│                                                             │
│  4. 验证通过条件                                             │
│     ├── 所有签名验证通过                                     │
│     ├── 所有证书在有效期内                                   │
│     ├── 根证书在信任库中                                     │
│     └── 证书未被吊销                                         │
│                                                             │
└─────────────────────────────────────────────────────────────┘

如何防止中间人攻击

中间人攻击（MITM）的原理：

正常 HTTPS 通信：
客户端 ←───────────────→ 服务器
        TLS 加密通道

中间人攻击尝试：
客户端 ←─── 攻击者 ───→ 服务器
        ↑           ↑
      假证书      真证书

HTTPS 如何阻止中间人攻击：

┌─────────────────────────────────────────────────────────────┐
│                中间人攻击失败的原因                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  攻击者尝试方案 1：使用自己的证书                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 攻击者生成：攻击者证书 + 攻击者私钥                    │   │
│  │ 客户端验证：证书签发者是"攻击者"                       │   │
│  │ 结果：❌ "攻击者"不在信任的 CA 列表中                  │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  攻击者尝试方案 2：伪造服务器证书                            │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 攻击者伪造：Subject=example.com 的证书                │   │
│  │ 问题：需要有效 CA 的签名                              │   │
│  │ 结果：❌ 没有CA私钥，无法生成有效签名                  │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  攻击者尝试方案 3：篡改真实证书                              │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 攻击者：修改证书中的公钥为自己的公钥                   │   │
│  │ 客户端验证：重新计算摘要 ≠ 签名解密的摘要              │   │
│  │ 结果：❌ 签名验证失败                                 │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  攻击者尝试方案 4：替换整个证书链                            │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 攻击者：提供假的证书链                                 │   │
│  │ 问题：根证书不在客户端信任库中                         │   │
│  │ 结果：❌ 证书链无法追溯到信任的根证书                   │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘

唯一可能的攻击场景（需要同时满足多个条件）：

⚠️ 以下情况中间人攻击可能成功：

1. 客户端安装了攻击者的根证书
   - 用户被诱导安装恶意根证书
   - 企业内部网络监控（需要管理员权限）
   - 恶意软件安装根证书

2. CA 被攻破或恶意签发
   - 2011年 DigiNotar 被黑客入侵
   - 伪造了 *.google.com 证书
   - 结果：DigiNotar 根证书被所有浏览器移除

3. 用户忽略证书警告
   - 浏览器显示"您的连接不是私密连接"
   - 用户点击"继续访问"
   - 此时中间人攻击成功

证书吊销机制

即使证书被正确签发，也可能需要吊销：

┌─────────────────────────────────────────────────────────────┐
│                    证书吊销原因                               │
├─────────────────────────────────────────────────────────────┤
│ - 私钥泄露或疑似泄露                                         │
│ - 证书信息变更（域名、组织等）                                │
│ - CA 被入侵或违规                                            │
│ - 证书持有人请求吊销                                         │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│                    吊销检查方式                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. CRL（证书吊销列表）                                       │
│     ┌───────────────────────────────────────────────────┐   │
│     │ CA 发布一个列表，包含所有已吊销证书的序列号         │   │
│     │ 客户端下载并缓存这个列表                           │   │
│     │ 缺点：列表可能很大，更新不及时                       │   │
│     └───────────────────────────────────────────────────┘   │
│                                                             │
│  2. OCSP（在线证书状态协议）                                  │
│     ┌───────────────────────────────────────────────────┐   │
│     │ 客户端实时查询：这个证书是否有效？                   │   │
│     │ CA 服务器响应：有效 / 已吊销 / 未知                 │   │
│     │ 缺点：暴露用户访问了哪些网站（隐私问题）             │   │
│     └───────────────────────────────────────────────────┘   │
│                                                             │
│  3. OCSP Stapling（OCSP 装订）                               │
│     ┌───────────────────────────────────────────────────┐   │
│     │ 服务器预先从 CA 获取 OCSP 响应                      │   │
│     │ 握手时一并发送给客户端                              │   │
│     │ 优点：客户端无需单独查询，保护隐私                   │   │
│     └───────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘

# Nginx 启用 OCSP Stapling
ssl_stapling on;
ssl_stapling_verify on;
resolver 8.8.8.8 8.8.4.4 valid=300s;

3.3 完整性验证：防止篡改

HTTPS 使用摘要算法（哈希算法）验证数据完整性。

摘要算法

const crypto = require('crypto')

// 常见摘要算法
const algorithms = ['MD5', 'SHA-1', 'SHA-256', 'SHA-384', 'SHA-512']

// 计算摘要
const data = 'Hello, World!'
const hash = crypto.createHash('sha256').update(data).digest('hex')

console.log(hash)
// 输出：dffd6021bb2bd5b0af676290809ec3a53191dd81c7f70a4b28688a362182986f

MAC（消息认证码）

// MAC = Hash(数据 + 密钥)
function calculateMAC(data, key) {
  const crypto = require('crypto')
  return crypto.createHmac('sha256', key)
    .update(data)
    .digest('hex')
}

// 验证流程：
// 发送方：计算 MAC，一起发送
// 接收方：重新计算 MAC，对比是否一致

4.3 密钥生成

// 密钥生成过程
function generateKeys(clientRandom, serverRandom, preMasterSecret) {
  // 1. 计算 Master Secret
  const masterSecret = PRF(preMasterSecret, 'master secret',
                          clientRandom + serverRandom)
  
  // 2. 扩展密钥
  const keyBlock = PRF(masterSecret, 'key expansion',
                       serverRandom + clientRandom)
  
  // 3. 提取各种密钥
  return {
    clientWriteKey: keyBlock.slice(0, 16),   // 客户端加密密钥
    serverWriteKey: keyBlock.slice(16, 32),  // 服务器加密密钥
    clientWriteMAC: keyBlock.slice(32, 48),  // 客户端 MAC 密钥
    serverWriteMAC: keyBlock.slice(48, 64),  // 服务器 MAC 密钥
    clientIV: keyBlock.slice(64, 76),        // 客户端初始向量
    serverIV: keyBlock.slice(76, 88)         // 服务器初始向量
  }
}

五、HTTP 与 HTTPS 的区别

5.1 核心区别对比

对比项 HTTP HTTPS
默认端口 80 443
安全性 明文传输，不安全 加密传输，安全
证书 不需要证书 需要 CA 证书
协议 HTTP + TCP HTTP + SSL/TLS + TCP
性能 较快 略慢（握手开销）
成本 免费 证书可能收费
SEO 无优势 搜索引擎优先

5.2 HTTPS 的优缺点

优点

✅ 数据加密，防止窃听
✅ 身份验证，防止冒充
✅ 完整性验证，防止篡改
✅ SEO 优化，搜索引擎优先收录
✅ 浏览器信任，显示安全锁图标
✅ 符合法规要求（GDPR 等）
✅ 提升用户信任度

缺点

❌ 需要购买证书（免费证书有限）
❌ 握手增加延迟
❌ 服务器计算开销增加
❌ 配置维护成本

6.2 Nginx 配置 HTTPS

server {
    listen 443 ssl http2;
    server_name example.com;
    
    # 证书路径
    ssl_certificate /etc/letsencrypt/live/example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/example.com/privkey.pem;
    
    # TLS 版本
    ssl_protocols TLSv1.2 TLSv1.3;
    
    # 加密套件
    ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256;
    ssl_prefer_server_ciphers off;
    
    # HSTS
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
    
    # 其他配置
    root /var/www/html;
    index index.html;
}

# HTTP 重定向到 HTTPS
server {
    listen 80;
    server_name example.com;
    return 301 https://$server_name$request_uri;
}

8.2 证书错误

常见错误：
├── NET::ERR_CERT_AUTHORITY_INVALID：证书不受信任
├── NET::ERR_CERT_DATE_INVALID：证书过期
├── NET::ERR_CERT_COMMON_NAME_INVALID：域名不匹配
└── NET::ERR_CERT_REVOKED：证书已被吊销

解决方案：
├── 更换受信任的 CA
├── 续期或更换证书
├── 确保证书域名匹配
└── 使用 OCSP 或 CRL 检查吊销状态
