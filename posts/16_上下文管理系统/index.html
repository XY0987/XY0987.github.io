上下文管理系统 — Claude Code 源码深度分析

一句话理解

Claude Code 的上下文管理是一条成本递增的 5 级压缩管线：先用最便宜的方式（裁剪旧工具结果）腾出空间，实在不够再用最贵的方式（调用 API 全量摘要）。每次对话开始前都会检查一遍，确保 context window 不会撑爆。

5 级压缩 — “成本递增的防御层”

类比：整理房间。先把大件东西搬到储藏室（工具结果存磁盘），再扔掉明显没用的东西（裁剪历史），再整理一下（清理旧结果），实在太乱了才考虑大扫除（全量摘要）。

级别 名称 做什么 成本
① Tool Result Budget 大工具结果 → 磁盘 + 预览 极低（本地 IO）
② Snip Compact 裁剪手动标记的历史片段 低（内存操作）
③ Micro-compact 清理旧 tool result 内容 低（cache_edits API）
④ Context Collapse 生成投影视图（不修改原始消息） 低（本地计算）
⑤ Auto-compact 全量摘要 + Session Memory 高（API 调用）

Token 阈值 — “什么时候触发什么”

类比：油箱警告灯。快没油时亮黄灯（warning），更少时亮红灯（blocking），中间有个自动加油点（auto-compact）。

Auto-compact 两种模式 — “有没有会议纪要”

类比：开完会要写总结。有现成的会议纪要（Session Memory）直接用，没有的话要重新回忆整理（调 API 全量摘要）。

  • 有 Session Memory → trySessionMemoryCompaction：直接用已提取的会话笔记替代全量摘要，零 API 成本，节省数秒延迟
  • 无 Session Memory → compactConversation：启动 Forked Agent 调用 API 生成摘要，成本高但效果好

prompt-too-long 恢复 — “撑爆了怎么办”

类比：行李超重。先把最重的东西拿出来（collapse drain），还超就托运（reactive compact），实在不行就报错。

核心函数详解 — “它到底做了什么”

1. applyToolResultBudget() — 工具结果预算

文件: src/utils/toolResultStorage.ts

具体做了什么:

  1. 按 API 消息分组: collectCandidatesByMessage() — 把所有工具结果按它们所在的 API 消息分组，因为 prompt cache 是以消息为单位的，同一消息内的替换要一起决策。

  2. 三态决策（每组独立）:

    • mustReapply：这条消息之前替换过 → 直接重新应用（Map lookup，零 IO）
    • frozen：之前见过但没替换 → 永远不动（保护 prompt cache 稳定）
    • fresh：第一次见到 → 可以替换

  3. 触发条件: frozen + fresh 总大小 > 200K → 从 fresh 中选最大的持久化到磁盘，返回 <persisted-output> 预览 + 路径。

  4. 状态持久化: ContentReplacementState 写入 transcript，resume 时可完整重建，不会因为重启而丢失替换状态。

为什么 frozen 永远不动: 一旦某条消息的工具结果被标记为”见过但不替换”，后续轮次改变主意会导致 prompt cache bust。frozen 语义确保决策不可逆，cache 永远稳定。

2. microcompactMessages() — 微压缩

文件: src/services/compact/microCompact.ts

具体做了什么:

  1. 时间判断: 距上次对话超过 60 分钟 → Time-Based MC：直接 content-clear 旧 tool result（把内容置空），不需要 API 调用。

  2. Cached MC（时间未到）: 使用 cache_edits API 机制——把旧 tool result 的内容替换为简短摘要，但利用 API 的 cache 编辑能力，不破坏已有的 prompt cache。

  3. 哪些 tool result 会被清理: 不是最近 N 轮的、且内容超过阈值大小的 tool result。

为什么有两种模式: Time-Based MC 适合长时间中断后恢复的场景（旧内容大概率不再需要，直接清空最省事）。Cached MC 适合连续对话（保留 cache 命中率，用 cache_edits 最小化破坏）。

3. applyCollapsesIfNeeded() — Context Collapse 投影视图

文件: src/services/compact/autoCompact.ts

这是最轻量的压缩方式，核心特点是不修改原始消息数组。

具体做了什么:

  1. 生成投影视图: 对历史消息生成摘要，存入 collapseStore（独立存储，不改 REPL 消息数组）。

  2. 发给 API 时替换: 构建 API 请求时，把被 collapse 的消息替换为摘要版本。原始消息保留在内存中。

  3. 跨 turn 持久化: collapse 结果存在 collapseStore 中，不会因为下一轮对话而失效。

  4. 触发时机: 在 auto-compact 之前运行。如果 collapse 后 token 数降到阈值以下，auto-compact 就不需要触发了。

为什么不修改原始消息: 原始消息保留意味着用户可以随时 /clear 重置，collapse 是可逆的。而且不修改消息数组不会破坏 prompt cache（cache 是基于原始消息内容计算的）。

4. autoCompactIfNeeded() — 自动压缩触发

文件: src/services/compact/autoCompact.ts

具体做了什么:

  1. 阈值计算:

    • effectiveWindow = contextWindow - min(maxOutput, 20K)
    • autoCompactThreshold = effectiveWindow - 13K（留 13K 给压缩后的摘要）
    • 当前 token 数 ≥ autoCompactThreshold → 触发

  2. 断路器检查: consecutiveAutoCompactFailures >= MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES(3) → 停止重试，防止无限循环。

  3. Session Memory 优先:

    • 有 Session Memory → trySessionMemoryCompaction()：用已提取的会话笔记直接替代全量摘要，零 API 成本
    • 无 Session Memory → compactConversation()：启动 Forked Agent 调用 API 生成摘要

  4. 压缩后清理: postCompactCleanup() — 清理 prompt cache 状态、重置 collapse store、更新 token 计数。

5. compactConversation() — 全量摘要压缩

文件: src/services/compact/compact.ts

具体做了什么:

  1. 构建压缩提示词: buildCompactPrompt() — 把当前对话历史格式化，加上”请生成一份详细摘要”的指令。

  2. 启动 Forked Agent: runForkedAgent() — 子代理共享主对话的 prompt cache（节省 token），有独立的执行状态。

  3. 生成摘要: Forked Agent 调用 API，生成包含关键信息的摘要文本。

  4. 替换消息历史: 用摘要替换原始消息历史，保留最近几轮对话（确保上下文连续性）。

  5. 注入 Session Memory: 如果有 Session Memory，把会话笔记也注入到压缩后的上下文中。

为什么用 Forked Agent 而不是直接调用 API: Forked Agent 复用主对话的 prompt cache，压缩请求的 cache 命中率极高，大幅降低压缩成本。

6. tokenCountWithEstimation() — Token 计数规范函数

文件: src/utils/tokens.ts

这是整个系统唯一的 token 计数规范函数，所有阈值比较都必须用它。

具体做了什么:

  1. 从后向前找最近的 assistant usage: 遍历消息历史，找到最近一条有 usage 字段的 assistant 消息。

  2. 处理并行 tool calls 的陷阱: 同一个 message.id 可能有多条分裂记录（并行工具调用时 API 会分裂消息）。需要回退到第一个同 id 的 assistant 消息，否则会漏算。

  3. 计算 usage total: input + cache_creation + cache_read + output（四项都要加）。

  4. 加上新消息的粗估: 该 usage 记录之后的所有新消息，用 roughTokenCountEstimation（content.length / 4，保守系数 × 4/3）估算。

  5. 无 usage 时全部粗估: 如果找不到任何 usage 记录（如对话刚开始），全部用粗略估算。

为什么不直接用 API 返回的 usage: API 的 usage 是上一轮的，当前轮新增的消息还没有 usage。必须把新消息的估算值加上去，才能准确预测下一次 API 调用的 token 数。

7. buildEffectiveSystemPrompt() — 系统提示词构建

文件: src/utils/systemPrompt.ts

具体做了什么:

按优先级链组装最终的 System Prompt（后面的覆盖前面的）：

  1. overrideSystemPrompt → 完全替代默认 prompt（最高优先级）
  2. Coordinator prompt → 多 Agent 协调者的特殊指令
  3. Agent prompt → proactive 模式追加；普通模式替换
  4. Custom prompt（--system-prompt 参数）→ 用户自定义
  5. Default prompt → getSystemPrompt() 生成的标准提示词
  6. appendSystemPrompt → 始终追加到末尾（最低优先级，但不被覆盖）

静态段 vs 动态段（SYSTEM_PROMPT_DYNAMIC_BOUNDARY 分割）:

  • 静态段（BOUNDARY 之前）：角色定义、系统行为、编码规范等，全局缓存，跨用户共享
  • 动态段（BOUNDARY 之后）：session_guidance、memory、env_info、language、mcp_instructions 等，按会话变化
  • **DANGEROUS_uncachedSystemPromptSection**：每轮重算，显式标注”这会破坏 cache”，用于 MCP 指令等每轮变化的内容

核心流程图

压缩管线执行顺序

query.ts 的每轮循环开始时，按固定顺序执行 5 级压缩检查。下面是完整的决策链路：

第一步：工具结果预算（最先执行，成本最低）

大工具结果存磁盘，只保留预览。这一步几乎没有成本，但能显著减少 context 大小。

第二步：Snip Compact

裁剪用户手动标记的历史片段（/snip 命令标记的范围）。

第三步：Micro-compact

清理旧 tool result。时间久了（>60min）直接清空；时间未到用 cache_edits 最小化破坏。

第四步：Context Collapse

生成投影视图，不修改原始消息，把被折叠的消息替换为摘要版本，不会影响PromptCache。如果 collapse 后降到阈值以下，第五步就不需要了。

第五步：Auto-compact

最后的手段。优先用 Session Memory（零 API 成本），没有则调 API 全量摘要，把所有历史消息全量压缩，会影响PromptCache。

Token 管理数据流

tokenCountWithEstimation(messages) — 规范函数
├─ 从后向前找最近的 assistant 的 usage
├─ 处理并行 tool calls (同 message.id 的分裂记录)
│   → 回退到第一个同 id 的 assistant
├─ usage total = input + cache_creation + cache_read + output
├─ + 该记录之后所有新消息的 roughTokenCountEstimation
└─ 无 usage → 全部粗略估算 (content.length / 4)

calculateTokenWarningState(tokenUsage, model)
├─ percentLeft: ((threshold - tokenUsage) / threshold) * 100
├─ isAboveWarningThreshold: tokenUsage >= threshold - 20K
├─ isAboveAutoCompactThreshold: tokenUsage >= autoCompactThreshold
└─ isAtBlockingLimit: tokenUsage >= effectiveWindow - 3K

Tool Result Budget 数据流

每个工具执行后:
  processToolResultBlock(tool, result, toolUseID)
  ├─ size <= getPersistenceThreshold(tool) → 原样返回
  └─ size > threshold → persistToolResult()
      ├─ 写入 tool-results/{toolUseId}.txt
      └─ 返回 <persisted-output> 预览 + 路径

每轮 query loop 开始:
  applyToolResultBudget(messages, state)
  ├─ collectCandidatesByMessage → 按 API 消息分组
  ├─ 每组独立检查:
  │   ├─ mustReapply: 之前替换过 → 重新应用 (Map lookup, 0 I/O)
  │   ├─ frozen: 之前见过未替换 → 不动 (cache 稳定)
  │   └─ fresh: 新见到 → 可替换
  ├─ frozen + fresh > 200K → 选最大的 fresh 持久化
  └─ 状态写入 transcript → resume 时可重建

设计模式总结

模式 实现细节 为什么这样设计
多级缓存淘汰 Snip → MC → Collapse → Autocompact，成本递增 大多数情况下低成本方案就够用，只有极少数情况才需要高成本压缩。按成本排序让平均成本最低。
投影视图 Context Collapse 不修改 REPL 消息数组，摘要存 collapseStore 不修改原始消息意味着 collapse 可逆，且不破坏 prompt cache（cache 基于原始内容计算）。
Session Memory Compact 用已提取的会话笔记替代 API 调用 会话笔记已经存在，直接复用是零成本的。避免了一次 API 调用（数秒延迟 + token 费用）。
Forked Agent 缓存共享 compact 使用 runForkedAgent 复用 prompt cache 子代理不需要重新发送整个上下文，直接复用主对话的 cache，压缩请求的 cache 命中率极高。
systemPromptSection 缓存 BOUNDARY 之前全局缓存，DANGEROUS_uncached 每轮重算 静态段内容不变，全局缓存跨用户共享。MCP 指令每轮可能变化，必须重算，显式标注避免误用。
State 原子更新 每个 continue 站点 state = {...state, newValues} 不可能只更新部分字段导致不一致。展开语法确保每次更新都是完整的新对象。
断路器 MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES = 3 连续失败说明压缩本身有问题（如 API 错误），继续重试没有意义。断路器防止无限循环消耗资源。
prompt-too-long 多级恢复 collapse drain → reactive compact → surface error 先轻量后重量，最后才暴露错误。用户感知到的是”自动处理好了”而不是”报错了”。
Mark-after-filter 去重 Tool Result Budget 的 seenIds/replacements 一旦见过则命运固定（frozen 或 replaced），后续轮次不改变决策，保护 prompt cache 稳定性。
BOUNDARY 分割 SYSTEM_PROMPT_DYNAMIC_BOUNDARY 分割静态/动态段 静态段可全局缓存（跨所有用户共享同一 cache），动态段按用户/会话变化。最大化 cache 命中率。

对 Agent 开发的参考价值

1. 分层压缩策略

  • 成本递增设计是关键：先尝试低成本方案（content-clear、投影视图），最后才用高成本方案（API 摘要）。每一级都是独立的”保险丝”，前一级成功就不触发后一级。
  • Context Collapse 运行在 auto-compact 之前：如果 collapse 降到阈值以下，auto-compact 不触发，节省一次 API 调用。
  • Session Memory Compact 是创新：利用已有的会话笔记替代 API 调用，零成本压缩。

2. Token 管理

  • tokenCountWithEstimation 是唯一的规范函数，所有阈值比较都用它，避免不同地方用不同方法导致不一致。
  • 并行 tool calls 的陷阱：同 message.id 的分裂记录需要回退到第一个，否则漏算。这是 Anthropic API 的特殊行为，需要特别处理。
  • 粗估 × 4/3 填充：roughTokenCountEstimation 的保守系数，宁可高估也不低估，避免触发 prompt-too-long。

3. System Prompt 构建

  • 静态/动态分段：BOUNDARY 之前可全局缓存（跨用户共享），之后按会话变化。设计自己的 System Prompt 时，把不变的内容放前面，变化的内容放后面，最大化 cache 命中率。
  • **DANGEROUS_uncachedSystemPromptSection**：每轮重算，显式标注缓存破坏原因。任何需要每轮更新的内容都应该显式标注，防止误以为被缓存了。
  • buildEffectiveSystemPrompt 的优先级链：override > coordinator > agent > custom > default。设计多层 prompt 覆盖时，明确每层的优先级，避免隐式覆盖。

4. 工具结果预算

  • 两级预算：单工具级（50K chars）+ 单消息聚合级（200K chars）。单工具级防止单个大输出撑爆，聚合级防止多个中等输出累积撑爆。
  • State 持久化：ContentReplacementState 写入 transcript，resume 时重建。任何影响 context 的状态都应该持久化，否则重启后状态丢失。
  • Cache 稳定性：frozen 结果永不被新替换，防止 cache bust。一旦做出决策就不改变，是保护 prompt cache 的通用原则。

5. 恢复机制设计

  • prompt-too-long: 先 collapse drain（轻量）→ reactive compact（重量）→ 错误。渐进升级是通用模式：先尝试最便宜的修复。
  • max_output_tokens: 先 escalate 8K→64K → multi-turn recovery ×3 → 错误。输出截断时先扩大输出限制，再多轮续写，最后才报错。
  • 断路器：连续失败超阈值后停止重试。任何自动重试机制都应该有断路器，防止无限循环。
