工具调用系统 — Claude Code 源码深度分析

一句话理解

LLM 返回”我要调用工具”，系统先检查你有没有权限，再决定并行还是串行跑，跑完把结果塞回给 LLM 继续思考。整个过程有 Hook 可以拦截，有 ToolSearch 可以按需加载工具 schema，节省 token。

白话版运行流程

整体一条线

LLM 返回 ToolUseBlock[]
        │
        ▼
  partitionToolCalls()
  "这些工具能不能一起跑？"
        │
   ┌────┴────┐
   ▼         ▼
并行跑     串行跑
(只读工具)  (写入工具)
   │         │
   └────┬────┘
        ▼
  runToolUse() × N
  "跑单个工具"
        │
   ┌────┴──────────────┐
   ▼                   ▼
权限检查通过         权限不够
   │                   │
tool.call()         拒绝/弹确认框
   │
   ▼
结果 → 塞回 LLM

权限检查 — “你能不能干这件事”

类比：公司门禁系统。进门要刷卡，但有些区域还要额外审批，有些区域任何人都进不去。

三道关卡（按顺序，任一拦截就停）：

  1. 第一关：规则检查（最高优先级，bypassPermissions 也绕不过）

    • 有 deny 规则 → 直接拒绝
    • 有 ask 规则 → 弹确认框
    • 操作 .git/、.claude/ 等安全路径 → 弹确认框

  2. 第二关：模式检查

    • bypassPermissions 模式 → 直接放行
    • 工具在白名单里 → 直接放行

  3. 第三关：AI 分类器（auto 模式下）

    • 调用 classifyYoloAction() 让 AI 判断这个操作危不危险
    • 危险 → 拒绝；安全 → 放行

ToolSearch — “按需加载工具说明书”

类比：图书馆的书目索引。你不需要把所有书都搬到桌上，先看目录，需要哪本再去取。

为什么需要：Claude Code 有 40+ 个 MCP 工具，每个工具的 schema（参数说明）都很长。如果每次请求都把所有工具的完整 schema 发给 API，会浪费大量 token。

怎么运作：

  1. 初始请求：只发工具名称（defer_loading: true），不发完整 schema
  2. LLM 看到工具名列表，觉得需要某个工具
  3. LLM 调用 ToolSearch("select:mcp__github__create_issue") 查询
  4. 系统返回该工具的完整 schema，注入上下文
  5. LLM 现在可以正常调用这个工具了

并发 vs 串行 — “哪些工具可以一起跑”

类比：厨房里同时炒几个菜。只读操作（看菜谱、量食材）可以同时做；写入操作（切菜、炒菜）要一步一步来，不然乱套。

  • isConcurrencySafe = true（只读工具）→ 最多 10 个并行
  • isConcurrencySafe = false（写入工具）→ 串行执行

以下是源码级别的详细分析，如果你只想了解整体思路，上面的部分已经足够。

核心函数详解 — “它到底做了什么”

1. buildTool() — 工具工厂函数

文件: src/Tool.ts

具体做了什么:

  1. 注入安全默认值: 所有未显式声明的字段都走最保守路径：

    • isConcurrencySafe = false（串行，防止并发写冲突）
    • isReadOnly = false（视为写操作，需要更严格权限）
    • isEnabled = () => true（默认启用）
    • validateInput = () => null（默认不做额外校验）

  2. Zod schema 绑定: 将工具的 inputSchema（Zod 类型）与工具定义绑定，后续 toolToAPISchema() 会把它转成 JSON Schema 发给 API。

  3. 返回冻结对象: 防止运行时意外修改工具定义。

为什么这样设计: 新工具开发者如果忘记声明某个字段，系统自动走最保守路径，而不是走最宽松路径。这是 fail-closed 原则的体现。

2. assembleToolPool() — 组装工具池

文件: src/tools.ts

具体做了什么:

  1. 获取内置工具: 调用 getTools(permissionContext) → 内部调用 getAllBaseTools() 获取 43+ 个内置工具，再用 filterToolsByDenyRules() 根据用户配置的 deny 规则移除被禁用的工具，最后用 isEnabled() 过滤。

  2. 获取 MCP 工具: 从 appState.mcp.tools 读取已连接的 MCP 服务器工具，同样经过 filterToolsByDenyRules() 过滤。

  3. 合并去重: uniqBy([...builtInSorted, ...mcpSorted], 'name') — 内置工具在前，MCP 工具在后，按名称去重（内置优先）。

  4. 内置工具排在前面的原因: 保持 API 请求中工具列表的顺序稳定，最大化 prompt cache 命中率。如果 MCP 工具排在前面，每次 MCP 工具变化都会导致整个工具列表的 cache 失效。

3. runTools() — 工具编排入口

文件: src/services/tools/toolOrchestration.ts

具体做了什么:

  1. 接收 ToolUseBlock[]: LLM 一次可能返回多个工具调用请求。

  2. 调用 partitionToolCalls(): 把工具调用列表分成若干批次：

    • 扫描列表，遇到 isConcurrencySafe = false 的工具就”切一刀”
    • 切出来的每一段：如果全是 concurrencySafe 的 → 并行批；否则 → 串行批
    • 例：[Read, Read, Bash, Read] → [[Read, Read], [Bash], [Read]]

  3. 按批次执行: 并行批用 Promise.all() 同时跑（最多 10 个），串行批逐个跑。

  4. 收集结果: 所有 ToolResultBlock 汇总后返回，顺序与输入的 ToolUseBlock[] 一致。

4. runToolUse() — 单工具执行

文件: src/services/tools/toolExecution.ts

这是执行单个工具调用的核心函数，包含完整的校验→Hook→权限→执行→结果处理链路。

具体做了什么（逐步）:

  1. 查找工具: findToolByName(name) — 先按主名查找，找不到再查 alias（工具重命名后旧 transcript 仍可用）。

  2. 解析输入: inputSchema.safeParse(rawInput) — 用 Zod 校验 LLM 传来的参数。如果校验失败，返回 buildSchemaNotSentHint（提示 LLM 先用 ToolSearch 加载完整 schema）。

  3. 自定义校验: tool.validateInput(parsedInput) — 工具自己的业务逻辑校验（如路径合法性）。

  4. backfillObservableInput(): 浅克隆输入对象，注入衍生字段（如从路径推断出的文件类型）。原始 input 不突变，保证 prompt cache 稳定。

  5. 投机性分类器启动（仅 Bash 工具）: 在 Pre-Hook 运行前就启动 classifyYoloAction()，与 hooks 并行执行，用 hooks 的运行时间掩盖分类器的延迟。

  6. 运行 Pre-Hook: runPreToolUseHooks() — 外部 Hook 可以在这里 allow/deny/ask/修改 input。

  7. 解析 Hook 决策: resolveHookPermissionDecision() — 根据 Hook 结果决定是否继续走完整权限管线（详见下文）。

  8. 执行工具: tool.call(input, context) — 实际执行工具逻辑。

  9. 运行 Post-Hook: runPostToolUseHooks() — 可审计/修改工具输出（常用于 MCP 工具的输出过滤）。

  10. 处理结果: processToolResultBlock() — 截断超大输出（maxResultSizeChars），持久化到 transcript。

  11. 错误处理: 执行失败 → runPostToolUseFailureHooks() → 返回错误信息给 LLM（不抛出，让 LLM 自己决定如何处理）。

5. hasPermissionsToUseTool() — 权限决策管线

文件: src/utils/permissions/permissions.ts

这是整个权限系统的核心，三层决策，任一层拦截就短路返回。

具体做了什么（逐步）:

第零步: 检查 abortController.signal.aborted → 如果已中止，立即抛 AbortError。

第一层：规则检查（bypassPermissions 也无法跳过）:

  • getDenyRuleForTool() — 检查 settings.json 中的 deny 规则，匹配则直接拒绝
  • getAskRuleForTool() — 检查 ask 规则，匹配则弹确认框
  • tool.checkPermissions() — 工具自定义权限逻辑（如 Bash 工具检查命令是否在白名单）
  • safetyCheck() — 检查是否操作 .git/、.claude/ 等安全关键路径，是则弹确认框（bypassPermissions 对此免疫）

第二层：模式快速放行:

  • bypassPermissions 或 plan+bypass 模式 → 直接 ALLOW
  • toolAlwaysAllowedRule() — 工具在白名单 → 直接 ALLOW

第三层：模式相关处理（passthrough 到这里的请求）:

  • dontAsk 模式: ask → 直接 DENY（无头环境不能弹框）
  • auto 模式:
    • requiresUserInteraction → ASK
    • acceptEdits 快速路径 → ALLOW（用户已接受所有编辑）
    • 安全工具白名单 → ALLOW
    • classifyYoloAction() AI 分类器 → shouldBlock=true 则 DENY，否则 ALLOW
  • shouldAvoidPermissionPrompts（无头代理）: 先跑 runPermissionRequestHooksForHeadlessAgent()，没有 Hook 决策则 DENY

6. resolveHookPermissionDecision() — Hook 安全屏障

文件: src/services/tools/toolExecution.ts

这是 Hook 系统的安全不变量：Hook 的 allow 决策不能绕过用户配置的 deny/ask 规则。

具体做了什么:

  • Hook 返回 allow:

    1. 如果工具 requiresUserInteraction 且 Hook 没有修改 input → 走完整权限管线（不能绕过）
    2. 调用 checkRuleBasedPermissions():
      • 无规则冲突 → 允许（跳过弹框，但 deny/ask 规则仍然检查过了）
      • 有 deny 规则 → 拒绝（deny 规则覆盖 Hook 的 allow！）
      • 有 ask 规则 → 走完整权限管线（ask 规则要求用户确认）

  • Hook 返回 deny: 直接拒绝，不再走后续流程。

  • Hook 返回 ask 或未返回: 走完整权限管线（canUseTool()），可能带 forceDecision。

7. ToolSearchTool.call() — 按需加载工具 Schema

文件: src/tools/ToolSearchTool/ToolSearchTool.ts

具体做了什么:

  1. 解析查询: 检查是否有 select: 前缀：

    • select:mcp__github__create_issue → 精确查找，直接返回该工具的完整 schema
    • 普通关键词 → searchToolsWithKeywords() 评分排序

  2. 关键词评分搜索:

    • 工具名称部分匹配 → 加分
    • description 词边界匹配 → 加分
    • searchHint 字段匹配 → 加分
    • 按分数降序返回 Top-N 结果

  3. 返回 tool_reference 块:

    { "type": "tool_result", "content": [
  { "type": "tool_reference", "tool_name": "mcp__github__create_issue" }
]}

    API 层面会把 tool_reference 展开为完整 schema，注入到上下文中。

  4. 追踪已发现工具: extractDiscoveredToolNames(messages) 扫描消息历史，把已经通过 ToolSearch 发现的工具名记录下来。后续 API 请求中，这些工具的完整 schema 会自动包含（不再 defer）。

  5. 工具调用时Zod验证不通过（如：缺少必需参数、意外参数、类型不匹配），会将错误信息传递给大模型，让大模型自己修正

核心流程图

工具执行管线

runTools() 是整个工具执行的”调度中心”。LLM 每次返回工具调用请求，都要经过这里的分区→并发/串行→单工具执行链路。下面用文字走一遍完整决策链路：

第一步：分区

partitionToolCalls() 把工具调用列表切成若干批次。规则很简单：遇到 isConcurrencySafe = false 的工具就切一刀。切出来的每一段，如果全是并发安全的就并行跑，否则串行跑。

第二步：执行

  • 并行批：Promise.all() 同时跑，最多 10 个
  • 串行批：逐个跑，等上一个完成再跑下一个

第三步：单工具执行

每个工具调用经过：查找 → 解析 → 校验 → backfill → 投机分类器 → Pre-Hook → 权限检查 → 执行 → Post-Hook → 结果处理。

ToolSearch 延迟加载流程

初始 API 请求:
├─ 所有 isDeferredTool() = true 的工具:
│   → 只发送 name (defer_loading: true)
│   → 不发送完整 schema → 节省大量 token
├─ 非 deferred 工具: 完整 schema
└─ ToolSearchTool: 完整 schema (自身不 defer)

模型需要工具时:
  1. 看到 <system-reminder> 中的 deferred 工具名列表
  2. 调用 ToolSearch(query: "select:mcp__github__create_issue")
  3. ToolSearchTool.call():
     ├─ "select:" 前缀 → 精确查找
     └─ 关键词 → searchToolsWithKeywords() 评分排序
  4. 返回 tool_reference 块:
     { type: 'tool_result', content: [
       { type: 'tool_reference', tool_name: 'mcp__...' }
     ]}
  5. API 展开 tool_reference → 完整 schema 注入上下文
  6. 模型现在可以正常调用该工具

后续请求:
  extractDiscoveredToolNames(messages) 追踪已发现工具
  → 已发现工具的完整 schema 在后续请求中自动包含

  │   └─ isEnabled() 过滤
  ├─ filterToolsByDenyRules(mcpTools) — 允许的 MCP 工具
  └─ uniqBy([...builtInSorted, ...mcpSorted], 'name')
      → 去重（内置优先）, 内置在前保持 cache 稳定

Hook 安全不变量

resolveHookPermissionDecision(hookResult, tool, input, context, canUseTool):

hookResult.behavior === 'allow':
  ├─ requiresUserInteraction && !updatedInput → canUseTool() (完整管线)
  ├─ checkRuleBasedPermissions():
  │   ├─ null (无规则冲突) → 允许 (bypass prompt)
  │   ├─ deny → 拒绝 (deny 规则覆盖 hook!)
  │   └─ ask → canUseTool() (ask 规则要求对话)
  └─ 关键: Hook 的 allow 不能绕过 settings.json 的 deny/ask 规则

hookResult.behavior === 'deny':
  └─ 直接拒绝

hookResult.behavior === 'ask' | undefined:
  └─ canUseTool() (正常权限流, 可能带 forceDecision)

设计模式总结

模式 实现细节 为什么这样设计
Fail-closed 默认 buildTool(): isConcurrencySafe=false, isReadOnly=false 新工具开发者忘记声明属性时走最保守路径，而不是最宽松路径。安全漏洞比性能损失更难接受。
并发/串行自动分区 partitionToolCalls() 根据 isConcurrencySafe 分批，并行批最多 10 个 只读工具并行提速，写入工具串行保序。调用方无需手动管理并发，系统自动做最优决策。
投机性分类器 Bash 工具在权限检查前启动 classifyYoloAction()，与 hooks 并行 AI 分类器有网络延迟，提前启动让延迟被 hooks 的运行时间掩盖，用户感知不到等待。
ToolSearch 延迟加载 defer_loading: true → 只发名称，按需加载完整 schema 40+ MCP 工具的完整 schema 可能占数千 token。延迟加载让初始请求极轻量，只有真正用到的工具才消耗 token。
多层权限流水线 deny → tool-level → checkPermissions → rules → hooks → classifier → user 每层短路，覆盖全粒度。规则检查（纯内存）排在 AI 分类器（网络请求）前面，大多数请求在前两层就能决策。
Alias 回退 findToolByName 先查主名再查 alias 工具重命名后，旧 transcript 中记录的工具名仍然可以找到对应实现，保证历史记录可回放。
backfillObservableInput 浅克隆 + 注入衍生字段，原始 input 不突变 原始 input 保持不变，prompt cache 不会因为衍生字段的注入而失效。同时向后兼容旧版本的 input 格式。
Mark-after-filter 去重 Tool Result Budget 的 seenIds + replacements 决策一旦做出不可逆，同一 tool_use_id 的结果不会被重复处理，prompt cache 稳定。
Hook 安全屏障 resolveHookPermissionDecision 中 hook allow 仍检查 rules Hook 是第三方扩展点，不能让第三方 Hook 绕过用户在 settings.json 中配置的 deny/ask 规则。这是硬编码的安全不变量。
关键词评分搜索 ToolSearch 的 name 部分匹配 + description 词边界匹配 + searchHint 加权 精准工具发现，避免返回不相关工具浪费 token。searchHint 让工具作者可以自定义搜索关键词。

对 Agent 开发的参考价值

1. 工具定义最佳实践

  • buildTool() 提供安全默认值，减少遗漏风险。新工具至少要显式声明 isConcurrencySafe 和 isReadOnly，否则走最保守路径（串行 + 写操作权限）。
  • isConcurrencySafe = true: 只读操作设 true 可大幅提速，多个只读工具可以并行执行。
  • maxResultSizeChars: 控制工具输出大小，防止上下文膨胀。大输出工具（如文件读取）必须设置。
  • shouldDefer + searchHint: MCP 工具设延迟加载，节省 token。searchHint 填写工具的使用场景关键词，提升 ToolSearch 的召回率。

2. 权限系统设计

  • 分层短路是高效设计：每层检查完立即返回，不走后续层。规则检查（纯内存）排在 AI 分类器（网络请求）前面，大多数请求在前两层就能决策。
  • safetyCheck bypass 免疫：.git/、.claude/ 等安全关键路径即使 bypassPermissions 也要确认，这是不可绕过的硬编码保护。
  • Denial 追踪 + 熔断：连续拒绝超阈值后退化为人工确认，防止 AI 分类器误判导致的无限循环。
  • 投机性并行：分类器可在 hooks 运行时同步启动，用 hooks 的运行时间掩盖分类器延迟。

3. MCP 集成

  • ...MCPTool 展开模式：一个模板对象 + 每个工具覆盖特定方法，减少重复代码。
  • mcp__server__tool 命名约定：支持服务器级规则匹配（如 deny: mcp__github__* 禁用整个 GitHub MCP 服务器）。
  • **passthrough + suggestions**：MCP 工具默认要权限，建议在工具定义中包含 localSettings allow 规则，减少用户手动配置。

4. ToolSearch 优化

  • defer_loading 让 API 只看到工具名称，按需加载完整 schema。适合 schema 较长的 MCP 工具。
  • tool_reference 是 API 层面的懒加载机制，由 Anthropic API 负责展开，不需要客户端额外处理。
  • extractDiscoveredToolNames 追踪已发现工具集，compact 时快照到 boundary，保证压缩后已发现的工具仍然可用。

5. Hook 系统

  • PreToolUse: 可 allow/deny/ask/modify input/prevent continuation。适合实现审计日志、输入过滤、自定义权限逻辑。
  • PostToolUse: 可审计/修改 MCP 工具输出。适合过滤敏感信息、格式化输出。
  • 关键安全约束: Hook 的 allow 不能绕过 deny/ask 规则——这是硬编码的安全不变量。设计自己的 Hook 时不要假设 allow 决策一定会被执行。
