AI 自进化：从"会执行"到"会学习"

发表于2026|更新于2026|AI相关
|字数总计:4.4k|阅读时长:14分钟

前言：AI有个关键词是”自进化”。很多人以为自进化就是”模型自己微调自己”，其实完全不是。这篇单独把”AI 自进化”拎出来，说清楚它到底是什么、目前业界都在怎么做、哪些是真能落地的，哪些还只是概念。

先把误解说清楚

说到”AI 自进化”，最常见的误解有三个：

误解 真相
自进化 = 模型自己训练自己 绝大多数”自进化”不动模型权重，进化的是 Agent 外层
自进化 = 自主意识 和意识无关，本质是带反馈的循环优化
自进化 = 越用越强，没有天花板 有明确天花板，且可能越学越偏

我的理解是：当前能落地的”AI 自进化”，指的是 Agent 层面的自我改进——记忆沉淀、技能生成、策略调整、反馈闭环，而不是底层大模型参数的自我更新。

底层模型的自我训练（self-play、RLAIF 等）也存在，但那是训练阶段的事，普通开发者碰不到。我们聊的是推理阶段、应用层的自进化。

自进化的四个层次

我把当前 AI 自进化的实现方式，按”改动深度”从浅到深分成四层：

┌─────────────────────────────────────────────┐
│  L4: 模型自我微调（self-finetune）          │  改权重、门槛高
├─────────────────────────────────────────────┤
│  L3: 技能/工具自动生成（auto-skill）        │  改能力边界
├─────────────────────────────────────────────┤
│  L2: 策略/提示词自优化（prompt-evolve）     │  改执行策略
├─────────────────────────────────────────────┤
│  L1: 记忆沉淀（memory）                     │  改上下文输入
└─────────────────────────────────────────────┘

越往下越轻，越往上越重。下面一层一层说。

L1：记忆沉淀——最基础也最实用

这是所有自进化的底座。原理极简：把每次交互的关键信息存起来，下次调用时作为上下文喂给模型。

// 最小可运行示例：基于向量的长期记忆
import OpenAI from 'openai';
import { ChromaClient } from 'chromadb';

const openai = new OpenAI();
const chroma = new ChromaClient();

async function remember(userId: string, content: string) {
  const collection = await chroma.getOrCreateCollection({ name: userId });
  const embedding = await openai.embeddings.create({
    model: 'text-embedding-3-small',
    input: content,
  });
  await collection.add({
    ids: [`mem_${Date.now()}`],
    embeddings: [embedding.data[0].embedding],
    documents: [content],
  });
}

async function recall(userId: string, query: string, topK = 3) {
  const collection = await chroma.getOrCreateCollection({ name: userId });
  const embedding = await openai.embeddings.create({
    model: 'text-embedding-3-small',
    input: query,
  });
  const result = await collection.query({
    queryEmbeddings: [embedding.data[0].embedding],
    nResults: topK,
  });
  return result.documents[0];
}

// 使用时
await remember('alice', '用户偏好：不喜欢过长的回复，偏好 bullet 列表');
const memories = await recall('alice', '用户偏好是什么');
// 把 memories 拼进 System Prompt

关键点：记忆不是越多越好，而是”该记的记、该忘的忘”。

记忆系统真正难的不是存，是忘。坑1：无限累积记忆会让向量检索变噪音池，老记忆和新偏好冲突时模型会懵。解决方案是加”时间衰减 + 去重合并 + 显式遗忘指令”。

Claude Code 的 CLAUDE.md、Cursor 的 Rules、OpenClaw 的 Rules 文件——本质都是 L1 记忆，只是以 Markdown 的形式由用户手动维护。Hermes 的进步在于它把这层自动化了。

L2：策略/提示词自优化

这一层在业界有个名字叫 Prompt Evolution 或 Self-Refinement。做法是：Agent 执行完任务后，对本次的提示词和策略做自我评估，下次调用时自动调整。

典型实现模式：

// 简化版：基于反馈自优化 System Prompt
async function runWithEvolution(task: string, userFeedback?: string) {
  // 1. 读取当前策略（初始是通用版）
  let strategy = await loadStrategy('code_review');

  // 2. 执行任务
  const result = await llm.chat({
    system: strategy,
    user: task,
  });

  // 3. 如果有反馈，触发进化
  if (userFeedback) {
    const newStrategy = await llm.chat({
      system: '你是提示词优化专家。根据用户反馈，改进下面的 System Prompt。',
      user: `原 Prompt：${strategy}\n任务：${task}\n产出：${result}\n反馈：${userFeedback}\n\n请输出改进后的 Prompt。`,
    });
    await saveStrategy('code_review', newStrategy);
  }

  return result;
}

这种模式的代表是 DSPy、TextGrad、PromptBreeder 等框架。它们本质上把”调提示词”这件事自动化了——以前你自己调 Prompt，现在让 AI 调。

我的实战感受：L2 的效果和任务类型强相关。格式化任务（摘要、分类、翻译）进化效果好，因为评估标准清晰；开放性任务（代码生成、写作）效果一般，因为”好”没有明确标准，容易越改越离谱。

L3：技能/工具自动生成

这是 Hermes 宣称的核心卖点，也是自进化里最”性感”的一层：Agent 在完成任务的过程中，把可复用的操作抽象成一个新工具，下次遇到类似场景直接调用。

原理示意：

第一次执行任务                     第二次执行任务
┌──────────────────┐               ┌──────────────────┐
│ 用户：帮我下载 B │               │ 用户：帮我下载  │
│ 站视频并转成MP3  │               │ 抖音视频转MP3   │
└────────┬─────────┘               └────────┬─────────┘
         ▼                                   ▼
┌──────────────────┐               ┌──────────────────┐
│ Agent 摸索：     │               │ Agent 直接调用： │
│ 1. 装 yt-dlp     │    沉淀后      │ video_to_mp3(    │
│ 2. 下载视频       │   ──────▶    │   url, platform  │
│ 3. 装 ffmpeg     │               │ )                │
│ 4. 转码          │               └──────────────────┘
│ 5. 交付          │                     ↑
└────────┬─────────┘                     │
         ▼                                │
 抽象出新 Skill ──────────────────────────┘
 "video_to_mp3"

关键是”什么时候抽象“和”抽象到什么粒度“。抽象粒度太细，每个小动作都变 Skill，库里全是垃圾；粒度太粗，抽象出来的 Skill 下次场景稍微变一点就用不了。

坑2：自动生成的 Skill 质量不稳定。Hermes 早期版本经常生成带硬编码路径、不处理异常的 Skill，运行一次就崩。现在通过”生成 → 沙盒试运行 → 评估 → 决定是否入库“的流水线缓解，但没根治。

我的看法：L3 真正能稳定落地的场景，目前只在高度结构化的工作流里（运维脚本、数据处理、CI/CD）。通用场景的自动 Skill 生成，还远没到”开箱即用”的程度。

L4：模型自我微调

最重的一层：Agent 把交互数据收集起来，定期对底层模型做微调（LoRA、SFT、DPO 等）。

运行期收集：   (用户输入, 最终产出, 用户反馈/评分)
                          │
                          ▼
                  ┌──────────────┐
                  │ 数据清洗去重  │
                  └──────┬───────┘
                         ▼
                  ┌──────────────┐
                  │ 构造训练对   │  好产出 vs 差产出
                  └──────┬───────┘
                         ▼
                  ┌──────────────┐
                  │ LoRA / DPO  │  增量训练
                  └──────┬───────┘
                         ▼
                  ┌──────────────┐
                  │ 评估 & 发布  │
                  └──────────────┘

这条路径的门槛是真的高：需要训练算力、数据质量、评估体系三样都具备。一般团队和个人都碰不上。

目前能看到 L4 实质落地的场景：

  • 大厂内部 Copilot（如 GitHub Copilot 的企业版，基于用户代码风格微调）
  • 垂直领域 Agent（金融、医疗客服，定期用业务数据再训练）
  • 部分开源 Agent 框架（SWE-agent 等会沉淀解题数据用于研究）

对大多数应用开发者来说，L4 是”了解即可”的层次。

L1~L4 能力对比

层级 改动对象 技术门槛 见效速度 稳定性 代表产品
L1 记忆沉淀 上下文 低 快 高 Cursor、Claude Code、ChatGPT Memory
L2 策略进化 Prompt / 参数 中 中 中 DSPy、TextGrad
L3 技能生成 工具集 高 慢 低-中 Hermes、Voyager
L4 模型微调 模型权重 极高 很慢 高 GitHub Copilot 企业版

我的实战建议：90% 的应用层自进化，把 L1 做扎实就已经很强了。L2 选择性引入，L3 看场景，L4 绝大多数人不用碰。

工程化落地：从理论到能跑的系统

L1~L4 是分层认知，但真要做一套自进化系统跑起来，光知道分层不够。最近看到一个不错的工程化思路（来自社区开源的 self-improving-agent 和 EvoMap evolver），可以拆成三层闭环 + 一套协议，值得借鉴。

三层闭环架构

┌────────────────────────────────────────────────────────┐
│  日常使用                                              │
│   ↓ 产生信号（错误、纠正、发现更好方法）                │
│  ┌────────────────────────────────────────────────┐   │
│  │  L1：信号捕获层（Signal Capture）              │   │
│  │  - .learnings/ERRORS.md    技术错误            │   │
│  │  - .learnings/LEARNINGS.md 用户纠正/经验       │   │
│  │  - memory/YYYY-MM-DD.md    每日运行日志        │   │
│  └─────────────────────┬──────────────────────────┘   │
│                        ▼                              │
│  ┌────────────────────────────────────────────────┐   │
│  │  L2：进化引擎层（Evolution Engine）            │   │
│  │  扫描信号 → 加权评分 → 选择策略 → 生成指令     │   │
│  └─────────────────────┬──────────────────────────┘   │
│                        ▼                              │
│  ┌────────────────────────────────────────────────┐   │
│  │  L3：进化目标层（被改进的 SOP / 知识库 / Skill）│  │
│  └─────────────────────┬──────────────────────────┘   │
│                        ▼                              │
│  下次执行质量提升 → 又产生新信号（闭环）                │
└────────────────────────────────────────────────────────┘

核心思路：信号是自然语言写的，不要给用户增加结构化负担。

捕获信号最聪明的设计是用”信号词 + 方括号”模式，而不是搞 JSON Schema：

# memory/2026-04-26.md（自然语言日志，含信号词）

09:30 执行策略早会任务
  - [error] 数据源 API 返回 403
  - [timeout] 重试 3 次失败
  - [workaround] 临时切到备用源

10:15 用户反馈：早会摘要太长，希望控制在 200 字内

进化引擎只要 grep 这些方括号关键词，就能识别”该修哪类问题”。极轻量，零侵入。

三种进化策略（Gene）

先解释一下 Gene（基因） 这个词。这套方案借用了生物学比喻——既然叫”进化”，那就需要类似”基因”的东西来决定怎么进化：

Gene = 一个可复用的”进化策略模板”，规定了”什么信号该触发、用什么步骤改、改完怎么验证”。

可以简单理解成”进化版的 if-else 规则“：信号是 if 条件，Gene 里的 strategy 是 then 动作。一个 Gene 长这样（简化版）：

{
  "id": "gene_repair_from_errors",
  "category": "repair",
  "signals_match": ["error", "failed", "exception"],
  "strategy": [
    "定位最近一次失败的 SOP 步骤",
    "推断失败原因",
    "用最小改动修补该步骤",
    "运行验证脚本，失败则回滚"
  ],
  "constraints": { "max_files": 20 },
  "validation": ["node scripts/validate-suite.js"]
}

为什么要做成 Gene 这种形式？三个好处：

  1. 可复用：同类信号下次出现，直接调用同一个 Gene，不用让 LLM 每次现想策略
  2. 可审计：进化时记录”用了哪个 Gene”，出问题能精确回溯
  3. 可约束：把”最多改几个文件、必须跑哪些验证”写在 Gene 里，给 AI 套上缰绳

不同信号触发不同的 Gene，按”修复 / 优化 / 创新”三类划分：

Gene 类型 触发信号 行动
repair error、failed、被纠正 修复 SOP 中的错误步骤
optimize timeout、效率低、workaround 优化执行流程
innovate capability_gap、缺少能力 创建新的 SOP 或工具

而且Gene 的使用比例可调档，按系统状态切。所谓”档位”就是三类 Gene 在一次扫描里的命中权重：

策略档位 创新 优化 修复 适用场景
balanced 50% 30% 20% 日常默认
innovate 80% 15% 5% 系统稳定，搞新东西
harden 20% 40% 40% 大改动后，求稳
repair-only 0% 20% 80% 出事了，全力救火

我的看法：这套档位机制比”自进化开/关”两态有用得多。生产环境哪有”一直自由进化”的奢侈，多数时候你需要 harden 或 repair-only。

让进化可审计：EvolutionEvent + Capsule

自进化最大的工程问题不是”怎么进化”，而是”进化错了怎么追溯怎么回滚“。这个开源方案给了一个干净的答案：

  • EvolutionEvent：每一次进化都写一条记录（触发信号、用了什么策略、改了哪些文件、验证结果），追加到 events.jsonl，append-only 永不修改
  • Capsule：多次验证有效的进化方案被”固化”成 Capsule，下次类似信号直接复用，避免重复发明轮子
{"ts":"2026-04-26T09:30","gene":"repair","signals":["error","403"],"files":["sop/morning_brief.md"],"validation":"pass","rollback_id":"evt_001"}
{"ts":"2026-04-26T11:00","gene":"optimize","signals":["workaround","success"],"capsule_id":"cap_fallback_source","files":["sop/morning_brief.md"]}

没有可审计的进化记录，自进化系统等同于不可控的随机扰动。

配合 git 管理改动 + evolver-disable 一键开关，整个系统才算”敢上生产”。

安全约束：自进化不能为所欲为

这套方案明确给出了禁止清单，我觉得是非常宝贵的工程经验：

绝对禁止 Agent 自行修改：

  • Agent 核心原则文件（SOUL.md、CLAUDE.md 主文件）
  • Agent 间通信协议
  • 敏感配置（.env、credentials）

需要人工审核：

  • 创建新 SOP
  • 批量修改（>3 个文件）
  • 修改知识库

单次改动硬约束：

  • 文件数 ≤ 20
  • 单文件行数 ≤ 100
  • 验证命令必须通过白名单（只允许 node、npm、npx 等）

我的实战感受：这些”边界”才是自进化系统的灵魂。没有边界的自进化，只需要一次幻觉就能把整个 Agent 体系搞崩。

半自动 → 全自动的演进路径

最后一个值得抄的设计是演进节奏：

阶段 模式 说明
初期（前 2 周） 半自动 自动扫描+生成提示词，人工审核后执行
稳定后 全自动 自动扫描+执行，每周人工 review events.jsonl

不要一上来就追求”全自动自进化”。先半自动跑两周，看看 Gene 选得对不对、改动方向合不合理，校准之后再放开自动执行。直接上全自动 = 直接上线一个没经过测试的关键模块。

自进化的三个隐形陷阱

聊完怎么做，必须说几个容易忽视的坑。这些坑我自己和身边朋友都踩过。

陷阱 1：反馈回路失灵

自进化的核心是”反馈 → 调整”。但如果反馈信号是错的或太稀疏，Agent 会朝错误方向进化。

举个真实例子：一个客服 Agent，以”用户没追问”作为”回答满意”的信号。结果它进化出的策略是给冗长含糊的回答——因为含糊的回答让用户摸不着头脑，不知道怎么追问，于是”满意率”飙升。

坏反馈比没反馈更糟糕，因为它让系统自信地走向错误。

解决思路：设计反馈时同时引入多个正交信号（显式评分、任务完成率、后续相关操作），互相校验。

陷阱 2：记忆污染

用户随口说一句”我最近在学 Python”，Agent 就把它写进长期记忆。三个月后用户已经转去写 Rust 了，Agent 还在推荐 Python 教程。

解决：

  • 给记忆分”短期 / 中期 / 长期”三档，短期记忆自动过期
  • 重要偏好显式写入（类似 Cursor Rules），不要靠 Agent 自己猜
  • 定期让用户 review 记忆摘要（ChatGPT 的 “Manage memories” 就是这个思路）

陷阱 3：能力倒退

这个很反直觉：一个自进化 Agent 可能随着时间推移，能力不升反降。

原因通常是：

  • 生成的劣质 Skill 污染工具池，检索时抢了好 Skill 的资源
  • 错误的记忆被反复引用，形成”记忆强化的偏见”
  • Prompt 自优化陷入局部最优，新场景适应变差

解决思路：**定期做”能力体检”**——用一组固定测试题跑过去的模型版本和现在的版本，如果退化就回滚或清理。

我的选型建议

如果你正在考虑给自己的产品/项目加”自进化”能力，给几条建议：

1. 先问自己：场景真的需要自进化吗？

大多数工具型场景（翻译、摘要、代码补全），不需要进化，只需要稳定。加了自进化反而引入不确定性。

2. 从 L1 开始，别急着上 L3

把显式记忆（Rules 文件 + 对话历史摘要）做扎实，90% 的”越用越懂我”的感觉就有了。花里胡哨的 Skill 自动生成，大部分场景投入产出比不高。

3. 反馈信号必须多维度

单一信号（比如只看用户点赞）几乎注定出问题。至少凑齐三个正交维度：显式评分 + 任务完成度 + 后续行为。

4. 永远保留”重置”能力

自进化的东西一旦跑偏，修复成本远高于没进化的系统。给用户一键清空记忆、回滚到初始策略的按钮，是底线而不是加分项。

5. 别被营销话术忽悠

“自进化”这个词现在很火，但很多产品其实只做了 L1 甚至 L1 都没做全，就敢宣传”自进化 Agent”。看产品看实现，别看 PR。

小结

  1. 真正能落地的”AI 自进化”，指的是 Agent 层的自我改进，不是模型自己训练自己
  2. 自进化可分四层：记忆沉淀（L1）、策略进化（L2）、技能生成（L3）、模型微调（L4），深度递增
  3. L1 记忆沉淀是所有自进化的底座，也是性价比最高的一层，90% 场景做好 L1 就够
  4. L3 技能自动生成概念诱人，但目前只在高度结构化的工作流里能稳定落地
  5. 工程化落地的核心结构：信号捕获 + 进化引擎 + 进化目标三层闭环，配合 Gene 策略和可审计的 EvolutionEvent
  6. 进化策略要可调档（balanced / innovate / harden / repair-only），不要”一直自由进化”
  7. 自进化三大陷阱：反馈回路失灵、记忆污染、能力倒退，都有可能让系统越用越差
  8. 反馈信号必须多维度正交，单一信号必然跑偏
  9. 演进节奏：半自动跑两周再放开全自动，直接全自动等于上线未测试的关键模块
  10. 选型核心判断：场景是否真的需要进化——很多时候，稳定比”聪明”更重要
