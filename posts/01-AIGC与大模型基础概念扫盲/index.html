AIGC与大模型基础概念扫盲

发表于2025|更新于2026|AI相关
|字数总计:2.5k|阅读时长:8分钟

前言：2022年底 ChatGPT 横空出世，2025年初 DeepSeek-R1 刷爆朋友圈——短短两年多，AI 技术栈发生了翻天覆地的变化。作为一个开发者，我觉得我们更应该了解这些内容。

行业时间线

行业核心节点 事件
2022.11.30 ChatGPT 发布，全球AI竞赛开启
2023.03.14 GPT-4 发布，多模态能力初现
2023.07 Meta 开源 LLaMA 2，开源生态爆发
2024.02 Google 发布 Gemini 1.5，上下文窗口飙升至100万token
2024.04 Meta 发布 LLaMA 3，性能逼近GPT-4
2025.01.20 DeepSeek-R1 发布，开源推理模型，国产大模型崛起

从 ChatGPT 到 DeepSeek-R1，仅仅两年多时间，我最深的感受是——AI技术的普及速度远超我们的学习速度。如果还停留在”AI就是聊天机器人”的认知阶段，2025年的技术生态会让你完全懵圈。所以这篇文章，我会分享一下我掌握的核心概念。

什么是AIGC

AIGC（AI Generated Content）即人工智能生成内容。指的是利用人工智能技术自动生成文本、图像、音频、视频等内容。与传统的内容生成方式不同，AIGC 不是简单的检索或模板填充，而是基于深度学习的真正”创作”。

AIGC的核心特征

  1. 基于大模型驱动：不是简单的检索或复制，而是真正的”理解+创作”
  2. 多模态：支持文本、图像、音频、视频等多种形式
  3. 交互式：可以根据用户反馈进行调整和优化

典型的 AIGC 应用（2025年最新格局）：

领域 代表产品
文本生成 ChatGPT、Claude、DeepSeek
图像生成 Midjourney、DALL-E 3、Stable Diffusion 3
视频生成 Sora、Runway Gen-3、可灵
代码生成 GitHub Copilot、Cursor

LLM：大语言模型

LLM（Large Language Model）即大语言模型，是 AIGC 文本生成领域的核心技术。

参数规模大

这里我做一个通俗的类比：如果把大模型比作人类大脑，参数就是神经元之间的突触连接。参数越多，模型能”记住”和”理解”的模式就越多。

模型 参数量 类比
GPT-3（2020） 1750亿 一个百科全书式的学者
GPT-4（2023） 未公开，业界推测 1.8万亿（MoE架构） 一个全领域的专家团队
LLaMA 3 70B（2024） 700亿 一个某领域的资深专家
DeepSeek-R1（2025） 671亿（MoE架构，激活37亿） 小身材大智慧，MoE的胜利

我的整体体验：参数量不是万能指标！DeepSeek-R1 用 MoE（混合专家）架构，虽然总参数671B，但每次推理只激活37B参数，既省算力又保持了强大的推理能力。这也是为什么它能在消费级硬件上跑起来。不要迷信参数量，架构设计才是关键。

训练数据大

训练一个 LLM 需要海量文本数据：互联网网页、书籍、论文、代码库、对话数据等。GPT-3 的训练数据量约 45TB 文本数据，而 GPT-4 和后续模型使用的数据规模更为庞大。

核心名词解释

Token

Token 是 LLM 处理文本的基本单位，可以理解为”词片段”。

// Token 示例
// 英文："Hello, world!" → ["Hello", ",", " world", "!"]（4个token）
// 中文："你好世界" → ["你好", "世界"]（约2个token，取决于tokenizer）

// 实际验证：使用 tiktoken 库（OpenAI 的 tokenizer）
// npm install tiktoken
import { encoding_for_model } from 'tiktoken';

const enc = encoding_for_model('gpt-4');
const tokens = enc.encode('Hello, world!');
console.log(`Token数量: ${tokens.length}`); // 4
console.log(`Token列表: ${tokens}`);

const chineseTokens = enc.encode('你好世界');
console.log(`中文Token数量: ${chineseTokens.length}`); // 通常2-4个

理解 Token 很重要，因为它直接关系到你的钱包：

  1. API 计费：按 token 数量计算，输入和输出分别计价
  2. 上下文限制：模型一次能处理的 token 总数有上限
  3. 推理速度：token 越多，响应越慢、成本越高

在实际项目中，中文每个汉字大约消耗 1-2 个 token，而一个英文单词通常是 1 个 token。这意味着**同样的语义信息，中文消耗的 token 数量比英文多 30%-50%**。所以做中文 AI 应用时，成本控制要更加注意。

Context Window（上下文窗口）

上下文窗口是指模型一次能处理的最大 token 数量。这就是模型的”工作记忆”。

2025年主流模型上下文窗口（变化非常快）：

模型 上下文窗口 发布时间
GPT-3.5 4K / 16K 2023.03
GPT-4 8K / 32K 2023.03
GPT-4 Turbo 128K 2023.11
Claude 3.5 Sonnet 200K 2024.06
Gemini 1.5 Pro 1M (100万) 2024.02
GPT-4o 128K 2024.05
DeepSeek-R1 128K 2025.01

上下文窗口的竞争从”够不够用”变成了”用不用得起”。Gemini 1.5 Pro 号称100万 token，但你真用100万 token 一次调用，费用惊人。实际应用中，上下文管理能力比上下文窗口大小更重要——这就是后面我们会讲到的 RAG 技术存在的核心意义。

Temperature（温度）

Temperature 控制模型输出的随机性，范围通常在 0-2 之间。

// temperature = 0：确定性输出，适合事实查询、代码生成
// 同一个问题，每次回答几乎相同
const factualResponse = await openai.chat.completions.create({
  model: 'gpt-4',
  messages: [{ role: 'user', content: '1+1等于几？' }],
  temperature: 0  // 总是输出"2"
});

// temperature = 0.7：默认值，平衡创造性和准确性
// 适合大多数对话场景

// temperature = 1.5+：高创造性，适合头脑风暴、创意写作
// 但可能出现"胡说八道"
  • 代码生成/数据处理：temperature = 0，要的是准确
  • 日常对话/客服：temperature = 0.3-0.7
  • 创意写作/营销文案：temperature = 0.8-1.2
  • 超过 1.2 基本就是在”掷骰子”了，慎用

Prompt Engineering（提示词工程）

Prompt Engineering 是与 LLM 交互的核心技能：

<!-- ❌ 模糊的prompt —— 输出质量随缘 -->
写一篇文章

<!-- ✅ 结构化的prompt —— 输出质量稳定可控 -->
你是一个拥有10年经验的前端架构师。请写一篇关于React Hooks的技术文章。

要求：
1. 目标读者：有1-2年React经验的开发者
2. 核心内容：useState、useEffect、useMemo、useCallback
3. 每个Hook附带一个实际业务场景的代码示例
4. 重点讲解常见的性能陷阱和最佳实践
5. 字数：1500-2000字
6. 格式：Markdown，带代码高亮

Fine-tuning（微调）vs RAG

这两个概念经常被混淆。简单说：

  • 微调：改变模型本身——“让学生学习新知识”
  • RAG：给模型提供参考资料——“让学生开卷考试”
维度 Fine-tuning RAG
原理 用领域数据继续训练模型 检索相关文档注入prompt
成本 高（需要GPU训练） 低（只需向量数据库）
时效性 差（需要重新训练） 好（更新文档即可）
适用场景 学习特定风格/格式 基于私有知识问答
数据需求 高质量标注数据 原始文档即可

我的实战建议：90% 的场景应该优先考虑 RAG，成本低、见效快、可维护。微调只在需要改变模型输出风格/格式时才值得考虑。

模型 发布方 核心优势 适用场景

从LLM到应用：技术栈演进全景

1.0 时代（2023上半年）：直接调API

// 最原始的方式：一个 fetch 搞定
const response = await fetch('https://api.openai.com/v1/chat/completions', {
  method: 'POST',
  headers: {
    'Authorization': `Bearer ${API_KEY}`,
    'Content-Type': 'application/json'
  },
  body: JSON.stringify({
    model: 'gpt-3.5-turbo',
    messages: [{ role: 'user', content: 'Hello' }]
  })
});

const data = await response.json();
console.log(data.choices[0].message.content);

这个阶段的特点：能跑就行，但体验差（等待时间长、没有流式输出、无法访问私有数据）。

2.0 时代（2023下半年~2024）：工程化增强

  • Function Calling（2023.06）：让 LLM 调用外部工具
  • RAG：检索增强生成，让 LLM 能访问私有知识
  • SSE 流式输出：逐字输出，提升用户体验
  • LangChain/LlamaIndex：LLM 应用开发框架

3.0 时代（2024下半年~至今）：Agent与标准化

  • AI Agent：让 AI 自主规划和执行任务
  • MCP 协议（2024.11）：标准化的 AI 工具调用接口
  • Skills：AI 能力的模块化封装
  • Vibe Coding（2025.02）：AI 编程新范式
  • OpenSpec/OpenClaw（2025~2026）：前沿方向

开发者需要关注什么

作为开发者，在 AI 时代我认为最核心的能力排序是：

  1. 理解原理：不只要会调 API，要理解 Token、Temperature、上下文管理背后的逻辑
  2. Prompt 技能：这是与 AI 沟通的核心能力，也是性价比最高的技能投资
  3. 工程化能力：RAG、流式调用、Agent 架构等工程化实践
  4. 成本意识：Token 消耗、延迟、并发——AI 应用的成本结构和传统应用完全不同
  5. 安全合规：提示词注入防御、数据隐私、内容审核
  6. 持续学习：这个领域的半衰期太短了，两个月不关注就可能错过关键技术节点

小结

AIGC 和 LLM 不是炒作，而是真正改变软件开发范式的技术变革
