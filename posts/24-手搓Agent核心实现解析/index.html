不用框架，从零手搓一个AI Agent

发表于2026|更新于2026|AI相关
|字数总计:4.2k|阅读时长:16分钟

前言：之前写过 Agent 架构设计与实践，讲的是理论层面的东西。这篇反过来，直接从代码层面拆解：不用任何 Agent 框架，纯手写 Agent Loop，用最原始的 OpenAI Function Calling 协议，自己管循环、管工具、管重试、管上下文。项目是一个智能新闻聚合与推送系统，开源在 GitHub。

为什么不用 Agent 框架

先说结论：不是框架不好，是我需要的控制粒度太细了。

这个项目是一个智能新闻聚合与推送系统，Agent 需要做的事情包括：

  1. 读取用户画像，了解用户兴趣
  2. 从 RSS、GitHub Trending、微信公众号等多个源采集内容
  3. 过滤去重
  4. 用规则 + AI 双层评分
  5. 为高分内容生成个性化摘要
  6. 组装邮件推送
  7. 将本次决策经验写入记忆，供下次参考

整个流程中，Agent 需要自主决定调用哪些工具、以什么顺序调用、遇到错误怎么处理。如果某个环节用了框架的黑盒封装，限频重试逻辑你控制不了，Token 裁剪策略你改不了，推送意图检测你加不进去。

所以我的选择是：手写 Agent Loop + OpenAI Function Calling 协议 + NestJS 服务注入。所有逻辑透明可控，每一行都是自己的。

Agent Loop 核心实现

所有 Agent 框架的本质都是一个循环：调 LLM → 拿到 tool_calls → 执行工具 → 把结果喂回去 → 再调 LLM。我把这个循环手写了出来。

核心代码在 backend/src/modules/agent/agent.service.ts 的 runAgentLoop 方法中：

// Agent Loop 核心
for (let step = 0; step < this.maxSteps; step++) {
  const stepStart = Date.now();

  // 1. 每轮从 ToolRegistry 实时获取工具列表（支持动态注册）
  const currentTools = this.getCurrentTools(allowedToolNames);

  // 2. 调用 LLM（带 4 级重试 + 自动降级）
  let response: OpenAI.ChatCompletion;
  try {
    response = await this.callLLMWithRetry(sessionId, messages, currentTools);
  } catch (error) {
    this.logger.error(`[${sessionId}] LLM 调用最终失败: ${error.message}`);
    break;
  }

  const choice = response.choices[0];
  const assistantMessage = choice.message;
  const thinking = assistantMessage.content || '';
  const toolCallsRaw = assistantMessage.tool_calls || [];

  // 3. 解析 tool_calls —— 从 OpenAI 协议格式提取工具名和参数
  const toolCalls: AgentToolCall[] = toolCallsRaw
    .filter((tc) => tc.type === 'function')
    .map((tc) => ({
      id: tc.id,
      name: tc.function.name,
      args: JSON.parse(tc.function.arguments || '{}'),
    }));

  // 4. LLM 没有调用任何 Tool → 任务完成，退出循环
  if (toolCallsRaw.length === 0 || choice.finish_reason === 'stop') {
    // 推送意图检测 + 兜底逻辑（后文详述）
    break;
  }

  // 5. 把 LLM 响应加入消息历史
  messages.push(assistantMessage);

  // 6. 并行执行所有 Tool Call
  const toolResultMessages = await Promise.all(
    toolCalls.map(async (tc) => {
      const result = await this.toolRegistry.executeTool(tc.name, tc.args);
      // 结果超 8000 字符则智能截断
      const truncated = JSON.stringify(result).length > 8000
        ? this.smartTruncateToolResult(result, 8000)
        : JSON.stringify(result);
      return { role: 'tool', tool_call_id: tc.id, content: truncated };
    }),
  );

  // 7. Tool 结果回填消息历史 → 回到步骤 1
  messages.push(...toolResultMessages);

  // 8. Token 管理 —— 消息过长时裁剪早期轮次
  this.pruneMessagesIfNeeded(messages);
}

这就是一个完整 Agent 的核心。没有任何框架魔法，就是一个 for 循环。每一步做的事情很清楚：

步骤 做什么 为什么
1 实时获取工具列表 Skills 系统可能动态注册了新工具
2 调 LLM（带重试） 限频时自动切备用模型
3 解析 tool_calls 从 OpenAI 协议中提取工具名和 JSON 参数
4 检查是否结束 LLM 不再调工具 = 认为任务完成
5 消息历史追加 保持完整对话上下文
6 并行执行工具 Promise.all，LLM 一次选了 3 个工具就同时跑
7 结果回填 让 LLM 看到工具输出，决定下一步
8 Token 裁剪 防止上下文爆炸，丢弃早期非关键消息

Tool Registry —— 16 个工具的注册

Agent 的能力边界由工具决定。我设计了一个 AgentToolRegistry，把 NestJS 的 11 个 Service 封装成 16 个 OpenAI Function Calling 格式的工具。

@Injectable()
export class AgentToolRegistry {
  private tools: Map<string, ToolRegistryEntry> = new Map();

  constructor(
    private readonly collectorService: CollectorService,   // 内容采集
    private readonly filterService: FilterService,         // 过滤去重
    private readonly scorerService: ScorerService,         // AI 评分
    private readonly summaryService: SummaryService,       // 摘要生成
    private readonly notificationService: NotificationService, // 推送
    private readonly memoryService: MemoryService,         // 记忆
    private readonly feedbackService: FeedbackService,     // 反馈
    private readonly userService: UserService,             // 用户
    private readonly contentService: ContentService,       // 内容
    private readonly sourceService: SourceService,         // 数据源
    private readonly digestService: DigestService,         // 日报
  ) {
    this.registerAllTools();
  }

  // 按类别注册
  private registerAllTools(): void {
    this.registerPerceptionTools();  // 感知类 (4)
    this.registerActionTools();      // 行动类 (7)
    this.registerPushTools();        // 推送类 (2)
    this.registerMemoryTools();      // 记忆类 (3)
  }
}

每个工具的注册格式统一遵循 OpenAI Function Calling 协议：

this.register({

这个设计的好处是：工具定义和执行逻辑绑在一起，新增工具只需要加一个 register() 调用。

LLM 容错机制 —— 4 级重试 + 自动降级

线上跑 Agent 最怕什么？LLM 接口限频。一旦 429，整个流程就断了。我设计了一套 4 级重试策略：

private async callLLMWithRetry(
  sessionId: string,
  messages: OpenAI.ChatCompletionMessageParam[],
  tools: OpenAI.ChatCompletionTool[],
): Promise<OpenAI.ChatCompletion> {
  const callLLM = async (model: string) => {
    await this.rateLimiter.acquire(`Agent[${sessionId}]`);  // 主动限流
    return this.openai.chat.completions.create({
      model,
      max_tokens: 4096,
      messages,
      tools,
      tool_choice: 'auto',
    });
  };

  // 第 1 次：当前模型
  try {
    return await callLLM(this.getCurrentModel());
  } catch (error) {
    if (!this.isRateLimitError(error)) throw error;

    // 第 2 次：等待 30s，同一模型重试
    await sleep(30_000);
    try {
      return await callLLM(this.getCurrentModel());
    } catch (retryError) {
      if (!this.isRateLimitError(retryError)) throw retryError;

      // 第 3 次：切换到备用模型
      this.usingFallback = true;
      await this.sendRateLimitAlert(context, retryError.message, this.fallbackModel);
      try {
        return await callLLM(this.fallbackModel);
      } catch (fallbackError) {
        // 第 4 次：等待 60s，备用模型最后重试
        await sleep(60_000);
        return await callLLM(this.fallbackModel);  // 再失败就抛出
      }
    }
  }
}

重试策略的降级路径：

第 1 次  →  当前模型直接调用
  ↓ 限频
第 2 次  →  等 30s，同模型重试
  ↓ 再限频
第 3 次  →  切换备用模型 + 发告警邮件
  ↓ 还限频
第 4 次  →  等 60s，备用模型最后一搏
  ↓ 全失败
抛出错误 + 告警邮件

配合主动限流（rateLimiter.acquire() 基于滑动窗口的 RPM 控制），大部分情况下根本不会触发被动重试。

同时，isRateLimitError 方法做了非常细致的错误识别：

private isRateLimitError(error: any): boolean {
  // HTTP 429
  if (status === 429) return true;
  // 标准关键词
  if (msg.includes('rate limit') || msg.includes('too many requests')) return true;
  // 业务层限频（某些 API 用 400 + 特定错误码表示限频）
  if (status === 400 && msg.includes('频率超出')) return true;
  return false;
}

为什么要这么细？因为不同的 LLM 提供商（OpenAI / 混元 / DeepSeek / 其他国产模型）返回限频错误的格式完全不统一。有的用 429，有的用 400 + 业务错误码。

推送意图检测 —— 防止 LLM “说了不做”

这是实际跑 Agent 时踩出来的坑：LLM 有时候会在文字中说”好的，我现在发送推送”，但实际上它 finish_reason=stop 了，根本没调用推送工具。

原因可能是上下文太长、模型注意力分散，或者纯粹是概率性的 hallucination。

解决方案——推送意图检测 + 自动注入提醒：

if (!digestSent && thinking && this.looksLikePushIntent(thinking)) {
  // LLM 嘴上说要推送，手上没调工具
  messages.push(assistantMessage);
  messages.push({
    role: 'user',
    content: `你刚才说要发送推送，但没有调用推送工具。请立即调用 ${digestToolName} 工具完成推送。`,
  });
  continue;  // 不退出循环，让 LLM 继续执行
}

looksLikePushIntent 是一个关键词匹配方法，覆盖了各种表达推送意图的说法：

private looksLikePushIntent(text: string): boolean {
  const pushKeywords = [
    '发送推送', '发送每日精选', '推送给用户',
    'send_daily_digest', 'send_github_trending',
    '现在发送', '开始推送', '执行推送',
    '下一步.*推送', '接下来.*推送',
  ];
  // ...
}

这类”工程直觉”型的处理，在框架里是很难加的。你得理解 LLM 的行为特征，才能针对性地打补丁。

安全网 —— Agent 失败后的三层兜底

Agent 是概率性的。即使做了 4 级重试 + 推送意图检测，仍然有可能整个流程跑完了，推送没发出去。

所以我设计了三层兜底：

Agent 正常推送  →  成功
       ↓ 失败
自动推送已分析内容  →  成功（降级，但用户有东西看）
       ↓ 也失败
发送告警邮件给管理员  →  人工介入

if (!digestSent) {
  return this.handleUnsentDigest(
    userId, sessionId, label, steps, startTime,
    enableAutoDigest,     // 是否尝试自动推送
    enableFallbackAlert,  // 是否发告警邮件
  );
}

handleUnsentDigest 会从数据库中捞出已经被 Agent 评过分的内容，按分数排序，直接调推送服务发出去。即使 Agent 的 LLM 大脑宕了，用户也不会收到空邮件。

双层评分系统 —— 规则 + AI

评分是 Agent 决定”推什么给用户”的关键环节。我设计了规则预筛 + AI 深度评分的双层架构。

第一层：规则评分（5 维度加权）

const breakdown = {
  relevance:     this.scoreRelevance(content, user),     // 相关性 45%
  quality:       this.scoreQuality(content),              // 质量   20%
  timeliness:    this.scoreTimeliness(content),            // 时效性 20%
  novelty:       this.scoreNovelty(content, recentTitles), // 新颖性 10%
  actionability: this.scoreActionability(content),         // 可操作 5%
};

const finalScore =
  breakdown.relevance * weights.relevance +
  breakdown.quality * weights.quality +
  breakdown.timeliness * weights.timeliness +
  breakdown.novelty * weights.novelty +
  breakdown.actionability * weights.actionability;

每个维度都是确定性的规则计算：

  • 相关性（45%）：内容标签与用户兴趣标签的匹配度
  • 质量（20%）：基于内容长度、引用源可信度等
  • 时效性（20%）：发布时间与当前时间的衰减
  • 新颖性（10%）：与近 7 天已推送内容的标题相似度（越不同越高分）
  • 可操作性（5%）：内容是否包含可执行的建议/方法

重点是：评分权重是每个用户可自定义的，存在用户偏好中。Agent 会通过 read_user_profile 工具获取用户的个性化权重。

第二层：AI 深度评分

规则评分快但粗糙。对于进入候选池的高分内容，Agent 可以通过 LLM 做更精准的语义理解评分，覆盖规则引擎无法判断的场景。

Agent 记忆系统

传统的 LLM 调用是无状态的——每次都是从零开始。但一个好的信息管家应该记住上次的经验。

我实现了一个基于 MySQL 持久化的记忆系统，Agent 通过 3 个记忆工具与之交互：

store_memory         → 存储记忆（决策经验 / 源质量评估 / 偏好变化）
query_memory         → 查询记忆（关键词匹配，支持按类型过滤）
analyze_source_quality → 分析数据源质量并记入记忆

记忆的存储格式：

{
  type: 'decision_experience',        // 记忆类型
  memoryKey: 'github_trending_quality', // 检索关键字
  value: {                             // 记忆内容（JSON）
    observation: 'GitHub Trending 技术文章质量稳定但重复率较高',
    suggestion: '后续可降低 GitHub 源的新颖性权重',
    confidence: 0.85
  },
  userId: 'xxx'
}

实际效果：Agent 第一次跑的时候是”新手”，到第 10 次、第 50 次，它会积累大量对数据源质量、用户偏好变化的经验，推送质量会随使用时间持续提升。

三种运行模式

同一个 runAgentLoop 方法，通过传入不同的配置（systemPrompt、工具白名单、推送工具名），实现了三种运行模式：

模式 入口方法 可用工具数 推送工具 用途
每日推送 runDailyDigest() 全部 16 个 send_daily_digest 完整流程：采集→评分→推送
GitHub 推送 runGithubTrending() 12 个（去掉微信相关） send_github_trending 只跑 GitHub 源
仅分析 runAnalysisOnly() 部分（跳过采集） 无 只做评分分析，不推送 
// 每日推送入口
async runDailyDigest(userId: string): Promise<AgentResult> {
  const user = await this.userService.findById(userId);
  let systemPrompt = this.buildAgentSystemPrompt(user);

  // Skills 增强：注入已启用 Skill 的描述 + 注册 load_skill 工具
  const enhanced = await this.skillEnhancer.enhance(systemPrompt, userId);
  systemPrompt = enhanced.systemPrompt;

  return await this.runAgentLoop({
    userId,
    label: 'Agent',
    systemPrompt,
    userMessage: `执行今日信息采集和推送任务。
当前时间: ${new Date().toISOString()}
用户 ID: ${userId}
你的目标是为用户产出一份高质量的每日精选推送。
请自主决定执行步骤，合理使用可用工具。`,
    tools: this.toolRegistry.getToolDefinitions(),
    digestToolName: 'send_daily_digest',
    enableAutoDigest: true,
    enableFallbackAlert: true,
  });
}

这种设计的好处在于：核心循环只写一份，不同场景通过配置注入来区分。新增一种运行模式只需要写一个新的入口方法。

Skills 插件系统

仅有 16 个内置工具是不够的。我实现了一套可插拔的 Skills 系统，参照了 Claude Code / CodeBuddy 的 Skills 设计标准。

Skill 格式

每个 Skill 就是一个 SKILL.md 文件：

---
name: tech-news-scorer
description: 使用 Hacker News API 获取实时热度评分作为评分参考
---

# 工具说明
本 Skill 提供 get_hn_score 工具，根据文章 URL 查询其在 Hacker News 上的热度。

# 使用规则
- 仅在评分阶段使用
- 结果作为 quality 维度的补充参考

渐进式加载（3 阶段）

阶段 1: 系统提示词注入 Skill 名称 + 描述（轻量）
阶段 2: Agent 判断需要 → 调用 load_skill 工具 → 加载完整 SKILL.md 内容
阶段 3: Skill 注册的脚本工具可被 Agent 调用

为什么要三阶段？因为如果一次性把所有 Skill 的完整内容塞进 system prompt，Token 会爆。渐进式加载让 Agent 自己决定”我需要哪个 Skill”，按需加载。

类型系统

整个 Agent 的类型定义非常精简，只有 76 行：

// 一轮 Agent 步骤
interface AgentStep {
  step: number;
  thinking: string;          // LLM 的思考文本
  toolCalls: AgentToolCall[];
  toolResults: AgentToolResult[];
  durationMs: number;
}

// 工具调用
interface AgentToolCall {
  id: string;
  name: string;
  args: Record<string, any>;
}

// 工具执行结果
interface AgentToolResult {
  toolUseId: string;
  toolName: string;
  result: any;
  isError: boolean;
  durationMs: number;
}

// Agent 最终结果
interface AgentResult {
  sessionId: string;
  report: string;
  stepsUsed: number;
  totalDurationMs: number;
  isSuccess: boolean;
  isFallback: boolean;    // 是否走了降级逻辑
  digestSent: boolean;    // 推送是否成功
  contentCount: number;
}

// OpenAI Function Calling 工具定义
interface OpenAIToolDefinition {
  type: 'function';
  function: {
    name: string;
    description: string;
    parameters: {
      type: 'object';
      properties: Record<string, any>;
      required?: string[];
    };
  };
}

这些类型跟 OpenAI 的 API 格式高度对齐。这就是不用框架的好处——你的代码直接对接协议本身，没有中间抽象层的类型转换。

一次完整的执行流程

用一个实际例子串起来：

Step 1:  Agent 调用 read_user_profile → 了解到用户关注 "前端/AI/Rust"
Step 2:  Agent 调用 query_memory → 查到上次经验："GitHub 源质量好但重复率高"
Step 3:  Agent 调用 get_user_sources → 获取 3 个 RSS + 2 个公众号 + GitHub
Step 4:  Agent 并行调用 collect_github + collect_wechat → 采集 47 篇内容
Step 5:  Agent 调用 filter_and_dedup → 去重后剩 31 篇
Step 6:  Agent 调用 score_contents → 五维度评分，Top 10 出炉
Step 7:  Agent 调用 batch_generate_summaries → 为 Top 10 生成个性化摘要
Step 8:  Agent 调用 send_daily_digest → 邮件推送成功
Step 9:  Agent 调用 store_memory → 记下本次经验："今日 RSS 源产出 2 篇高质量文章"
Step 10: Agent 输出执行报告 → finish_reason=stop，循环结束

整个过程中，Agent 自主决定了调用顺序。有时候它会在 Step 3 之后先查记忆再决定要不要采集某个源，有时候它会跳过摘要直接推送——这就是 Agent 和 Workflow 的本质区别。

小结

  1. Agent 本质就是一个循环 + 工具调用，不要被框架吓到
  2. 容错比功能重要——限频重试、推送兜底、告警邮件，这些才是线上能跑的关键
  3. 记忆让 Agent 有灵魂——没有记忆的 Agent 就是一次性脚本
  4. 类型即协议——把 OpenAI 的 Function Calling 协议直接映射成 TypeScript 类型，不需要额外抽象
  5. 工具注册表 + 配置注入的设计，让同一个 Loop 支撑多种运行模式
