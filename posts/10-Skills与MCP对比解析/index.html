Skills与MCP对比解析：AI能力封装的两种路径

发表于2026|更新于2026|AI相关
|字数总计:7k|阅读时长:24分钟

一句话区分 Skills 和 MCP

MCP 是”接口协议”，Skills 是”操作手册”。

类比现实世界：

  • MCP = USB 接口标准（定义了设备怎么连接）
  • Skills = 写给新同事的岗位手册（告诉他碰到这类事情该怎么做）
层次关系：

┌──────────────────────────────┐
│       AI 应用 / Agent        │  ← 应用层
├──────────────────────────────┤
│        Skills 能力层          │  ← 业务能力封装（知识+流程+脚本）
├──────────────────────────────┤
│        MCP 协议层             │  ← 标准化的工具/资源访问
├──────────────────────────────┤
│     数据源 / API / 工具       │  ← 底层资源
└──────────────────────────────┘

Skills 到底是什么

不是代码，是”操作手册”

一个核心误区：很多人第一次听到 Skills，会以为它是某种代码模块或函数库。实际上——

Skill = 一个说明书 + 工具，里面是写给 AI 看的”入职手册 + 参考资料库 + 工具箱”。

把 AI Agent 想象成刚入职的新同事：

  • Skills：递给他的操作手册——“碰到这类任务，按这个流程做”
  • Rules：公司规章制度——“所有代码用 TypeScript，不许用 var”
  • MCP：外部系统的接口文档——“查数据库走这个接口，调天气API走那个接口”

标准目录结构

一个 Skill 的标准结构长这样：

my-skill/
├── SKILL.md          # 必需：指令 + 元数据（技能说明书）
├── scripts/          # 可选：可执行脚本（自动化操作）
│   ├── main.py
├── references/       # 可选：参考文档（领域知识）
│   ├── api-docs.md
│   └── style-guide.md
└── assets/           # 可选：模板、资源文件
    └── report-template.md

每个部分的职责：

组成部分 是否必需 职责 类比
SKILL.md ✅ 必需 定义”我是谁、什么时候用我、怎么做” 岗位说明书
scripts/ 可选 确定性操作（校验、转换、上传） 工具箱
references/ 可选 详细领域知识（按需加载，省Token） 参考资料柜
assets/ 可选 模板、配置文件等静态资源 文件模板夹

SKILL.md：技能的灵魂

SKILL.md 是整个 Skill 的核心，由两部分组成：

1. YAML 元数据（给 AI 路由器看的）——Agent 扫描这部分来决定”要不要用这个 Skill”

2. Markdown 正文（给 AI 执行看的）——匹配成功后，Agent 读取正文来执行任务

---
# === 元数据区（YAML Front Matter）===
name: meeting-minutes          # 技能唯一标识，kebab-case
description:
  当用户提供会议内容、录音转写文本，
  或要求生成会议纪要时使用。
  能够提取议题、结论、行动项，
  并在涉及预算时触发财务合规检查。
---

# 会议纪要生成 Skill

## 你的角色
你是一位专业的会议记录员，擅长从会议内容中提取关键信息。

## 工作流程
1. 识别会议基本信息（时间、参会人、主题）
2. 提取每个议题的讨论要点和结论
3. 整理行动项（负责人 + 截止日期）
4. 如果提到金额/预算/采购 → 读取 `references/finance-policy.md` 进行合规检查
5. 如果用户要求上传 → 执行 `scripts/upload.py`

## 输出格式
```markdown
# 会议纪要
- **会议主题**: xxx
- **时间**: xxx
- **参会人**: xxx

## 议题与结论
| 议题 | 讨论要点 | 结论 |
|------|---------|------|
| ... | ... | ... |

## 行动项
| 事项 | 负责人 | 截止日期 |
|------|--------|---------|
| ... | ... | ... |

## 风险提醒（如有）
...

## 禁止事项
- ❌ 不得臆造未提及的议题或结论
- ❌ 缺失信息写"待确认"，不要编造
- ❌ 不要输出会议内容的原文复制

注意看 description 字段——这是触发的关键。Agent 通过扫描所有 Skill 的 description 来判断当前任务该调用哪个 Skill。写得不好，Skill 就永远不会被触发。

渐进式加载：三层披露机制

Skills 有一个精妙的设计：不是一股脑全部塞进上下文，而是分层按需加载。

┌─────────────────────────────────────────────────┐
│  第一层：元数据（name + description）              │  ← 始终加载，用于路由匹配
│  Token 消耗：极少（几十个Token）                   │
├─────────────────────────────────────────────────┤
│  第二层：SKILL.md 正文                            │  ← 匹配成功后加载
│  Token 消耗：中等（建议500行以内）                  │
├─────────────────────────────────────────────────┤
│  第三层：references/ + scripts/                   │  ← 正文中指示时才加载
│  Token 消耗：按需（可能很大，但只在需要时读取）       │
└─────────────────────────────────────────────────┘

这就是为什么 Skill 要分层组织——**核心规则放 SKILL.md，详细知识放 references/**。如果把所有内容都塞进 SKILL.md，每次触发都会消耗大量 Token。

Skill 的三种类型

根据目录内容不同，Skill 分为三类：

1. 纯指令型（只有 SKILL.md）

最轻量，没有脚本，纯靠 Prompt 指导 AI 行为。适合编码规范、写作模板等场景。

vue-list-page/
└── SKILL.md    # 定义 Vue 列表页的标准写法

2. 脚本型（SKILL.md + scripts/）

包含可执行脚本，Agent 通过 JSON 传参调用脚本，获取确定性结果。适合需要精确计算、数据校验、外部操作的场景。

weather-query/
├── SKILL.md
└── scripts/
    └── main.py    # 调用天气API，返回结构化天气数据

3. 知识密集型（SKILL.md + references/ + scripts/）

完整形态，既有执行脚本又有参考文档。适合复杂的垂直领域场景。

financial-audit/
├── SKILL.md
├── scripts/
│   ├── check_budget.py     # 金额提取和审批规则校验
│   └── upload.py           # 上传生成的报告
├── references/
│   ├── finance-policy.md   # 财务审批制度
│   └── tax-rules.md        # 税务法规摘要
└── assets/
    └── report-template.md  # 审计报告模板

MCP vs Skills 具体例子对比

以”分析 GitHub 仓库”为例，看看 MCP 和 Skills 分别怎么做：

用 MCP（提供原子工具）：

MCP Server 暴露一组独立的工具函数，AI 自行决定调用顺序：

// MCP Server 提供的工具列表
{
  "tools": [
    { "name": "get_repo_info",       "description": "获取仓库基本信息" },
    { "name": "list_files",          "description": "列出文件目录" },
    { "name": "read_file",           "description": "读取文件内容" },
    { "name": "get_issues",          "description": "获取Issues列表" },
    { "name": "get_commit_history",  "description": "获取提交历史" }
  ]
}

// AI 拿到工具列表后，自主规划：
// "我应该先调 get_repo_info，再 list_files 看看结构..."
// → 调用顺序、分析策略、输出格式——全靠 AI 自己想
// → 结果好坏取决于 AI 的推理能力

用 Skill（封装完整流程）：

一个文件夹，把分析流程、评判标准、输出格式全部固化：

github-repo-analyzer/
├── SKILL.md
├── scripts/
│   └── scan_structure.py      # 扫描项目结构并输出文件树
└── references/
    └── quality-criteria.md    # 代码质量评判标准

SKILL.md 内容：

---
name: github-repo-analyzer
description: 
  当用户要求分析一个GitHub仓库的代码质量、架构设计或维护状况时使用。
  也适用于"这个项目怎么样"、"帮我评估这个仓库"等请求。
---

# GitHub 仓库分析 Skill

## 分析流程（严格按此顺序执行）
1. 获取仓库基本信息和统计数据
2. 执行 `scripts/scan_structure.py` 扫描项目结构
3. 分析技术栈和核心入口文件
4. 参考 `references/quality-criteria.md` 中的标准进行评分
5. 生成结构化分析报告

## 评分维度
| 维度 | 权重 | 说明 |
|------|------|------|
| 项目规范性 | 30% | README、LICENSE、CI/CD、.gitignore |
| 代码质量 | 30% | 函数复杂度、命名规范、错误处理 |
| 维护活跃度 | 20% | 提交频率、Issue响应、PR处理 |
| 文档完整性 | 20% | API文档、注释、CHANGELOG |

## 输出格式
（固定的报告模板，包含概述、技术栈、各维度评分、改进建议...）

## 禁止事项
- ❌ 不得跳过任何评分维度
- ❌ 不得在没有数据支撑的情况下给高分

关键区别：

  • MCP 方式：AI 拿到5个工具，自由发挥——可能分析得很好，也可能遗漏关键维度
  • Skill 方式：分析流程、评分标准、输出格式全部预定义——结果稳定、可复现

核心差异对比

维度 Skills MCP
本质 操作手册（知识+流程+规则） 接口协议（工具连接标准）
载体 文件夹（SKILL.md + scripts + references） Server/Client + JSON-RPC 协议
粒度 粗粒度（完整业务流程） 细粒度（原子操作）
知识承载 ✅ 内置领域知识和评判标准 ❌ 不承载知识
执行方式 预定义流程，AI 按手册执行 AI 自主编排工具调用
触发方式 任务匹配时按需加载 显式调用工具函数
Token 开销 渐进式披露，按需加载 每次对话携带完整工具定义，开销较大
标准化 平台有差异（但结构趋同） 跨平台统一标准
灵活性 中等（流程相对固定） 高（AI自由组合）
可靠性 高（流程确定，输出稳定） 中（依赖AI决策质量）
适用场景 垂直领域深耕、团队SOP 通用工具生态

MCP 的 Token 开销问题：

MCP 的工具定义（每个工具的名称、描述、参数 JSON Schema）需要在每轮对话中都完整注入到 System Prompt 里，这样 LLM 才知道有哪些工具可以调用。假设你连接了 3 个 MCP Server，每个提供 10 个工具，每个工具的定义平均占 200 个 Token，那么：

3 × 10 × 200 = 6,000 Token（每轮对话的固定开销）

这意味着：

  • 你连接的 MCP Server 越多，每轮对话的固定 Token 成本越高
  • 无论这轮对话是否使用某个工具，它的定义都会被带入上下文
  • 多轮对话场景下，这个开销会被累积放大

Skills 的 Token 优势：

Skills 的三层渐进式披露机制完美解决了这个问题：

  • 第一层（元数据）：只有 name + description，几十个 Token，始终加载
  • 第二层（指令正文）：匹配后才加载 SKILL.md 正文，中等消耗
  • 第三层（references/）：只在 AI 判断需要时才读取，完全按需

一个 Skill 在未被触发时，对上下文的开销可能只有 30~50 Token（仅元数据），而一组 MCP 工具定义动辄 数千 Token，还是每轮都要带上。

一句话选择法

缺"连接外部数据/系统"的能力 → 选 MCP
缺"标准化流程和专业产出"的能力 → 选 Skills
两者都缺 → MCP + Skills 组合（最强形态）

实战：开发一个完整的 Skill

以”技术文章质量分析 Skill”为例，展示一个真实可用的 Skill 怎么开发。

目录结构

tech-article-analyzer/
├── SKILL.md                        # 核心：分析流程和评分规则
├── scripts/
│   └── extract_stats.py            # 提取文章统计信息（字数、代码块数等）
├── references/
│   └── scoring-criteria.md         # 详细评分标准（五级描述）
└── assets/
    └── report-template.md          # 报告输出模板

SKILL.md

---
name: tech-article-analyzer
description: 
  分析技术文章的质量、深度和实用性，给出结构化评分和改进建议。
  当用户说"分析这篇文章"、"文章质量怎么样"、"帮我评审一下"时触发。
  也适用于"这篇技术博客写得好不好"等请求。
---

# 技术文章质量分析 Skill

## 你的角色
你是一位资深技术编辑，擅长评估技术文章的深度、实用性和结构质量。

## 分析流程

### 第一步：基础信息提取
执行 `scripts/extract_stats.py` 获取文章的统计数据：
- 总字数、段落数
- 代码块数量及语言分布
- 标题层级结构
- 外部链接数量

### 第二步：四维度评分
参考 `references/scoring-criteria.md` 中的详细评分标准，对文章进行四维度评分：

| 维度 | 权重 | 核心判断点 |
|------|------|-----------|
| 技术深度 | 30% | 是否深入原理？有独创见解还是概念搬运？ |
| 实用性 | 30% | 代码能跑吗？有踩坑经验吗？ |
| 结构清晰度 | 20% | 逻辑是否递进？知识路径是否清晰？ |
| 原创度 | 20% | 有个人思考还是纯翻译/搬运？ |

每个维度打1-5分，加权计算总分（百分制）。

### 第三步：生成改进建议
基于评分结果，给出3条最关键的改进建议，每条包含：
- 问题描述
- 改进方向
- 具体操作

### 第四步：输出报告
使用 `assets/report-template.md` 的格式输出最终报告。

## 评分校准规则
- 有完整可运行代码 + 踩坑经验 → 实用性至少4分
- 只有概念解释，无代码 → 实用性最多2分
- 标题层级超过3级且逻辑递进 → 结构至少4分
- 全文无个人观点/经验 → 原创度最多2分

## 禁止事项
- ❌ 不得给出没有依据的高分（每个分数必须引用文章中的证据）
- ❌ 不得遗漏任何评分维度
- ❌ 改进建议不得空泛（如"提高代码质量"），必须具体可操作

# 技术文章评分标准详细定义

## 技术深度（weight: 30%）

| 分数 | 标准 | 典型特征 |
|------|------|---------|
| 5 | 有独创见解，深入底层原理 | 源码分析、性能基准测试、独特的架构思考 |
| 4 | 系统讲解，有实战经验 | 完整的技术方案、踩过的坑、解决过的问题 |
| 3 | 概念清晰，有代码示例 | 正确解释了核心概念，附带代码演示 |
| 2 | 表面讲解，缺乏深度 | 只说了"是什么"，没说"为什么" |
| 1 | 纯概念搬运，无实质内容 | 官方文档翻译、百科式罗列 |

## 实用性（weight: 30%）

| 分数 | 标准 | 典型特征 |
|------|------|---------|
| 5 | 代码可直接运行，附踩坑经验 | 完整项目代码、环境配置、常见错误解决 |
| 4 | 有完整代码示例 | 关键功能的完整实现代码 |
| 3 | 有代码片段 | 核心逻辑的代码片段，需要读者自行补全 |
| 2 | 只有概念，无代码 | 纯文字描述技术方案 |
| 1 | 纯理论，脱离实践 | 与实际开发完全脱节 |

## 结构清晰度（weight: 20%）

| 分数 | 标准 | 典型特征 |
|------|------|---------|
| 5 | 逻辑递进，有清晰的知识路径 | 从简到难、前后呼应、有总结回顾 |
| 4 | 结构合理，层次分明 | 标题有层级、段落有组织 |
| 3 | 基本有序 | 有结构但逻辑不够紧密 |
| 2 | 结构松散 | 内容东一块西一块 |
| 1 | 杂乱无章 | 无法找到阅读主线 |

## 原创度（weight: 20%）

| 分数 | 标准 | 典型特征 |
|------|------|---------|
| 5 | 全新视角，独家方法论 | 首次提出的框架/思路、独特的实验 |
| 4 | 有个人思考和经验 | 结合自身项目的反思、对比分析 |
| 3 | 适度引用+个人理解 | 综合多方资料并加入自己的解读 |
| 2 | 大量引用，少量原创 | 以引用为主，个人内容很少 |
| 1 | 纯翻译/搬运 | 直接翻译或复制其他文章 |

scripts/extract_stats.py

#!/usr/bin/env python3
"""提取技术文章的统计信息"""

import sys
import json
import re

def extract_stats(content: str) -> dict:
    lines = content.split('\n')

    # 统计字数（去除代码块和空行）
    in_code_block = False
    text_chars = 0
    for line in lines:
        if line.strip().startswith('```'):
            in_code_block = not in_code_block
            continue
        if not in_code_block:
            text_chars += len(line.strip())

    # 统计代码块
    code_blocks = re.findall(r'```(\w*)\n(.*?)```', content, re.DOTALL)
    languages = {}
    for lang, code in code_blocks:
        lang = lang or 'unknown'
        languages[lang] = languages.get(lang, 0) + 1

    # 统计标题层级
    headings = []
    for line in lines:
        match = re.match(r'^(#{1,6})\s+(.+)', line)
        if match:
            headings.append({
                'level': len(match.group(1)),
                'text': match.group(2).strip()
            })

    # 统计外部链接
    links = re.findall(r'https?://[^\s\)]+', content)

    return {
        'total_chars': len(content),
        'text_chars': text_chars,
        'total_lines': len(lines),
        'code_blocks': len(code_blocks),
        'code_languages': languages,
        'headings': headings,
        'heading_max_depth': max((h['level'] for h in headings), default=0),
        'external_links': len(links)
    }

if __name__ == '__main__':
    content = sys.stdin.read()
    stats = extract_stats(content)
    print(json.dumps(stats, ensure_ascii=False, indent=2))

         ▼
第四步：按模板格式输出报告

注意第三层的 references/scoring-criteria.md 只在需要评分时才加载——这就是渐进式披露的价值。如果这些评分标准全塞在 SKILL.md 里，每次触发都会消耗大量上下文。

Skills + MCP 的最佳组合

Skills 和 MCP 不是竞争关系，而是互补关系。最强形态是：MCP 负责连接数据，Skills 负责指挥流程。

以”代码审查”场景为例：

code-review/
├── SKILL.md
├── scripts/
│   └── complexity.py        # 计算圈复杂度
└── references/
    └── review-checklist.md  # 审查清单

---
name: code-review
description:
  对代码进行专业审查。当用户说"帮我review代码"、
  "看看这段代码有什么问题"、"代码审查"时触发。
---

# 代码审查 Skill

## 工作流程
1. 通过 MCP GitHub Server 的 `read_file` 工具获取代码
2. 执行 `scripts/complexity.py` 计算圈复杂度
3. 参考 `references/review-checklist.md` 逐项检查
4. 生成审查报告
5. 通过 MCP Slack Server 的 `send_message` 工具发送到 #code-review 频道

## 审查维度
- 命名规范
- 错误处理
- 性能隐患
- 安全漏洞
- 可测试性

## 严重级别
- 🔴 Critical：必须修复（安全漏洞、数据丢失风险）
- 🟡 Warning：建议修复（性能问题、不规范用法）
- 🔵 Info：可选优化（代码风格、命名建议）

在这个例子中：

  • MCP 提供了 GitHub 读取代码和 Slack 发送消息的能力（连接）
  • Skill 定义了审查流程、审查标准、输出格式（流程+知识）
  • 两者缺一不可
职责分工：

MCP:  "我能帮你读GitHub代码、发Slack消息"  （能做什么）
Skill: "读到代码后，按这个标准审查，       （怎么做）
        用这个格式输出，发到这个频道"

Skill 存放位置

以 Claude Code 为例（其他 AI IDE 类似），Skill 有四个存放层级：

存放位置 路径 适用范围
企业级 通过 managed settings 下发 组织内所有用户
个人级 ~/.claude/skills/<skill-name>/SKILL.md 你的所有项目
项目级 .claude/skills/<skill-name>/SKILL.md 仅当前项目
插件级 <plugin>/skills/<skill-name>/SKILL.md 插件启用的地方

同名覆盖规则：企业级 > 个人级 > 项目级。插件 Skill 使用 plugin-name:skill-name 命名空间，不会和其他层级冲突。

推荐做法：

  • 个人习惯类（如写作风格、commit 规范）→ 放个人目录
  • 团队标准类（如代码审查、API 设计规范）→ 放项目目录，git 管理

自动发现机制：当你在子目录中操作文件时，Claude Code 会自动向上遍历目录树，发现嵌套的 .claude/skills/ 目录并加载其中的 Skills。这对 monorepo 场景特别有用——每个 package 可以有自己的 Skills。

Skills 高级用法

前面讲了 Skills 的基础概念和结构，但 Skills 还有很多”隐藏能力”是很多人不知道的。这部分基于 Claude Code 官方文档，把这些高级用法梳理清楚。

Frontmatter 完整字段速查

SKILL.md 的 YAML frontmatter 支持的字段远不止 name 和 description：

字段 是否必需 说明
name 否 Skill 名称，省略则用目录名。只能用小写字母、数字和连字符
description 推荐 告诉 AI 什么时候用这个 Skill。超过 250 字符会被截断
argument-hint 否 自动补全时显示的参数提示，如 [issue-number]
disable-model-invocation 否 设为 true 则 AI 不能自动触发，只能用户手动 /name 调用
user-invocable 否 设为 false 则从 / 菜单隐藏，只有 AI 能调用
allowed-tools 否 Skill 激活时允许 AI 免确认使用的工具列表
model 否 指定使用的模型
effort 否 努力程度：low / medium / high / max
context 否 设为 fork 则在独立子 Agent 中执行
agent 否 当 context: fork 时，指定使用哪个子 Agent
hooks 否 Skill 生命周期内的 Hooks 配置
paths 否 Glob 模式，限制只在操作匹配路径的文件时才激活
shell 否 内联 shell 命令使用的 shell，bash（默认）或 powershell

控制谁能调用 Skill

默认情况下，用户和 AI 都能调用任何 Skill。但有些场景你需要精确控制：

场景1：只允许用户手动触发

部署、发消息这类有副作用的操作，你不希望 AI 自作主张：

---
name: deploy
description: 部署应用到生产环境
disable-model-invocation: true  # AI 不能自动触发
---

场景2：只允许 AI 自动触发

一些背景知识类的 Skill，用户手动调用没意义：

---
name: legacy-system-context
description: 旧系统的架构说明，当涉及旧系统迁移时使用
user-invocable: false  # 从 / 菜单隐藏
---

场景3：通过权限规则精细控制

在 /permissions 中可以用规则语法控制 AI 对 Skill 的访问：

# 只允许特定 Skill
Skill(commit)
Skill(review-pr *)

# 禁止特定 Skill
Skill(deploy *)

# 禁止所有 Skill
Skill

参数传递与变量替换

Skills 支持动态参数，这让同一个 Skill 能处理不同的输入。

基础参数：$ARGUMENTS 捕获用户输入的所有参数

---
name: review
description: 审查指定文件的代码质量
argument-hint: "<file-path>"
---

请审查以下文件的代码质量：$ARGUMENTS

调用方式：/review src/auth/login.ts

索引参数：$ARGUMENTS[N] 或简写 $N 访问特定参数

---
name: compare
description: 对比两个文件的实现差异
argument-hint: "<file1> <file2>"
arguments:
  - file1
  - file2
---

对比 $0 和 $1 的实现差异，分析各自的优缺点。

调用方式：/compare src/v1/auth.ts src/v2/auth.ts

多词参数用引号包裹：/my-skill "hello world" second，此时 $0 是 hello world，$1 是 second。

内置变量：

变量 说明
${CLAUDE_SESSION_ID} 当前会话 ID，用于日志关联
${CLAUDE_SKILL_DIR} Skill 的 SKILL.md 所在目录的绝对路径

${CLAUDE_SKILL_DIR} 特别有用——在 !command 中引用 Skill 自带的脚本时，不用担心当前工作目录是什么：

---
name: analyze
description: 分析代码库结构
---

执行以下脚本获取项目统计信息：
!`python3 ${CLAUDE_SKILL_DIR}/scripts/analyze.py`

内联 Shell 命令（!command 语法）

这是 Skills 最强大的”隐藏功能”之一。在 SKILL.md 中可以直接嵌入 shell 命令，Skill 加载时会自动执行命令并将输出注入上下文。

单行命令：用 !`command` 语法

---
name: git-status
description: 了解当前 Git 状态后再操作
---

当前 Git 状态：
!`git status --short`

当前分支：
!`git branch --show-current`

最近 5 次提交：
!`git log --oneline -5`

基于以上信息，$ARGUMENTS

多行命令块：用 ```! 代码块

---
name: project-overview
description: 快速了解项目全貌
---

项目依赖信息：
```!
cat package.json | python3 -c "
import sys, json
pkg = json.load(sys.stdin)
print(f'项目名: {pkg.get(\"name\", \"unknown\")}')
print(f'版本: {pkg.get(\"version\", \"unknown\")}')
deps = pkg.get('dependencies', {})
print(f'依赖数: {len(deps)}')
for name, ver in list(deps.items())[:10]:
    print(f'  {name}: {ver}')
"

基于以上项目信息，$ARGUMENTS


**关键点**：
- 命令在 Skill **加载时**执行，不是在 AI 回复时执行
- 命令的 stdout 会替换原始的 `!command` 语法，注入到 Skill 内容中
- 这意味着 AI 看到的是命令的**执行结果**，而不是命令本身
- 出于安全考虑，通过 MCP 加载的 Skills 禁止使用此功能

### 条件激活（paths 字段）

`paths` 字段让 Skill 只在操作特定文件时才激活，避免不相关的 Skill 污染上下文：

```yaml
---
name: test-conventions
description: 测试文件的编写规范
paths:
  - "src/**/*.test.ts"
  - "src/**/*.spec.ts"
---

编写测试时遵循以下规范：
1. 使用 describe/it 结构
2. 每个 it 只测一个行为
3. 测试描述用中文
...

只有当你操作了匹配 src/**/*.test.ts 或 src/**/*.spec.ts 的文件时，这个 Skill 才会出现在 AI 的上下文中。使用 gitignore 风格的路径匹配语法。

在子 Agent 中执行（context: fork）

默认情况下，Skill 的内容会注入当前对话（inline 模式）。但对于自包含的任务，可以让 Skill 在独立的子 Agent 中执行：

---
name: deep-research
description: 深度研究某个技术主题
context: fork
agent: Explore  # 使用内置的 Explore Agent（只读，速度快）
---

深度研究 $ARGUMENTS：
1. 搜索代码库中所有相关实现
2. 分析核心逻辑和设计模式
3. 总结关键发现，附具体文件引用

agent 字段可以指定内置 Agent（Explore、Plan、general-purpose）或自定义 Agent（.claude/agents/ 中定义的）。

inline vs fork 的选择：

  • inline：Skill 内容融入当前对话，适合参考类内容（编码规范、API 约定）
  • fork：独立子 Agent 执行，适合自包含任务（研究、分析、批量操作）

工具预授权（allowed-tools）

Skill 激活时，可以预先授权 AI 使用特定工具，免去每次确认：

---
name: quick-fix
description: 快速修复 lint 错误
allowed-tools:
  - Read
  - Edit
  - Bash(npm run lint *)
---

找到并修复所有 lint 错误：
1. 运行 `npm run lint` 查看错误列表
2. 逐个修复
3. 再次运行确认全部通过

这样 AI 在执行这个 Skill 时，使用 Read、Edit 和匹配 npm run lint * 的 Bash 命令都不需要用户确认。

Skill 内嵌 Hooks

Skill 可以在自己的生命周期内注册 Hooks，实现”Skill 执行时自动触发副作用”：

---
name: careful-edit
description: 谨慎编辑模式，每次写文件后自动运行测试
hooks:
  PostToolUse:
    - matcher: "Write|Edit"
      hooks:
        - type: command
          command: "npm test"
          once: true  # 只执行一次后自动移除
---

进入谨慎编辑模式。每次修改文件后会自动运行测试确保没有破坏。

once: true 表示这个 Hook 执行成功一次后就自动移除，不会反复触发。

选择指南

你需要 → 连接外部系统（数据库、API、文件系统等）
  → MCP（提供连接能力）

你需要 → 标准化的业务流程（代码审查、报告生成、数据分析等）
  → Skills（提供流程+知识）

你需要 → Skill 指挥 MCP 工具完成底层操作
  → MCP + Skills（最强组合）

决策流程图：

你的需求是什么？
      │
      ├── "AI 需要访问外部数据/系统"
      │     → MCP
      │
      ├── "AI 需要按固定流程产出标准化结果"
      │     → Skills
      │
      ├── "AI 需要连接外部系统 + 按标准流程产出"
      │     → MCP + Skills（最强组合）
      │
      └── "AI 需要遵守全局规则（代码风格等）"
            → Rules（不是 Skill 也不是 MCP）

Skills 的未来：AI 能力市场

**Skills 正在成为 AI 时代的”App”**。

就像移动互联网时代的 App Store，AI 时代正在形成 “Skill Store”：

  • 开发者可以发布和分享 Skill（GitHub 仓库）
  • 用户可以一键安装 Skill 扩展 AI 能力
  • Skill 之间可以组合和协作

为什么 Skills 比”长 Prompt”好

有人会问：我直接写一个很长的 System Prompt，把流程和知识都塞进去，不就行了？

区别在于：

维度 长 Prompt Skill
Token 消耗 每次对话全量加载 渐进式按需加载
可维护性 一个巨大的文本块 分层组织，各司其职
可复用性 复制粘贴 文件夹复制/包管理安装
团队协作 无法版本管理 Git 管理，可 Code Review
确定性操作 纯靠 AI 推理 脚本执行，结果确定

Skill 的核心价值不是”给 AI 更多信息”，而是”以工程化的方式管理 AI 的行为”。

小结

Skills vs MCP 核心要点：

  1. Skill 是文件夹，不是代码模块——核心是 SKILL.md（元数据 + 指令），辅以 scripts、references、assets
  2. MCP 是协议层——解决”AI 怎么连接工具”
  3. Skills 是能力层——解决”AI 怎么按标准完成业务”
  4. 渐进式披露——三层加载机制（元数据 → 正文 → 资源），精准控制 Token 消耗
  5. 两者互补——MCP 负责连接，Skills 负责流程，组合使用最强
  6. !command 语法是 Skills 的杀手锏——加载时自动执行 shell 命令，将动态信息注入上下文
  7. 调用控制很重要——disable-model-invocation 防止 AI 自作主张，user-invocable: false 隐藏纯背景知识
  8. context: fork 让 Skill 在独立子 Agent 中执行——适合自包含的研究、分析任务
