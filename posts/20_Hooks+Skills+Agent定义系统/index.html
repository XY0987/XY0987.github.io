Hooks + Skills + Agent 定义系统 — Claude Code 源码深度分析

一句话理解

Claude Code 的扩展三件套：Hooks 是”在特定时机自动触发的钩子”，Skills 是”用户/团队定义的可复用工作流”，Agent 定义是”给子 Agent 装上不同的大脑和工具箱”。三者共同构成了 Claude Code 的可编程扩展层。

白话版理解

Hooks — 自动触发的”监听器”

类比：就像 Git 的 pre-commit hook，或者 IDE 的”保存时自动格式化”。你告诉系统：”每次 Claude 写文件之后，帮我跑一下 prettier”，系统就会在每次写文件后自动执行这个命令。

五种触发方式：

触发方式 类比 例子
命令 Hook（command） 跑一个 shell 命令 写完文件后自动格式化
提示词 Hook（prompt） 让 LLM 判断一下 “这个 bash 命令安全吗？”
Agent Hook（agent） 派一个子 Agent 去验证 “帮我跑一下测试，看看有没有挂”
HTTP Hook（http） 调用远程 Webhook 通知外部审计系统、远程审批
函数 Hook（function） TypeScript 回调（内部专用） SDK 结构化输出验证

27 个事件点：从”用户提交消息”到”工具调用前后”，再到”会话开始/结束”，几乎覆盖了 Claude Code 运行的每一个关键节点。

Skills — 可复用的”工作流脚本”

类比：就像 IDE 的代码片段（snippet），但功能强大得多。你可以写一个 SKILL.md 文件，描述一套完整的工作流程（比如”提交代码并创建 PR”），然后用 /my-skill 命令一键触发。

四种来源：

内置 Skills（随 CLI 打包）
    ↓
插件 Skills（通过 /plugin 安装）
    ↓
用户 Skills（~/.claude/skills/）
    ↓
项目 Skills（.claude/skills/）

条件激活：Skills 还支持”路径触发”——只有当你操作了匹配特定路径的文件时，这个 Skill 才会出现在上下文里。比如只有修改了 *.test.ts 文件时，测试相关的 Skill 才会被激活。

Agent 定义 — 给子 Agent 装上”专属大脑”

类比：就像公司里的不同岗位。”探索专员”只能查文件不能改文件，”架构师”只能出方案不能动代码，”验证员”专门找 bug 不能修 bug。每个岗位有自己的工具权限和行为准则。

三种 Agent 类型：

类型 来源 特点
内置 Agent 随 CLI 打包 代码写死，不可修改
自定义 Agent .claude/agents/ 目录 JSON 文件定义，可版本控制
插件 Agent 通过插件安装 第三方扩展

以下是源码级别的详细分析。

1.3 Hook 的五种类型

类型一：Command Hook（最常用）

{
  "type": "command",
  "command": "prettier --write $FILE",
  "timeout": 30,
  "statusMessage": "格式化中..."
}

退出码语义：

  • exit 0：成功，stdout 仅在 transcript 模式下显示
  • exit 2：阻断，stderr 直接注入模型上下文
  • 其他：显示 stderr 给用户，但继续执行

JSON 输出控制（hook 可以输出 JSON 来影响行为）：

{
  "systemMessage": "显示给用户的消息",
  "continue": false,           // 阻断执行
  "stopReason": "阻断原因",
  "hookSpecificOutput": {
    "hookEventName": "PreToolUse",
    "permissionDecision": "allow|deny|ask",  // 权限决策
    "updatedInput": { ... },                  // 修改工具输入
    "additionalContext": "注入模型的上下文"   // PostToolUse 专用
  }
}

类型二：Prompt Hook（LLM 判断）

{
  "type": "prompt",
  "prompt": "这个 bash 命令安全吗？$ARGUMENTS"
}

仅适用于工具事件：PreToolUse、PostToolUse、PermissionRequest。

类型三：Agent Hook（派子 Agent）

{
  "type": "agent",
  "prompt": "验证测试是否通过：$ARGUMENTS"
}

同样仅适用于工具事件，会派出一个子 Agent 来执行验证逻辑。

类型四：HTTP Hook（远程 Webhook）

{
  "type": "http",
  "url": "https://my-server.com/webhook/hook-event",
  "timeout": 30
}

HTTP Hook 向指定 URL 发送 HTTP 请求，适用于需要与外部服务集成的场景（如通知系统、审计日志、远程审批等）。

类型五：Function Hook（TypeScript 回调，内部专用）

// src/utils/hooks/sessionHooks.ts
export type FunctionHook = {
  type: 'function'
  id?: string
  timeout?: number
  callback: FunctionHookCallback  // (messages, signal) => boolean | Promise<boolean>
  errorMessage: string
  statusMessage?: string
}

这是内部 SDK 专用的 hook 类型，不暴露在 settings.json 配置中。用于结构化输出强制验证等场景。

1.4 Hook 的三种注册来源

来源一：settings.json 持久化配置

// ~/.claude/settings.json 或 .claude/settings.json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          {
            "type": "command",
            "command": "prettier --write $(jq -r '.tool_input.file_path')"

来源二：Skill/Agent frontmatter 内嵌

name: my-skill

  PostToolUse:
    - matcher: "Write"
      hooks:
        - type: command
          command: "npm test"
          once: true   # 只执行一次后自动移除
---

Agent 的特殊处理：Agent 的 Stop hook 会被自动转换为 SubagentStop，因为子 Agent 触发的是 SubagentStop 而不是 Stop。

来源三：插件注册

// src/utils/plugins/loadPluginHooks.ts

1.5 Hook 执行流程（以 PreToolUse 为例）

用户调用工具
    │
    ▼
toolExecution.ts: runPreToolUseHooks()
    │
    ▼
hooks.ts: executePreToolHooks()
    │  构造 PreToolUseHookInput（含 tool_name, tool_input 等）
    ▼
hooks.ts: executeHooks()
    │  getMatchingHooks() — 按 matcher 过滤匹配的 hooks
    │  去重（相同 command/prompt/agent 只执行一次）
    ▼
    ├─→ command hooks → 并行执行所有 shell 命令
    ├─→ prompt hooks  → 并行调用 LLM 判断
    ├─→ agent hooks   → 并行派出子 Agent
    └─→ function hooks → 并行执行 TypeScript 回调
    │
    ▼
聚合结果 (AggregatedHookResult)
    │
    ├─ blocked=true → 阻断工具调用，返回错误给模型
    ├─ updatedInput  → 用修改后的输入执行工具
    ├─ additionalContext → 工具执行后注入模型上下文
    └─ systemMessage → 显示消息给用户

关键代码路径：

  • src/services/tools/toolExecution.ts — 工具执行主流程，调用 runPreToolUseHooks / runPostToolUseHooks
  • src/services/tools/toolHooks.ts — hook 执行的中间层，处理结果并 yield 消息
  • src/utils/hooks.ts — hook 执行的核心逻辑（3000+ 行）

1.6 Session Hook — 临时的会话级 Hook

Session Hook 是存储在内存中的临时 hook，随会话结束而消失。它们存储在 AppState.sessionHooks（一个 Map<sessionId, SessionStore>）中。

// 添加 session hook（Skill 调用时注册）
addSessionHook(setAppState, sessionId, 'PostToolUse', 'Write', hookCommand)

// 添加一次性 hook（once: true）
// 执行成功后通过 onHookSuccess 回调自动移除

// 添加 function hook（SDK 内部使用）
addFunctionHook(setAppState, sessionId, 'PostToolUse', '', callback, '验证失败')

为什么用 Map 而不是 Record：高并发场景下（如 parallel() 同时启动 N 个 schema-mode agents），每次 addFunctionHook 都会触发 setAppState。用 Map 的 .set() 是 O(1) 且不改变容器引用，避免了 O(N²) 的复制开销和不必要的 store listener 触发。

1.7 PostSampling Hook — 采样后的内部钩子

// src/utils/hooks/postSamplingHooks.ts
// 这是一个内部 API，不暴露在 settings.json 中
// 在模型采样完成后调用，用于内部功能（如 cost tracking）

export function registerPostSamplingHook(hook: PostSamplingHook): void {
  postSamplingHooks.push(hook)
}

2.2 Skill 的文件结构

Skills 使用目录格式存储，每个 Skill 是一个独立目录：

.claude/skills/
└── my-workflow/
    ├── SKILL.md          # 主文件（必须）
    └── helper.sh         # 辅助脚本（可选）

SKILL.md 完整 frontmatter 字段：

---
name: my-workflow                    # Skill 名称（用于 /my-workflow 调用）
description: 一句话描述这个 Skill     # 显示在 /help 中
argument-hint: "<branch> [message]"  # 参数提示
arguments:                           # 参数名列表（用于 $arg_name 替换）
  - branch
  - message
allowed-tools:                       # 允许使用的工具（最小权限原则）
  - Bash(git:*)
  - Write
  - Edit
when_to_use: |                       # 告诉模型何时自动调用此 Skill
  当用户想提交代码并创建 PR 时使用。
  触发词：'提交代码', 'create PR', 'push and PR'
model: sonnet                        # 指定模型（可选）
context: fork                        # 执行上下文：inline（默认）或 fork
agent: my-agent                      # 指定使用的 Agent 类型
user-invocable: true                 # 是否可以用 /skill-name 调用
effort: medium                       # 努力程度：low/medium/high 或整数
paths:                               # 条件激活：只有操作这些路径时才激活
  - "src/tests/**"
hooks:                               # Skill 执行时注册的 hooks
  PostToolUse:
    - matcher: "Write"
      hooks:
        - type: command
          command: "npm test"
          once: true
version: "1.0"                       # 版本号
---

2.3 Skill 加载流程

启动时
    │
    ├─→ initBundledSkills()     — 注册所有内置 Skills
    │
    └─→ getSkillDirCommands(cwd)  — 从目录加载
            │
            ├─ managed skills dir  (企业策略，最高优先级)
            ├─ user skills dir     (~/.claude/skills/)
            ├─ project skills dirs (.claude/skills/ 向上遍历到 home)
            ├─ additional dirs     (--add-dir 指定的目录)
            └─ legacy commands dir (.claude/commands/ 兼容旧格式)
            │
            ▼
        去重（通过 realpath 解析符号链接，避免重复加载）
            │
            ▼
        分离条件 Skills（有 paths 字段的）
            │
            ├─ 无条件 Skills → 立即加入上下文
            └─ 条件 Skills  → 存入 conditionalSkills Map，等待激活

动态发现：当 Claude 操作文件时，系统会向上遍历目录树，发现新的 .claude/skills/ 目录并动态加载其中的 Skills。

条件激活：

// 当文件被操作时，检查是否有条件 Skills 匹配
activateConditionalSkillsForPaths(filePaths, cwd)
// 使用 gitignore 风格的路径匹配（ignore 库）
// 匹配成功 → 移入 dynamicSkills Map → 下次对话时出现在上下文

/simplify — 三 Agent 并行代码审查

/simplify 触发
    │
    ▼
读取 git diff 获取变更内容
    │
    ▼
并行启动 3 个子 Agent：
    ├─ Agent 1: 代码复用审查（查找重复逻辑，建议使用现有工具函数）
    ├─ Agent 2: 代码质量审查（冗余状态、参数膨胀、抽象泄漏等）
    └─ Agent 3: 效率审查（不必要的工作、错过的并发机会、内存泄漏等）

    │

2.5 Skill 执行机制

// src/skills/loadSkillsDir.ts: createSkillCommand()
async getPromptForCommand(args, toolUseContext) {
  // 1. 拼接 base directory 前缀（让模型知道 Skill 的根目录）
  let finalContent = baseDir
    ? `Base directory for this skill: ${baseDir}\n\n${markdownContent}`
    : markdownContent

  // 2. 参数替换（$arg_name → 实际值）
  finalContent = substituteArguments(finalContent, args, true, argumentNames)

  // 3. 替换特殊变量
  finalContent = finalContent.replace(/\$\{CLAUDE_SKILL_DIR\}/g, skillDir)
  finalContent = finalContent.replace(/\$\{CLAUDE_SESSION_ID\}/g, getSessionId())

  // 4. 执行内联 shell 命令（!`...` 语法，MCP Skills 禁止此功能）
  if (loadedFrom !== 'mcp') {
    finalContent = await executeShellCommandsInPrompt(finalContent, ...)
  }

  return [{ type: 'text', text: finalContent }]
}

两种执行上下文：

  • inline（默认）：Skill 的提示词注入当前对话，由主 Agent 执行
  • fork：Skill 作为独立子 Agent 执行，有自己的上下文，适合自包含任务

自定义 Agent 存储在 .claude/agents/ 目录，支持两种格式：

JSON 格式（推荐）

3.5 Agent 加载优先级

policySettings（企业策略，最高优先级）
    ↓
flagSettings（Feature Flag 控制）
    ↓
plugin（插件提供）
    ↓
userSettings（~/.claude/agents/）
    ↓
projectSettings（.claude/agents/）
    ↓
built-in（内置，最低优先级）

同名覆盖规则：高优先级来源的 Agent 会覆盖低优先级的同名 Agent。

3.6 Agent 启动时的 Hook 注册

当一个 Agent 被派出时，如果它的定义中包含 hooks 字段，这些 hooks 会被注册为 session-scoped hooks：

// src/utils/hooks/registerFrontmatterHooks.ts
registerFrontmatterHooks(
  setAppState,
  agentId,        // 以 agentId 为 sessionId，确保 hooks 只在此 Agent 的会话中生效
  agentDef.hooks,
  `agent '${agentDef.agentType}'`,
  true,           // isAgent=true → Stop hooks 自动转换为 SubagentStop
)

Agent 结束时，其 session hooks 会被自动清理（clearSessionHooks(setAppState, agentId)）。

四、三者的协作关系

4.1 Skill 调用 Agent

Skill 可以通过 agent 字段指定使用特定 Agent 执行：

---
name: deep-review
context: fork
agent: code-reviewer    # 使用自定义的 code-reviewer Agent
---

或者在 Skill 的提示词中直接调用 Agent Tool：

## 步骤
1. 使用 Agent 工具派出 Explore Agent 搜索相关文件
2. 使用 Agent 工具派出 Plan Agent 设计方案

4.2 Agent 内嵌 Skill

Agent 定义中的 skills 字段可以预加载特定 Skills：

{
  "description": "全栈开发 Agent",
  "skills": ["simplify", "my-test-workflow"],
  "prompt": "你是一个全栈开发者..."
}

这些 Skills 会在 Agent 启动时注入到其上下文中，Agent 可以直接调用 /simplify 等命令。

4.3 Hook 触发 Skill/Agent

通过 agent 类型的 Hook，可以在特定事件发生时自动派出 Agent：

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          {
            "type": "agent",
            "prompt": "验证刚才的修改是否正确，运行相关测试"
          }
        ]
      }
    ]
  }
}

4.4 完整协作示例

用户："/batch 将所有 React class components 迁移到 hooks"
    │
    ▼
/batch Skill 被触发（bundled skill）
    │
    ▼
Phase 1: 进入 Plan Mode
    ├─ 派出 Explore Agent → 搜索所有 class components
    └─ 派出 Plan Agent → 设计迁移方案
    │
    ▼
Phase 2: 并行派出 N 个 general-purpose Agents
    ├─ 每个 Agent 在独立 worktree 中工作
    ├─ 每个 Agent 完成后调用 /simplify Skill（代码审查）
    └─ 每个 Agent 创建 PR
    │
    ▼
PostToolUse Hook（如果配置了）
    └─ 每次 Write/Edit 后自动运行 prettier
    │
    ▼
Phase 3: verification Agent（如果启用）
    └─ 验证迁移结果，输出 VERDICT: PASS/FAIL

五、核心设计理念

5.1 最小权限原则

每个 Agent 和 Skill 都应该只拥有完成任务所需的最小工具集：

  • Explore Agent：只读工具，明确禁止写操作
  • Plan Agent：只读工具，只能出方案不能动代码
  • Verification Agent：只读 + /tmp 写权限，不能修改项目文件

5.2 声明式配置

Agent 和 Skill 都通过声明式的 frontmatter/JSON 定义，而不是命令式代码：

  • whenToUse 字段让主 Agent 自动决定何时派出子 Agent
  • paths 字段让 Skill 自动在相关文件被操作时激活
  • hooks 字段让 Agent/Skill 在特定事件时自动触发副作用

5.3 分层覆盖

配置遵循”企业策略 > 用户 > 项目”的优先级，允许在不同层级定制行为，同时保证企业安全策略不被绕过。

5.4 会话隔离

Session Hooks 以 sessionId（或 agentId）为键存储，确保不同 Agent 的 hooks 互不干扰。Agent 结束时自动清理其 hooks，防止泄漏到其他 Agent。
