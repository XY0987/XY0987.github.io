MCP协议深度解析：AI工具调用的标准化革命

发表于2025|更新于2026|AI相关
|字数总计:2k|阅读时长:8分钟

MCP 对 AI 生态的意义，类似于 HTTP 对互联网的意义——不是某个产品，而是一个让整个生态互联互通的协议层。

为什么需要 MCP

没有 MCP 的混乱世界

// 给 OpenAI 写工具
const openaiTools = [{
  type: 'function',
  function: { name: 'get_weather', parameters: { /* OpenAI格式 */ } }
}];

// 给 Claude 写工具——格式不同！
const claudeTools = [{
  name: 'get_weather',
  input_schema: { /* Anthropic格式 */ }
}];

// 给 Gemini 写工具——又不一样！
const geminiTools = [{
  functionDeclarations: [{ name: 'get_weather', parameters: { /* Google格式 */ } }]
}];

// 同一个"查天气"功能，写了三遍！
// 如果天气API变了，要改三个地方！

MCP 的解决方案：一次开发，到处使用

MCP的核心理念：

┌─────────┐   ┌─────────┐   ┌─────────┐
│ ChatGPT │   │ Claude  │   │ Cursor  │   ← AI应用（MCP Client）
└────┬────┘   └────┬────┘   └────┬────┘
     │             │             │
     └─────────────┼─────────────┘
                   │
          ┌────────┴────────┐
          │  MCP 标准协议    │   ← 统一的接口
          └────────┬────────┘
                   │
     ┌─────────────┼─────────────┐
     │             │             │
┌────┴────┐  ┌────┴────┐  ┌────┴────┐
│天气Server│  │数据库   │  │GitHub   │   ← MCP Server
└─────────┘  │ Server  │  │ Server  │
             └─────────┘  └─────────┘

MCP 架构核心概念

三大能力

能力 说明 类比
Tools（工具） 可执行的函数（主动操作） 餐厅的菜单上可以点的菜
Resources（资源） 可读取的数据（被动读取） 餐厅的酒水单
Prompts（提示词模板） 预定义的Prompt模板 餐厅的推荐套餐

传输层演进

这里要重点讲 2025年3月26日的重大更新：

初版（2024.11）：
  ├── stdio —— 本地进程通信（通过标准输入输出）
  └── HTTP + SSE —— 远程通信（HTTP请求 + SSE长连接）

0326版（2025.03.26）：
  ├── stdio —— 不变
  └── Streamable HTTP —— 取代 HTTP+SSE
      ├── 统一的 /message 端点
      ├── 可选升级为 SSE 流
      ├── 支持无状态模式
      └── 原生兼容 Serverless

Streamable HTTP 的核心改进：

  1. 去掉了专用的 /sse 端点——不再需要维护长连接
  2. 统一 /message 端点——所有通信走一个入口
  3. 按需升级为流式——简单请求用普通HTTP，复杂请求自动升级为SSE
  4. 支持无状态部署——完美适配 Vercel、CloudFlare Workers 等 Serverless 平台

实战：开发一个 MCP Server

项目初始化

mkdir my-mcp-server
cd my-mcp-server
npm init -y
npm install @modelcontextprotocol/sdk

完整的 MCP Server 实现

// server.ts —— 一个提供待办事项管理能力的 MCP Server
import { Server } from '@modelcontextprotocol/sdk/server/index.js';
import { StdioServerTransport } from '@modelcontextprotocol/sdk/server/stdio.js';
import {
  CallToolRequestSchema,
  ListToolsRequestSchema,
  ListResourcesRequestSchema,
  ReadResourceRequestSchema
} from '@modelcontextprotocol/sdk/types.js';

// 简单的内存存储
interface Todo {
  id: string;
  title: string;
  completed: boolean;
  createdAt: string;
}

const todos: Todo[] = [];
let nextId = 1;

// 创建 Server 实例
const server = new Server(
  { name: 'todo-server', version: '1.0.0' },
  { capabilities: { tools: {}, resources: {} } }
);

// ========== 注册工具 ==========
server.setRequestHandler(ListToolsRequestSchema, async () => ({
  tools: [
    {
      name: 'add_todo',
      description: '添加一个待办事项',
      inputSchema: {
        type: 'object',
        properties: {
          title: { type: 'string', description: '待办事项标题' }
        },
        required: ['title']
      }
    },
    {
      name: 'complete_todo',
      description: '标记待办事项为完成',
      inputSchema: {
        type: 'object',
        properties: {
          id: { type: 'string', description: '待办事项ID' }
        },
        required: ['id']
      }
    },
    {
      name: 'list_todos',
      description: '列出所有待办事项',
      inputSchema: {
        type: 'object',
        properties: {
          filter: {
            type: 'string',
            enum: ['all', 'active', 'completed'],
            description: '过滤条件，默认 all'
          }
        }
      }
    }
  ]
}));

// 处理工具调用
server.setRequestHandler(CallToolRequestSchema, async (request) => {
  const { name, arguments: args } = request.params;

  switch (name) {
    case 'add_todo': {
      const todo: Todo = {
        id: String(nextId++),
        title: args.title as string,
        completed: false,
        createdAt: new Date().toISOString()
      };
      todos.push(todo);
      return {
        content: [{ type: 'text', text: `✅ 已添加待办: [${todo.id}] ${todo.title}` }]
      };
    }

    case 'complete_todo': {
      const todo = todos.find(t => t.id === args.id);
      if (!todo) {
        return {
          content: [{ type: 'text', text: `❌ 未找到ID为 ${args.id} 的待办事项` }],
          isError: true
        };
      }
      todo.completed = true;
      return {
        content: [{ type: 'text', text: `✅ 已完成: [${todo.id}] ${todo.title}` }]
      };
    }

    case 'list_todos': {
      const filter = (args.filter as string) || 'all';
      let filtered = todos;
      if (filter === 'active') filtered = todos.filter(t => !t.completed);
      if (filter === 'completed') filtered = todos.filter(t => t.completed);

      const text = filtered.length === 0
        ? '📝 暂无待办事项'
        : filtered.map(t =>
            `${t.completed ? '✅' : '⬜'} [${t.id}] ${t.title}`
          ).join('\n');

      return { content: [{ type: 'text', text }] };
    }

    default:
      return {
        content: [{ type: 'text', text: `未知工具: ${name}` }],
        isError: true
      };
  }
});

// ========== 注册资源 ==========
server.setRequestHandler(ListResourcesRequestSchema, async () => ({
  resources: [
    {
      uri: 'todo://summary',
      name: '待办事项概览',
      mimeType: 'text/plain',
      description: '当前所有待办事项的统计概览'
    }
  ]
}));

server.setRequestHandler(ReadResourceRequestSchema, async (request) => {
  if (request.params.uri === 'todo://summary') {
    const total = todos.length;
    const completed = todos.filter(t => t.completed).length;
    const active = total - completed;

    return {
      contents: [{
        uri: 'todo://summary',
        mimeType: 'text/plain',
        text: `待办统计: 总计${total}项, 进行中${active}项, 已完成${completed}项`
      }]
    };
  }
  throw new Error(`Unknown resource: ${request.params.uri}`);
});

// ========== 启动 ==========
const transport = new StdioServerTransport();
await server.connect(transport);
console.error('Todo MCP Server started');  // stderr 不影响 stdio 通信

MCP vs Function Calling：本质区别

维度 Function Calling MCP
耦合度 工具定义嵌在 API 调用中，紧耦合 工具定义独立于AI应用，松耦合
标准化 每个平台格式不同 统一标准协议
复用性 每个应用单独定义 一次开发，任何 MCP Client 都能用
发现性 静态定义 动态发现（Client可以查询Server的能力）
生态 各自为政 共享生态（社区贡献的Server可直接使用）

Function Calling 是”内联工具”，MCP 是”外挂工具”。就像编程中的”硬编码”vs”配置化”——MCP 让工具真正成为了可插拔的组件。

Streamable HTTP 深度解析

这是 2025年3月26日 MCP 最重要的更新，我认为值得专门讲解。

为什么要替换 HTTP+SSE

原方案的问题：

原方案（HTTP+SSE）：
Client ──POST──→ Server（发送请求）
Client ←──SSE──── Server（接收响应，需要维持长连接）

问题：
1. SSE 长连接在 Serverless 环境下不可行（函数执行有时间限制）
2. 断线重连时状态丢失
3. 需要两个端点（/message + /sse），部署复杂

Streamable HTTP 如何解决

新方案（Streamable HTTP）：
Client ──POST /message──→ Server
         ←── 普通HTTP响应（简单请求）
         ←── SSE流响应（复杂请求，按需升级）

改进：
1. 单一端点 /message
2. 简单请求直接返回，不需要长连接
3. 只在需要流式时才升级为SSE
4. 通过 Mcp-Session-Id 管理会话状态
5. Server可以完全无状态——适配 Serverless

// Streamable HTTP 服务端示例（概念代码）
app.post('/message', async (req, res) => {
  const { method, params } = req.body;
  const sessionId = req.headers['mcp-session-id'];

  // 简单请求：直接JSON响应
  if (method === 'tools/list') {
    return res.json({ tools: [...] });
  }

  // 需要流式的请求：升级为SSE
  if (method === 'tools/call' && isLongRunning(params)) {
    res.setHeader('Content-Type', 'text/event-stream');

    // 流式返回进度和结果
    res.write(`data: ${JSON.stringify({ type: 'progress', value: 30 })}\n\n`);
    // ... 执行工具 ...
    res.write(`data: ${JSON.stringify({ type: 'result', value: result })}\n\n`);
    res.end();
  }
});

MCP 的终局会是什么？

**MCP 正在从”工具协议”扩展为”AI 能力协议”**。

最初 MCP 只是让 AI 调用工具，但随着 Resources、Prompts 等能力的加入，它实际上在定义 AI 应用如何获取上下文（Context）的通用方式——这也是它名字中”Context Protocol”的真正含义。

对开发者的影响

如果你现在还在为每个 AI 平台单独开发工具接口，是时候切换到 MCP 了。一次投入，长期受益。

小结

MCP 核心要点：

  1. MCP 是 AI 工具调用的标准化协议——“一次开发，到处使用”
  2. 三大能力：Tools（工具）、Resources（资源）、Prompts（模板）
  3. Streamable HTTP 是最新传输层——解决了远程部署和Serverless的问题
  4. MCP 正在成为事实标准——主流AI平台都在跟进
