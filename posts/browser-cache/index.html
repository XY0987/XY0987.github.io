浏览器缓存完全指南：从强缓存到协商缓存

发表于2024|更新于2026|基础知识
|字数总计:2.1k|阅读时长:8分钟

浏览器缓存是前端性能优化中最重要的手段之一。合理使用缓存可以大幅减少网络请求，提升页面加载速度。本文将全面解析浏览器缓存机制。

一、为什么需要浏览器缓存

1.1 缓存的好处

┌─────────────┐
│   浏览器    │ ← 第一次请求：从服务器获取资源
└─────────────┘
       ↓
┌─────────────┐
│   缓存中    │ ← 后续请求：从缓存读取资源
└─────────────┘
  1. 减少网络请求：降低带宽消耗
  2. 提升加载速度：减少响应时间
  3. 减轻服务器压力：减少服务器请求量
  4. 提升用户体验：页面秒开

1.2 缓存的分类

缓存位置 存储位置 特点
Service Worker Cache 浏览器独立线程 可编程控制，离线缓存
Memory Cache 内存 快速，但容量小，关闭 Tab 即失效
Disk Cache 硬盘 容量大，持久化
Push Cache HTTP/2 Session 会话级缓存，时间短

二、浏览器缓存策略

浏览器缓存分为两种策略：

  1. 强缓存：不需要向服务器验证，直接使用本地缓存
  2. 协商缓存：需要向服务器验证缓存是否有效
请求流程：

     ┌──────────────┐
     │  发起请求    │
     └──────────────┘
            ↓
     ┌──────────────┐
     │ 强缓存有效？ │
     └──────────────┘
      ↓           ↓
    有效         无效
      ↓           ↓
┌──────────┐  ┌──────────────┐
│使用缓存  │  │ 协商缓存有效？│
└──────────┘  └──────────────┘
                ↓           ↓
              有效         无效
                ↓           ↓
          ┌──────────┐  ┌──────────┐
          │使用缓存  │  │请求服务器│
          └──────────┘  └──────────┘

三、强缓存

强缓存通过 HTTP 响应头设置，不需要向服务器发送请求。

3.1 Expires

Expires: Wed, 22 Oct 2028 07:28:00 GMT
  • HTTP/1.0
  • 指定资源的过期时间（绝对时间）
  • 缺点：依赖客户端时间，如果客户端时间不准确会导致缓存失效
// 示例：设置 Expires
// 服务端
response.setHeader('Expires', new Date(Date.now() + 86400000).toUTCString())

3.2 Cache-Control

Cache-Control: max-age=31536000
  • HTTP/1.1
  • 优先级高于 Expires
  • 使用相对时间，更可靠

Cache-Control 指令详解

指令 说明 示例
max-age 缓存最大有效期（秒） max-age=31536000
no-cache 强制使用协商缓存 no-cache
no-store 禁止缓存 no-store
public 可被任何缓存存储 public
private 只能被浏览器缓存 private
must-revalidate 过期后必须验证 must-revalidate
immutable 资源永不变化 immutable
stale-while-revalidate 后台异步更新 stale-while-revalidate=60 
# 常见配置示例

# 静态资源（长期缓存）
Cache-Control: max-age=31536000, immutable

# HTML 文件（不缓存）
Cache-Control: no-cache

# 需要验证的缓存
Cache-Control: max-age=0, must-revalidate

# CDN 可以缓存
Cache-Control: public, max-age=86400

# 只能浏览器缓存
Cache-Control: private, max-age=3600

3.3 强缓存的使用场景

// Webpack 打包的静态资源
// 文件名带 hash，可以设置长期缓存
// main.a1b2c3d4.js
// app.e5f6g7h8.css

// Express 设置强缓存
app.use(express.static('public', {
    maxAge: '1y',
    immutable: true
}))

四、协商缓存

协商缓存需要向服务器验证缓存是否有效。

4.1 Last-Modified / If-Modified-Since

# 首次请求响应
Last-Modified: Wed, 21 Oct 2028 07:28:00 GMT

# 再次请求
If-Modified-Since: Wed, 21 Oct 2028 07:28:00 GMT

# 服务端响应（未修改）
HTTP/1.1 304 Not Modified

工作流程：

  1. 浏览器首次请求资源，服务器返回 Last-Modified 头
  2. 浏览器再次请求，携带 If-Modified-Since 头
  3. 服务器比较时间，未修改返回 304，修改返回 200 + 新资源

缺点：

  • 只能精确到秒
  • 文件内容没变但修改时间变了会误判
  • 文件内容变了但修改时间没变会漏判

4.2 ETag / If-None-Match

# 首次请求响应
ETag: "a1b2c3d4e5f6"

# 再次请求
If-None-Match: "a1b2c3d4e5f6"

# 服务端响应（未修改）
HTTP/1.1 304 Not Modified

工作流程：

  1. 浏览器首次请求资源，服务器返回 ETag 头（资源唯一标识）
  2. 浏览器再次请求，携带 If-None-Match 头
  3. 服务器比较 ETag，未修改返回 304，修改返回 200 + 新资源

优点：

  • 更精确，基于内容生成标识
  • 不受时间限制

缺点：

  • 计算 ETag 需要消耗服务器资源

4.3 两种协商缓存的优先级

ETag 优先级 > Last-Modified
对比项 Last-Modified ETag
准确性 秒级精度 精确到内容变化
性能 只记录时间 需要计算哈希
优先级 低 高

4.4 协商缓存实现

// Express 实现
const express = require('express')
const app = express()

app.get('/api/data', (req, res) => {
  const data = { name: 'test' }
  const etag = require('crypto')
    .createHash('md5')
    .update(JSON.stringify(data))
    .digest('hex')
  
  // 设置 ETag
  res.set('ETag', etag)
  
  // 比较 ETag
  if (req.headers['if-none-match'] === etag) {
    return res.status(304).end()
  }
  
  res.json(data)
})

// Koa 实现
const Koa = require('koa')
const conditional = require('koa-conditional-get')
const etag = require('koa-etag')

const app = new Koa()

// 条件 GET 中间件
app.use(conditional())
// ETag 中间件
app.use(etag())

五、缓存决策指南

5.1 不同资源的缓存策略

资源类型 缓存策略 配置示例
HTML 协商缓存 Cache-Control: no-cache
JS/CSS（带 hash） 强缓存 Cache-Control: max-age=31536000, immutable
图片 强缓存 Cache-Control: max-age=86400
字体 强缓存 Cache-Control: max-age=31536000
API 数据 按需 根据业务场景决定

5.2 决策流程图

            ┌─────────────────┐
            │  资源是否会变化  │
            └─────────────────┘
                   ↓
         ┌─────────┴─────────┐
         ↓                   ↓
       不变化               会变化
         ↓                   ↓
  ┌──────────────┐   ┌──────────────┐
  │长期强缓存     │   │是否有版本控制│
  │max-age=1年   │   └──────────────┘
  │immutable     │          ↓
  └──────────────┘    ┌──────┴──────┐
                      ↓             ↓
                    有            没有
                      ↓             ↓
              ┌──────────────┐ ┌──────────────┐
              │文件名带 hash │ │使用协商缓存   │
              │长期强缓存    │ │no-cache      │
              └──────────────┘ └──────────────┘

5.3 实际配置示例

# Nginx 完整配置示例

server {
    listen 80;
    server_name example.com;
    
    root /var/www/html;
    
    # HTML - 协商缓存
    location ~* \.html$ {
        add_header Cache-Control "no-cache";
        etag on;
    }
    
    # 带 hash 的静态资源 - 强缓存
    location ~* \.(js|css)$ {
        if ($uri ~* "\.[a-f0-9]{8,}\.") {
            add_header Cache-Control "public, max-age=31536000, immutable";
        }
    }
    
    # 图片 - 强缓存
    location ~* \.(jpg|jpeg|png|gif|ico|webp)$ {
        add_header Cache-Control "public, max-age=86400";
    }
    
    # 字体 - 强缓存
    location ~* \.(woff|woff2|ttf|eot)$ {
        add_header Cache-Control "public, max-age=31536000";
    }
}

六、缓存位置详解

6.1 Service Worker Cache

// 注册 Service Worker
if ('serviceWorker' in navigator) {
  navigator.serviceWorker.register('/sw.js')
}

// sw.js - Service Worker 缓存策略
const CACHE_NAME = 'v1'
const urlsToCache = [
  '/',
  '/styles/main.css',
  '/scripts/app.js'
]

// 安装事件 - 缓存资源
self.addEventListener('install', event => {
  event.waitUntil(
    caches.open(CACHE_NAME)
      .then(cache => cache.addAll(urlsToCache))
  )
})

// 请求拦截 - 缓存优先策略
self.addEventListener('fetch', event => {
  event.respondWith(
    caches.match(event.request)
      .then(response => {
        if (response) {
          return response // 返回缓存
        }
        return fetch(event.request) // 网络请求
      })
  )
})

6.2 Memory Cache vs Disk Cache

内存缓存（Memory Cache）：
├── 快速读取
├── 容量小
├── 关闭 Tab 即失效
└── 适合小文件、频繁访问的资源

硬盘缓存（Disk Cache）：
├── 读取较慢
├── 容量大
├── 持久化存储
└── 适合大文件

// 浏览器自动判断存储位置
// 通常基于以下因素：
// 1. 文件大小
// 2. 访问频率
// 3. 内存占用情况

7.1 版本号控制

<link rel="stylesheet" href="style.css?v=1.0.0">

7.2 文件名 Hash

// Webpack 配置
module.exports = {

  }

// 生成的文件

// app.i9j0k1l2m3n4o5p6.js

7.3 强缓存 + 协商缓存组合

# 先检查强缓存，过期后走协商缓存
location /static/ {
    add_header Cache-Control "public, max-age=86400";
    etag on;
    last_modified on;
}

八、缓存相关 HTTP 状态码

状态码 说明 含义
200 OK 正常响应 从服务器获取资源
200 (from disk cache) 硬盘缓存 从硬盘读取缓存
200 (from memory cache) 内存缓存 从内存读取缓存
304 Not Modified 协商缓存有效 使用本地缓存

九、调试与验证

9.1 Chrome DevTools

  1. 打开 Network 面板
  2. 禁用缓存：Disable cache
  3. 查看响应头：Response Headers
  4. 查看缓存状态：Size 列显示缓存来源

9.2 常见问题排查

问题：强缓存不生效
├── 检查是否勾选 Disable cache
├── 检查 Cache-Control 是否正确
└── 检查是否在隐私模式

问题：协商缓存返回 200 而不是 304
├── 检查请求头是否携带验证字段
├── 检查服务端是否正确处理
└── 检查资源是否真的未修改

十、总结

10.1 缓存优先级

强缓存 > 协商缓存 > 请求服务器

Cache-Control > Expires
ETag > Last-Modified

10.2 最佳实践

  1. 静态资源：文件名加 hash，设置长期强缓存
  2. HTML：设置协商缓存，确保获取最新版本
  3. API：根据业务场景选择合适的缓存策略
  4. 图片/字体：设置合理的强缓存时间

10.3 性能收益

强缓存：0 RTT（往返时间）
协商缓存：1 RTT
无缓存：2+ RTT（DNS、TCP、TLS、请求）
