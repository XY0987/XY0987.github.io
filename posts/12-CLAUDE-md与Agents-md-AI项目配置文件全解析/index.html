CLAUDE.md与Agents.md：AI项目配置文件全解析

发表于2026|更新于2026|AI相关
|字数总计:4.3k|阅读时长:15分钟

前言：当你用 Cursor、Claude Code、CodeBuddy 等 AI IDE 写代码时，有没有注意到项目根目录下多出了一些 .md 文件？CLAUDE.md、Agents.md……它们到底是什么？有什么区别？怎么写才能让 AI 真正”听话”？

为什么需要这些文件

AI 编程的核心矛盾

传统编程：
  程序员写代码 → 计算机精确执行 → 确定性结果
  特点：输入确定，输出确定

AI 编程：
  程序员描述意图 → AI 理解意图 → AI 生成代码
  特点：同一句话，不同时刻可能生成不同的代码

这引出了一个关键问题：如何让 AI 在你的项目里保持一致的行为？

想象你新招了一个实习生：

  • 第一天他不知道你们的代码规范是什么
  • 不知道项目用的是 Vue 还是 React
  • 不知道 API 命名是 camelCase 还是 snake_case
  • 不知道哪些文件不能动

你会怎么做？写一份项目须知给他看。

这些 .md 配置文件，就是**写给 AI 的”项目须知”**。

没有配置文件 vs 有配置文件

没有配置文件：
  你："帮我写一个 API 接口"
  AI：（用 Express + JavaScript + var + 随机风格）
  你："不对，我们用 Koa + TypeScript……"
  AI：（改了，但命名又不对了）
  你："命名用 camelCase！"
  → 反复纠正，效率极低

有配置文件：
  CLAUDE.md / .cursor/rules 中已声明：
  - 技术栈：Koa + TypeScript
  - 命名规范：camelCase
  - 错误处理：统一用 AppError 类

  你："帮我写一个 API 接口"
  AI：（直接用 Koa + TypeScript + camelCase + AppError）
  → 一步到位

逐一解析：每个文件到底干什么

1. CLAUDE.md —— Claude Code 的”记忆文件”

CLAUDE.md 是 Anthropic 的 Claude Code（CLI 工具）专用的配置文件。当 Claude Code 启动时，会自动读取项目根目录下的 CLAUDE.md 作为上下文。

核心定位：Claude Code 的项目记忆——告诉 Claude “这个项目是什么、怎么做、有什么规矩”。

层级机制：

~/.claude/CLAUDE.md              ← 全局级（User）：所有项目生效，个人偏好
├── /project/CLAUDE.md           ← 项目级（Project）：当前项目生效，团队共享
└── /project/.claude/CLAUDE.md   ← 项目级（Project）：同上，放在 .claude/ 目录下也可以

Claude Code 会自动合并全局级和项目级的 CLAUDE.md，项目级优先级高于全局级。

标准写法：

# CLAUDE.md

## 项目概述
这是一个基于 Next.js 14 的电商平台，使用 App Router + Server Components。

## 技术栈
- 框架：Next.js 14（App Router）
- 语言：TypeScript（严格模式）
- 样式：Tailwind CSS + shadcn/ui
- 数据库：PostgreSQL + Prisma ORM
- 状态管理：Zustand
- 测试：Vitest + Playwright

## 代码规范
- 使用函数式组件，禁止 class 组件
- 组件文件使用 PascalCase，工具函数使用 camelCase
- 每个组件文件不超过 200 行，超过则拆分
- 使用 `type` 而非 `interface`（除非需要 extends）
- 错误处理统一使用自定义 AppError 类

## 项目结构

src/
├── app/           # Next.js App Router 页面
├── components/    # 可复用组件
│   ├── ui/        # 基础 UI 组件（shadcn）
│   └── features/  # 业务组件
├── lib/           # 工具函数和配置
├── hooks/         # 自定义 Hooks
├── types/         # TypeScript 类型定义
└── services/      # API 调用层

## 常用命令
- `pnpm dev` —— 启动开发服务器
- `pnpm build` —— 生产构建
- `pnpm test` —— 运行测试
- `pnpm lint` —— ESLint 检查
- `pnpm db:migrate` —— 数据库迁移

## 注意事项
- 数据库操作只能在 Server Components 或 API Routes 中进行
- 所有 API 响应格式统一为 `{ code, data, message }`
- 敏感信息存放在 `.env.local`，绝对不要硬编码
- 提交前必须通过 lint 和 type-check

2. .cursor/rules —— Cursor 的”项目规则”

.cursor/rules 是 Cursor IDE 专属的规则目录，放在项目的 .cursor/ 目录下。它定义了 AI 在 Cursor 中生成代码时应遵守的规则。（注：早期 Cursor 使用根目录下的 .cursorrules 文件，现已迁移到 .cursor/rules 目录。）

核心定位：告诉 Cursor AI “你在这个项目里应该怎么表现”。

与 CLAUDE.md 的关键区别：

维度 CLAUDE.md .cursor/rules
所属工具 Claude Code (CLI) Cursor IDE
格式 自由 Markdown Markdown 规则文件
层级 全局/项目两级 项目级（.cursor/rules 目录）
侧重点 项目全景（概述+规范+命令+知识） 行为规则（怎么生成代码）
读取时机 Claude Code 启动时 Cursor 每次对话时

标准写法：

# .cursor/rules/project.md

You are an expert in TypeScript, React, Next.js App Router, and Tailwind CSS.

## Code Style
- Write concise, technical TypeScript code with accurate examples.
- Use functional and declarative programming patterns; avoid classes.
- Prefer iteration and modularization over code duplication.
- Use descriptive variable names with auxiliary verbs (e.g., isLoading, hasError).

## Naming Conventions
- Files: kebab-case (e.g., `user-profile.tsx`)
- Components: PascalCase (e.g., `UserProfile`)
- Functions/variables: camelCase
- Constants: UPPER_SNAKE_CASE
- Types: PascalCase with `T` prefix for generics (e.g., `TUserProps`)

## TypeScript Usage
- Use TypeScript for all code; prefer `type` over `interface`.
- Avoid `any`; use `unknown` for truly unknown types.
- Use `satisfies` operator for type checking when applicable.
- Prefer `const` assertions for literal types.

## Error Handling
- Use custom error classes extending AppError.
- Implement proper error boundaries in React components.
- Always handle Promise rejections.
- Log errors with structured format: { error, context, timestamp }.

## Do NOT
- Do NOT use `var` keyword.
- Do NOT use default exports (use named exports).
- Do NOT write inline styles.
- Do NOT ignore TypeScript errors with @ts-ignore.

注意：.cursor/rules 中的规则内容倾向于指令式——直接告诉 AI 该做什么、不做什么。而 CLAUDE.md 更偏描述式——介绍项目背景和规范。

3. Agents.md —— 给编码 Agent 的”项目须知”

Agents.md 是一个开放社区标准（由 Agentic AI Foundation 在 Linux Foundation 下维护），目前已被 60k+ 开源项目采用。它的定位是”给 AI Agent 看的 README”——补充 README.md 中不包含的、AI 编码时需要知道的信息。

核心定位：告诉编码 Agent “这个项目怎么构建、怎么测试、有什么代码约定”。

支持的工具（主流工具已广泛支持）：

  • Cursor、GitHub Copilot、OpenAI Codex、Google Jules
  • JetBrains Junie、Gemini CLI、Windsurf、Cline/Roo Code
  • Aider、Devin、Amp、Kilo Code 等

适用场景：

  • 希望一个文件兼容多个 AI 编码工具（不被某个工具绑定）
  • 需要告诉 Agent 构建命令、测试命令、代码约定等
  • 多 Agent 协作时定义角色和分工

标准写法：

# Agents.md

## 团队结构

### 🏗️ Architect（架构师）
- **职责**：技术方案设计、架构决策、技术选型
- **权限**：可以修改 `/docs/architecture/` 下的文档
- **限制**：不直接写业务代码
- **输入**：需求文档、技术约束
- **输出**：架构设计文档、技术方案 ADR

### 💻 Developer（开发者）
- **职责**：功能实现、代码编写、单元测试
- **权限**：可以修改 `/src/` 和 `/tests/` 目录
- **限制**：不修改 CI/CD 配置、不修改数据库 Schema
- **输入**：架构方案、功能需求
- **输出**：代码实现、单元测试

### 🔍 Reviewer（审查员）
- **职责**：代码审查、质量把关、安全检查
- **权限**：只读所有代码，可以添加评论
- **限制**：不直接修改代码
- **输入**：Pull Request、代码变更
- **输出**：审查意见、改进建议

### 🧪 QA（测试工程师）
- **职责**：集成测试、E2E测试、性能测试
- **权限**：可以修改 `/tests/` 和 `/e2e/` 目录
- **限制**：不修改业务代码
- **输入**：功能实现、测试用例
- **输出**：测试报告、Bug 报告

## 协作流程

需求输入
  │
  ▼
Architect → 输出架构方案
  │
  ▼
Developer → 实现代码 + 单元测试
  │
  ▼
Reviewer → 代码审查
  │
  ├── 通过 → QA → 集成测试 → 合并
  └── 未通过 → Developer 修改 → 重新审查

## 通信规则
- Agent 之间通过结构化消息通信
- 每条消息必须包含：发送者、接收者、类型、内容
- 紧急问题可以 broadcast 给所有 Agent

CLAUDE.md       = 写给 Claude 的"入职手册"

编写指南：如何写好这些文件

通用原则

原则1：具体 > 抽象

❌ 坏的写法：
"请写出高质量的代码"

✅ 好的写法：
"函数不超过 30 行；每个函数只做一件事；
命名使用 camelCase；必须包含 JSDoc 注释"

原则2：示例 > 描述

❌ 坏的写法：
"API 返回统一格式"

✅ 好的写法：
"所有 API 返回格式如下：
```json
{
  "code": 0,
  "data": { "userId": 123, "name": "张三" },
  "message": "success"
}
```"

原则3：禁止项要明确

❌ 模糊的禁止：
"不要写不好的代码"

✅ 明确的禁止：
"禁止事项：
- ❌ 不使用 `any` 类型
- ❌ 不使用 `var` 声明变量
- ❌ 不使用 `// @ts-ignore`
- ❌ 不在组件中直接调用 fetch，使用 services 层"

原则4：篇幅控制

文件篇幅建议：

CLAUDE.md        100-300 行（精炼，每次启动都会加载）
.cursor/rules    50-150 行（简洁，每次对话都会注入）
Agents.md        100-200 行（清晰，角色定义不宜太复杂）

为什么篇幅很重要？ 这些文件的内容会被注入到 AI 的上下文窗口中，占用 Token。内容太多，既浪费 Token 又可能导致 AI “注意力分散”——什么都看了，什么都没记住。

CLAUDE.md 编写要点

## 必写内容（缺一不可）

1. 项目概述（一句话说清项目是什么）
2. 技术栈（框架、语言、工具链）
3. 项目结构（目录树 + 各目录职责）
4. 常用命令（dev、build、test、lint 等）
5. 核心规范（命名、错误处理、代码风格）

## 选写内容（按需添加）

6. 注意事项 / 地雷区（哪些文件不能动、哪些操作要小心）
7. 环境变量说明
8. 数据库相关信息
9. 部署流程
10. 已知问题和 workaround

关键技巧：利用 CLAUDE.md 的两级机制做分层管理。

# 全局 CLAUDE.md —— 个人通用规范（所有项目生效）
~/.claude/CLAUDE.md:
  "使用 TypeScript 严格模式，提交信息用中文，
   代码注释用英文，函数不超过 30 行"

# 项目级 CLAUDE.md —— 项目特定信息
/project/CLAUDE.md:
  "这是一个电商项目，用 Next.js + Prisma
   组件必须有 Props 类型定义
   API 响应格式统一为 { code, data, message }
   测试文件命名 *.test.ts，使用 describe/it 结构"

这种分层设计让你把通用规范放全局、项目规范放项目级，避免重复且便于维护。

.cursor/rules 编写要点

.cursor/rules 中的规则更偏命令式，建议采用”角色设定 + 规则列表”的结构：

## 推荐结构

1. 角色定义（"You are an expert in..."）
2. 代码风格规则（Do / Don't 列表）
3. 命名规范
4. 错误处理规则
5. 框架特定规则
6. 禁止事项

一个重要技巧：.cursor/rules 中的规则支持用英文写（因为底层 LLM 对英文指令的遵从度更高），但如果你的团队全是中文开发者，中文也完全可以。

Agents.md 编写要点

## 每个 Agent 必须定义

1. 角色名称和职责描述
2. 权限范围（能操作哪些文件/目录）
3. 限制条件（不能做什么）
4. 输入/输出定义
5. 与其他 Agent 的协作关系

## 常见错误

❌ Agent 角色定义太模糊：
   "Developer 负责写代码"

✅ 角色定义要具体：
   "Developer 负责根据架构方案实现功能代码，
    可修改 src/ 和 tests/ 目录，
    每个 PR 不超过 300 行变更，
    必须附带单元测试"

实战：一个完整项目的配置方案

以一个真实的 Next.js 全栈项目为例，展示如何搭配使用这些文件：

my-saas-project/
├── CLAUDE.md                    # Claude Code 项目记忆
├── Agents.md                    # Agent 协作定义（如果用 Multi-Agent）
├── .cursor/
│   └── rules/                   # Cursor 行为规则
├── .github/
│   └── copilot-instructions.md  # GitHub Copilot 指令
├── .codebuddy/
│   └── rules/
│       └── project-rules.md     # CodeBuddy 规则
├── src/
│   ├── components/
│   └── ...
└── ...

维护策略：

维护关系：

CLAUDE.md（项目概述 + 核心规范 + 命令）
      │
      │ 规则同步
      ├──────→ .cursor/rules（代码生成规则子集）
      └──────→ copilot-instructions.md（Copilot 规则子集）

更新流程：
1. 规范变更时，先更新 CLAUDE.md
2. 再同步到各工具的配置文件
3. Agents.md 仅在架构调整时更新

高频踩坑与最佳实践

踩坑1：文件太长，AI 反而表现更差

❌ 写了一个 1000 行的 CLAUDE.md
   → Claude 注意力分散，重要规则被淹没
   → Token 消耗巨大

✅ 核心规则控制在 200 行以内
   → 只保留最重要的项目概述、技术栈、命名规范、常用命令
   → 次要信息放在代码注释或 README 中

踩坑2：规则之间互相矛盾

❌ CLAUDE.md 写"使用 interface"
   .cursor/rules 写"使用 type"
   → AI 无所适从，随机选择

✅ 保持各配置文件内容一致
   → 各配置文件从同一套规范提取，确保一致性

踩坑3：只写了”做什么”，没写”不做什么”

❌ 只有正面规则：
   "使用 TypeScript"

✅ 正反结合更有效：
   "使用 TypeScript 严格模式
    ❌ 禁止使用 any
    ❌ 禁止使用 @ts-ignore
    ❌ 禁止使用 as unknown as XXX 做强制类型转换"

踩坑4：Agents.md 权限定义缺失

❌ 只定义了角色，没定义权限边界：
   "Reviewer 负责代码审查"
   → Reviewer Agent 可能直接改了你的代码

✅ 权限必须明确：
   "Reviewer 负责代码审查
    权限：只读 src/ 目录
    限制：不得直接修改任何代码文件
    输出：在 PR 评论中给出审查意见"

最佳实践清单

✅ 必须做的：
  □ 每个项目至少有一个 AI 配置文件
  □ CLAUDE.md / .cursor/rules 包含技术栈声明
  □ 包含项目目录结构说明
  □ 包含命名规范
  □ 包含禁止事项列表
  □ 使用 Git 版本管理这些文件

❌ 不要做的：
  □ 不要把敏感信息（密码、API Key）写入配置文件
  □ 不要写超过 500 行的单文件
  □ 不要在不同配置文件中写互相矛盾的规则
  □ 不要写过于抽象的规则（"写好代码"）
  □ 不要忘记更新——代码在变，配置也要跟着变

这些文件与 Rules、Skills 的关系

在前面的文章中我们讲过 Skills 和 MCP，现在把配置文件也放进来看完整的层次关系：

AI 编程配置体系全景：

┌──────────────────────────────────────────┐
│          AI IDE / Agent 应用              │
├──────────────────────────────────────────┤
│  Skills 能力层                           │  ← 垂直领域的完整能力包
│  （SKILL.md + scripts + references）      │     "碰到这类任务按这个流程做"
├──────────────────────────────────────────┤
│  配置文件层                              │  ← 项目级规则和上下文
│  （CLAUDE.md / .cursor/rules / Agents.md） │     "在这个项目里你要遵守这些规矩"

关系总结：

  • CLAUDE.md / .cursor/rules：告诉 AI”项目规矩”——规则层
  • Agents.md：告诉 AI”项目怎么构建、怎么约定”——通用指令层
  • Skills：告诉 AI”碰到这个任务该怎么做”——能力层
  • MCP：告诉 AI”怎么连接外部工具”——连接层

它们不是互相替代，而是各司其职、层层配合。

小结

AI 项目配置文件核心要点：

  1. **CLAUDE.md 是”入职手册”**——写给 Claude Code 的项目记忆，包含概述、技术栈、结构、命令、规范
  2. **.cursor/rules 是”行为守则”**——写给 Cursor 的代码生成规则，偏指令式、简洁
  3. **Agents.md 是”项目须知”**——开放社区标准，兼容 Cursor/Copilot/Codex 等 20+ 工具，定义构建、测试、代码约定
  4. 具体 > 抽象——“禁止使用 any”比”写好 TypeScript”有效 100 倍
  5. 示例 > 描述——给一个 JSON 示例比说”统一格式”清晰 10 倍
  6. 篇幅要克制——CLAUDE.md 不超过 300 行，.cursor/rules 不超过 150 行
  7. 保持一致性——各工具配置文件从同一套规范同步，避免互相矛盾
  8. 这些文件也要版本管理——随项目代码一起 Git 管理，团队共享
