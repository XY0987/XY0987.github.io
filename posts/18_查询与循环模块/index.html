查询与循环模块 — Claude Code 源码深度分析

一句话理解

query() 是 Claude Code 的心脏：它接收用户消息，不断循环”调用 LLM → 执行工具 → 把结果喂回 LLM”，直到 LLM 不再需要调用工具为止。每一次循环就是一个”回合（turn）”。

白话版运行流程

整体一句话

用户说一句话 → query() 开始跑 → LLM 回复（可能带工具调用）→ 执行工具 → 把结果喂给 LLM → LLM 再回复 → 直到 LLM 说”我做完了”→ 返回最终结果。

一个完整的例子

用户："帮我找一下 src/query.ts 里的 query 函数"

第1轮：
  → LLM 思考："我需要读文件"
  → LLM 调用 ReadFile 工具
  → 系统执行 ReadFile，返回文件内容
  → 把文件内容喂给 LLM

第2轮：
  → LLM 思考："我找到了，来回答用户"
  → LLM 输出文字回答（没有工具调用）
  → 循环结束，返回答案

三条主线

用户消息进来
    │
    ▼
query() 入口
    │
    ├─── 【预处理】压缩历史消息（防止 context 太长）
    │         ├─ 工具结果预算（大工具结果存磁盘，只保留预览）
    │         ├─ snip（剪掉不重要的旧消息）
    │         ├─ microcompact（微压缩：合并重复工具结果）
    │         ├─ context collapse（折叠旧对话）
    │         └─ autocompact（自动摘要：把整段历史变成一段总结）
    │
    ├─── 【调用 LLM】把消息发给 Claude API，流式接收回复
    │         ├─ 收到文字 → 直接 yield 给调用方
    │         ├─ 收到工具调用 → 记录下来，等流结束后执行
    │         └─ 收到错误 → 尝试恢复（fallback 模型/重试）
    │
    └─── 【执行工具 + 决定是否继续】
              ├─ 有工具调用 → 执行所有工具 → 把结果加入消息 → 继续下一轮
              └─ 没有工具调用 → 运行 Stop Hooks → 返回最终结果

什么时候循环会停下来？

停止原因 通俗解释
completed LLM 正常完成，没有更多工具要调用
aborted_streaming 用户按了 ESC（流式阶段）
aborted_tools 用户按了 ESC（工具执行阶段）
max_turns 达到最大回合数限制
stop_hook_prevented Stop Hook 脚本说”不要继续”
prompt_too_long 消息太长，压缩也救不了
model_error LLM 报错，无法恢复
blocking_limit Token 数量超过硬性上限
hook_stopped Hook 阻止了继续

什么是”压缩”？为什么需要它？

Claude 的 context window 是有限的（比如 200k tokens）。对话越来越长，消息越来越多，最终会超出限制。压缩就是在超限之前，把历史消息”缩短”：

  • 工具结果预算：大工具结果存磁盘，只保留预览（最轻量，几乎零成本）
  • snip：直接删掉不重要的旧消息（只删除低价值内容）
  • microcompact：把重复的工具结果合并（比如同一个文件读了 3 次，只保留最新的）
  • context collapse：把旧的对话折叠成摘要（保留结构，减少细节）
  • autocompact：把整段历史变成一段总结（最彻底，信息损失最大，但最省 token）

这五种压缩按”轻→重”的顺序依次尝试，能用轻的就不用重的。

什么是 Stop Hook？

Stop Hook 是用户自定义的脚本，在每次 LLM 完成回复后运行。它可以：

  • 检查：LLM 的回复是否符合要求（比如代码是否通过测试）
  • 阻止：如果不符合，返回错误，让 LLM 重新来过
  • 停止：直接叫停整个对话

类比：就像代码 review 流程——LLM 提交了代码，Stop Hook 是 CI 检查，不通过就打回去重做。

以下是源码级别的详细分析，如果你只想了解整体思路，上面的部分已经足够。

核心函数详解 — “它到底做了什么”

1. query() — 公开入口

文件: src/query.ts（第 ~200 行）

这是外部调用者（runAgent()、REPL 主线程等）使用的入口。它本身非常薄，只做一件事：命令生命周期管理。

具体做了什么:

  1. 调用 queryLoop() 执行真正的循环逻辑（yield* 透传所有消息）。
  2. 循环正常结束后（非 throw、非 .return()），遍历 consumedCommandUuids，对每个命令调用 notifyCommandLifecycle(uuid, 'completed')。
  3. 如果循环因 throw 或 .return() 退出，这段代码被跳过——这给了调用方一个”started-without-completed”的不对称信号，用于检测异常退出。

**为什么要分开 query() 和 queryLoop()**：

  • queryLoop() 是纯粹的循环逻辑，不关心命令生命周期。
  • query() 是对外接口，负责”善后”。分离使得 queryLoop() 可以独立测试。

2. queryLoop() — 核心循环

文件: src/query.ts（第 ~230 行起，贯穿全文）

这是整个系统最核心的函数，一个 async function*（异步生成器）。它实现了一个 while(true) 循环，每次迭代是一个”回合（turn）”。

关键设计：显式 State 对象

let state: State = {
  messages,           // 当前消息列表
  toolUseContext,     // 工具使用上下文（工具池、权限、abort 信号等）
  autoCompactTracking,// 压缩追踪状态
  maxOutputTokensRecoveryCount, // max_output_tokens 恢复次数
  hasAttemptedReactiveCompact,  // 是否已尝试响应式压缩
  maxOutputTokensOverride,      // 输出 token 上限覆盖
  pendingToolUseSummary,        // 上一轮工具摘要（异步生成中）
  stopHookActive,               // Stop Hook 是否活跃
  turnCount,                    // 当前回合数
  transition,                   // 上一次继续的原因（用于测试断言）
}

所有”继续”路径都是 state = { ...next }; continue，而不是递归调用。这避免了调用栈增长，使得长对话（数百轮）也不会栈溢出。

每次迭代的完整流程（按顺序）：

① 解构 state，读取本轮参数
② 启动 Skill 发现预取（异步，不阻塞）
③ yield { type: 'stream_request_start' }（通知调用方新一轮开始）
④ 初始化/递增 queryTracking（chainId + depth）
⑤ 获取 compaction boundary 之后的消息（getMessagesAfterCompactBoundary）
⑥ 应用工具结果预算（applyToolResultBudget，截断过大的工具结果）
⑦ snip 压缩（如果 HISTORY_SNIP 特性开启）
⑧ microcompact 压缩（合并重复工具结果）
⑨ context collapse（折叠旧对话，如果特性开启）
⑩ autocompact（自动摘要压缩，如果接近 token 上限）
⑪ 检查 blocking limit（如果超过硬性上限，直接返回错误）
⑫ 调用 LLM（callModel，流式接收）
⑬ 处理流式输出（yield 消息、收集 tool_use blocks）
⑭ 处理错误（fallback 模型、prompt-too-long 恢复）
⑮ 执行 post-sampling hooks（fire-and-forget）
⑯ 如果被 abort → 返回 aborted_streaming
⑰ yield 上一轮工具摘要（pendingToolUseSummary，Haiku 异步生成，在本轮 LLM 流式期间已完成）
⑱ 如果没有工具调用 → 运行 Stop Hooks → 检查 Token Budget → 返回 completed
⑲ 如果有工具调用 → 执行工具（runTools 或 StreamingToolExecutor）
⑳ 注入附件（getAttachmentMessages：队列命令）
㉑ 消费记忆预取（pendingMemoryPrefetch，如果已 settled 且未消费）
㉒ 注入 Skill 发现结果（collectSkillDiscoveryPrefetch）
㉓ 刷新工具列表（refreshTools，让新连接的 MCP 服务器可用）
㉔ 检查 maxTurns 限制
㉕ 构造 next State，继续下一轮

3. 预处理层 — 五级压缩

每轮开始时，按”轻→重”顺序依次执行：

3.0 applyToolResultBudget — 工具结果预算

做了什么: 大工具结果（超过阈值的 tool_result）持久化到磁盘（tool-results/{toolUseId}.txt），只保留 <persisted-output> 预览 + 路径。这是成本最低的压缩方式，几乎零 IO。

关键细节: 使用三态决策（mustReapply/frozen/fresh），frozen 结果永不被新替换（保护 prompt cache 稳定性）。状态持久化到 transcript，resume 时可完整重建。

3.1 snipCompact — 剪切压缩

特性门控: feature('HISTORY_SNIP')

做了什么: 直接删除历史消息中”不重要”的部分（具体策略在 snipCompact.js 中）。返回 { messages, tokensFreed, boundaryMessage }。

关键细节: snipTokensFreed 会传给 autocompact，让它的阈值检查反映 snip 已经释放的空间——否则 autocompact 会看到”stale”的 token 计数，在 snip 已经解决问题的情况下仍然触发。

3.2 microcompactMessages — 微压缩

做了什么: 合并重复的工具结果。比如同一个文件被读了 3 次，只保留最新的一次结果，删除前两次（但保留 tool_use 块，只删除 tool_result 内容）。

关键细节: 对于 CACHED_MICROCOMPACT 特性，压缩边界消息会延迟到 API 响应后才 yield，这样可以使用 API 实际报告的 cache_deleted_input_tokens 而不是客户端估算值。

3.3 contextCollapse — 上下文折叠

特性门控: feature('CONTEXT_COLLAPSE')

做了什么: 把旧的对话折叠成摘要。折叠是一种”读时投影”——折叠记录存储在独立的 collapse store 中，每次 projectView() 重放 commit log，把已折叠的消息替换为摘要。这使得折叠在跨轮次之间持久化。

关键细节: 在 autocompact 之前运行——如果折叠能把 token 数降到 autocompact 阈值以下，就不需要 autocompact（保留更细粒度的上下文）。

3.4 autoCompactIfNeeded — 自动摘要压缩

做了什么: 当 token 数接近上限时，把整段历史消息变成一段总结。这是最彻底的压缩，信息损失最大，但能把 context 从 180k tokens 压缩到 20k tokens。

关键细节:

  • 压缩成功后，tracking 状态重置（turnId 新建，turnCounter 归零）。
  • 对于 task_budget，压缩前的 context 大小会从剩余预算中扣除（因为压缩后服务器看不到完整历史，无法自己计算）。
  • compact 和 session_memory 类型的 querySource 跳过 blocking limit 检查——这些是压缩 agent 本身，如果被阻塞会死锁。

4. LLM 调用层 — 流式处理

核心调用: deps.callModel(...) → queryModelWithStreaming()

流式处理逻辑（for await (const message of deps.callModel(...))）：

收到 message:
├─ type === 'assistant':
│   ├─ 提取 tool_use blocks → 加入 toolUseBlocks
│   ├─ 如果有 StreamingToolExecutor → 立即 addTool()（并发执行开始）
│   ├─ backfillObservableInput（为 SDK 流输出补充可观察字段）
│   └─ 检查是否需要 withhold（prompt-too-long / max-output-tokens）
│       ├─ 需要 withhold → 不 yield，但仍 push 到 assistantMessages
│       └─ 不需要 → yield yieldMessage
├─ 其他类型（progress、stream_event 等）→ 直接 yield
└─ 如果 StreamingToolExecutor 有完成的结果 → yield result.message

Fallback 模型机制：

  • 如果 callModel 抛出 FallbackTriggeredError，切换到 fallbackModel 重试。
  • 重试前：yield tombstone 消息（清理孤立的 assistant 消息）、清空 assistantMessages/toolResults/toolUseBlocks、重建 StreamingToolExecutor。
  • 如果 thinking blocks 存在（model-bound），在切换到 fallback 前 strip 掉（防止 API 400 错误）。

Streaming Fallback（与 Fallback 模型不同）：

  • 如果流式传输中途失败并触发 fallback，streamingFallbackOccured = true。
  • 后续收到的消息会先 yield tombstone 清理孤立消息，然后重新开始。

5. 工具执行层

5.1 StreamingToolExecutor — 流式并发执行

特性门控: config.gates.streamingToolExecution（statsig gate）

核心优势: 在 LLM 还在流式输出时，就开始并发执行已经收到的工具调用。不需要等 LLM 完全停止输出才开始执行工具。

工作流程:

  1. LLM 流式输出 tool_use block → streamingToolExecutor.addTool(toolBlock, message) 立即开始执行。
  2. 流式输出继续（LLM 可能还在输出其他工具调用）。
  3. 每次循环检查 streamingToolExecutor.getCompletedResults() → 已完成的工具结果立即 yield。
  4. 流结束后，streamingToolExecutor.getRemainingResults() 收集剩余结果。

abort 处理: 如果 abort 信号触发，executor 为所有排队/进行中的工具生成合成的 tool_result（内容为错误信息），确保 tool_use 和 tool_result 始终配对。

5.2 runTools() — 批量工具执行（非流式路径）

文件: src/services/tools/toolOrchestration.ts

当 streamingToolExecution 特性关闭时，等 LLM 完全停止输出后，批量执行所有工具调用。

5.3 附件注入

工具执行完成后，还会注入额外的”附件”消息：

getAttachmentMessages()（在 query.ts 中调用时 input=null）:
└─ 队列命令（queuedCommandsSnapshot）
    ├─ 主线程：drain agentId === undefined 的命令
    └─ 子代理：只 drain 自己的 task-notification

（记忆预取和 Skill 发现在 query.ts 中单独处理，不在 getAttachmentMessages 内部）
├─ 记忆预取（pendingMemoryPrefetch）
│   └─ 如果已 settled 且未消费 → 注入相关记忆
└─ Skill 发现（collectSkillDiscoveryPrefetch）
    └─ 注入发现的 Skill 内容

关键细节: 队列命令的 drain 策略：

  • 如果本轮运行了 SleepTool → drain 'later' 优先级（包括延迟通知）
  • 否则 → 只 drain 'next' 优先级（立即通知）
  • Slash 命令永远不在这里 drain（必须通过 useQueueProcessor 处理）

6. handleStopHooks() — Stop Hook 处理器

文件: src/query/stopHooks.ts

每次 LLM 完成回复（没有工具调用）后运行。这是一个 async function*（异步生成器），返回 StopHookResult。

执行顺序:

① 保存 cacheSafeParams（供 /btw 命令和 side_question 使用，仅 repl_main_thread/sdk）
② 模板 Job 分类（如果在 Job 环境中运行，仅 repl_main_thread 且非子代理）
③ 后台任务（非 bare 模式）：
   ├─ executePromptSuggestion（提示词建议，fire-and-forget，env 未禁用时）
   ├─ executeExtractMemories（记忆提取，fire-and-forget，仅主线程且 isExtractModeActive()）
   └─ executeAutoDream（自动梦境，fire-and-forget，仅主线程）
④ Chicago MCP 清理（computerUse 锁释放，仅主线程）
⑤ 执行 Stop Hooks（用户自定义脚本）：
   ├─ 收集 progress 消息 → yield
   ├─ 收集 blocking errors → yield + 加入 blockingErrors 列表
   ├─ 检查 preventContinuation → yield hook_stopped_continuation
   └─ 检查 abort 信号 → 立即返回
⑥ 如果是 Teammate：
   ├─ 执行 TaskCompleted Hooks（对所有 in_progress 任务）
   └─ 执行 TeammateIdle Hooks
⑦ 返回 { blockingErrors, preventContinuation }

三种 Hook 类型:

Hook 类型 触发时机 典型用途
Stop 每次 LLM 完成回复 运行测试、检查代码质量
TaskCompleted Teammate 完成 in_progress 任务 验证任务完成质量
TeammateIdle Teammate 进入空闲状态 通知 Leader “我空了”

阻塞错误处理:

  • Stop Hook 返回非零退出码 → 创建 userMessage（isMeta: true，对用户隐藏）→ 加入 blockingErrors。
  • queryLoop 收到 blockingErrors → 把错误消息加入消息列表 → 继续下一轮（让 LLM 看到错误并修正）。
  • 最多循环直到 LLM 修正或用户中断。

7. checkTokenBudget() — Token 预算检查

文件: src/query/tokenBudget.ts

特性门控: feature('TOKEN_BUDGET')

目的: 防止 LLM 在一个回合内无限输出（比如写了 100k tokens 的代码但质量越来越差）。

工作原理:

  1. 追踪当前回合已输出的 token 数（globalTurnTokens）。
  2. 如果未出现递减收益 且 **未达到预算的 90%**（COMPLETION_THRESHOLD）→ 返回 continue，并在 query.ts 中注入一条 nudge 消息（isMeta: true，对用户隐藏）。
  3. 如果出现递减收益（连续 3 次检查，每次新增 token < 500）→ 提前停止。
  4. 如果 agentId 存在（子代理）或 budget 为 null/0 → 直接返回 stop（跳过 token budget 检查，正常完成）。

递减收益检测:

isDiminishing = continuationCount >= 3
             && deltaSinceLastCheck < 500
             && lastDeltaTokens < 500

8. buildQueryConfig() — 不可变配置快照

文件: src/query/config.ts

目的: 在 query() 入口处一次性快照所有运行时配置，避免在循环中多次读取（可能因 statsig 更新而得到不一致的值）。

包含内容:

{
  sessionId,           // 当前会话 ID
  gates: {
    streamingToolExecution,  // 是否启用流式工具执行（statsig）
    emitToolUseSummaries,    // 是否生成工具摘要（env）
    isAnt,                   // 是否是 Anthropic 内部用户（env）
    fastModeEnabled,         // 是否启用快速模式（env）
  }
}

刻意排除: feature() 门控——这些是 bun:bundle 的 tree-shaking 边界，必须内联在使用处，不能快照。

9. productionDeps() — 依赖注入

文件: src/query/deps.ts

目的: 把 queryLoop() 的 I/O 依赖（callModel、压缩函数、uuid）抽象为可注入的接口，方便测试时替换为 fake。

type QueryDeps = {
  callModel: typeof queryModelWithStreaming  // LLM 调用
  microcompact: typeof microcompactMessages  // 微压缩
  autocompact: typeof autoCompactIfNeeded    // 自动压缩
  uuid: () => string                         // UUID 生成
}

测试价值: 这 4 个依赖在 6-8 个测试文件中被 spy，抽象为接口后，测试可以直接传入 fake，而不需要 jest.spyOn(module, 'fn') 的模块级 mock。

设计模式总结

模式 实现细节 为什么这样设计
显式 State 循环（非递归） while(true) + let state: State，所有”继续”路径是 state = next; continue。 避免调用栈增长，支持数百轮对话不栈溢出。transition 字段记录继续原因，方便测试断言。
异步生成器（AsyncGenerator） async function*，通过 yield 实时推送消息给调用方。 调用方（UI、子代理）可以实时接收流式消息，而不需要等整个循环结束。
五级压缩（轻→重） 工具结果预算 → snip → microcompact → collapse → autocompact，按顺序依次尝试。 能用轻的就不用重的，最大程度保留上下文细节。每级压缩都有独立的特性门控，可以独立开关。
流式工具并发执行 StreamingToolExecutor 在 LLM 流式输出时并发执行工具。 减少等待时间：不需要等 LLM 完全停止输出才开始执行工具，两者可以重叠。
Withhold + 恢复 对 prompt-too-long 和 max-output-tokens 错误，先 withhold（不 yield），尝试恢复，恢复失败才 yield。 防止 SDK 消费者（cowork/desktop）看到中间错误后终止会话，而实际上循环可以自动恢复。
依赖注入（QueryDeps） 4 个核心 I/O 依赖通过 deps 参数注入，生产代码用 productionDeps()，测试用 fake。 避免 6-8 个测试文件的 jest.spyOn 模块级 mock，测试更简洁、更可靠。
配置快照（QueryConfig） 在 query() 入口处一次性快照所有运行时配置。 防止循环中多次读取 statsig 得到不一致的值（statsig 可能在循环中更新）。
命令生命周期不对称 正常退出 → completed；异常退出（throw/.return()）→ 跳过。 给调用方一个”started-without-completed”信号，用于检测异常退出，无需额外的错误传播机制。
Token 预算递减收益检测 连续 3 次检查，每次新增 token < 500 → 提前停止。 防止 LLM 在预算内无限输出低质量内容（边际效益递减时主动停止）。
Stop Hook 阻塞重试 Stop Hook 返回错误 → 把错误加入消息 → 继续下一轮（让 LLM 修正）。 实现”LLM 自我纠错”循环：Hook 检测到问题 → LLM 看到错误 → LLM 修正 → Hook 再次检查。

对 Agent 开发的参考价值

1. 循环设计

  • 显式 State 而非递归: 长对话用 while(true) + state = next; continue 而不是递归，避免栈溢出。transition 字段记录每次继续的原因，是调试和测试的利器。
  • 异步生成器: 用 async function* 实现流式推送，调用方可以实时处理消息，而不需要等整个循环结束。

2. 压缩策略

  • 分级压缩: 按”信息损失从小到大”的顺序尝试（工具结果预算 → snip → microcompact → collapse → autocompact），能用轻的就不用重的。
  • snipTokensFreed 传递: 压缩释放的 token 数要传给下一级压缩，否则会出现”已经够用了但还是触发了重压缩”的问题。
  • 特殊 querySource 跳过 blocking limit: 压缩 agent 本身不能被 blocking limit 阻塞，否则会死锁。

3. 错误恢复

  • Withhold 模式: 对可恢复的错误，先不 yield，尝试恢复，恢复失败才 yield。这对 SDK 消费者非常重要——它们通常在看到任何错误后就终止会话。
  • 多级恢复: prompt-too-long 有 3 级恢复（collapse drain → reactive compact → 放弃）；max-output-tokens 有 2 级恢复（escalate → recovery message × 3）。

4. 工具执行

  • 流式并发: 在 LLM 还在输出时就开始执行工具，减少等待时间。
  • abort 时的合成 tool_result: abort 后必须为所有未完成的 tool_use 生成合成的 tool_result，否则 API 会报错（tool_use 和 tool_result 必须配对）。

5. 可测试性

  • 依赖注入: 把 I/O 依赖抽象为接口，测试时注入 fake，避免模块级 mock。
  • 配置快照: 把运行时配置快照为不可变对象，测试时可以精确控制每个 gate 的值。
  • transition 字段: 记录每次继续的原因，测试可以断言”走了哪条恢复路径”而不需要检查消息内容。
