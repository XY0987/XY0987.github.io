HTTP状态码详解：常见错误码及其含义

发表于2025|更新于2026|基础知识
|字数总计:3.1k|阅读时长:14分钟

前言

HTTP 状态码是服务器对客户端请求的响应状态标识，理解这些状态码对于 Web 开发、调试和运维都至关重要。本文将系统性地介绍 HTTP 状态码的分类和常见状态码的含义、原因及解决方案。

一、状态码分类

HTTP 状态码由三位数字组成，第一位数字表示响应类型：

类别 状态码范围 含义
1xx 100-199 信息性响应，表示请求已接收，继续处理
2xx 200-299 成功，请求已成功接收、理解、接受
3xx 300-399 重定向，需要进一步操作以完成请求
4xx 400-499 客户端错误，请求包含语法错误或无法完成
5xx 500-599 服务器错误，服务器无法完成有效请求 
状态码分类记忆：
- 1xx: 信息（Informational）- "请稍等"
- 2xx: 成功（Success）- "成功了"
- 3xx: 重定向（Redirection）- "去别的地方"
- 4xx: 客户端错误（Client Error）- "你错了"
- 5xx: 服务器错误（Server Error）- "我错了"

二、1xx 信息性响应

2.1 100 Continue

// 含义：客户端应继续发送请求体
// 场景：客户端发送 Expect: 100-continue 头时

// 客户端请求
POST /upload HTTP/1.1
Host: example.com
Expect: 100-continue
Content-Length: 10000

// 服务端响应（允许继续发送）
HTTP/1.1 100 Continue

// 客户端继续发送请求体
// 服务端最终响应
HTTP/1.1 200 OK

2.2 101 Switching Protocols

// 含义：服务器同意切换协议
// 场景：WebSocket 连接建立

// 客户端请求
GET /ws HTTP/1.1
Host: example.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==

// 服务端响应
HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=

三、2xx 成功响应

3.1 200 OK

// 含义：请求成功
// 场景：最常见的成功响应

// GET 请求：返回请求的资源
HTTP/1.1 200 OK
Content-Type: application/json

{
  "id": 1,
  "name": "张三"
}

// POST 请求：返回创建结果
HTTP/1.1 200 OK

{
  "success": true,
  "message": "创建成功"
}

3.2 201 Created

// 含义：创建成功
// 场景：POST 请求创建新资源

HTTP/1.1 201 Created
Location: /users/123
Content-Type: application/json

{
  "id": 123,
  "name": "张三"
}

// 最佳实践：创建资源后返回 201 而非 200

3.3 204 No Content

// 含义：成功但无返回内容
// 场景：DELETE 请求、PUT 更新

// 删除资源
DELETE /users/123
HTTP/1.1 204 No Content

// 更新资源
PUT /users/123
{
  "name": "李四"
}
HTTP/1.1 204 No Content

// 注意：204 响应不能有消息体

3.4 206 Partial Content

// 含义：部分内容
// 场景：断点续传、分片下载

// 客户端请求
GET /video.mp4 HTTP/1.1
Range: bytes=0-1023

// 服务端响应
HTTP/1.1 206 Partial Content
Content-Range: bytes 0-1023/10240
Content-Length: 1024
Content-Type: video/mp4

// 二进制数据...

四、3xx 重定向

4.1 301 Moved Permanently

// 含义：永久重定向
// 场景：网站迁移、URL 结构变更

HTTP/1.1 301 Moved Permanently
Location: https://new-domain.com/page

// 特点：
// - 浏览器会缓存新的 URL
// - 搜索引擎会将权重转移到新 URL
// - 后续请求直接访问新地址

// 应用示例：HTTP 转 HTTPS
app.use((req, res, next) => {
  if (req.protocol === 'http') {
    return res.redirect(301, `https://${req.headers.host}${req.url}`);
  }
  next();
});

4.2 302 Found / 307 Temporary Redirect

// 302 Found: 临时重定向（可能改变方法）
// 307 Temporary Redirect: 临时重定向（保持方法）

// 场景：临时维护页面、A/B 测试

// 302 示例
HTTP/1.1 302 Found
Location: /maintenance

// 307 示例（保持 POST 方法）
// POST /submit → 307 → POST /new-submit
HTTP/1.1 307 Temporary Redirect
Location: /new-submit

// 区别：
// - 302: 客户端可能将 POST 改为 GET
// - 307: 保证请求方法不变

4.3 304 Not Modified

// 含义：资源未修改，使用缓存
// 场景：协商缓存

// 客户端请求（带缓存验证）
GET /style.css HTTP/1.1
If-Modified-Since: Wed, 21 Oct 2015 07:28:00 GMT
If-None-Match: "abc123"

// 服务端响应（资源未变）
HTTP/1.1 304 Not Modified
ETag: "abc123"
Cache-Control: max-age=3600

// 304 响应没有消息体，节省带宽

五、4xx 客户端错误

5.1 400 Bad Request

// 含义：请求语法错误
// 原因：
// - 参数格式错误
// - JSON 解析失败
// - 缺少必填字段

// 示例响应
HTTP/1.1 400 Bad Request
Content-Type: application/json

{
  "error": "Bad Request",
  "message": "Invalid JSON syntax"
}

// 服务端处理
app.post('/api/users', (req, res) => {
  const { name, email } = req.body;
  
  if (!name || !email) {
    return res.status(400).json({
      error: 'Bad Request',
      message: 'name and email are required'
    });
  }
  
  // 处理逻辑
});

5.2 401 Unauthorized

// 含义：未认证（需要登录）
// 场景：访问需要认证的资源

HTTP/1.1 401 Unauthorized
WWW-Authenticate: Bearer realm="api"

{
  "error": "Unauthorized",
  "message": "Authentication required"
}

// 客户端处理
fetch('/api/profile', {
  headers: {
    'Authorization': 'Bearer ' + token
  }
})
.then(response => {
  if (response.status === 401) {
    // 跳转登录页
    window.location.href = '/login';
  }
});

5.3 403 Forbidden

// 含义：已认证但无权限
// 场景：权限不足

HTTP/1.1 403 Forbidden

{
  "error": "Forbidden",
  "message": "You don't have permission to access this resource"
}

// 401 vs 403
// 401: 不知道你是谁（需要登录）
// 403: 知道你是谁，但你不能访问（权限不足）

// 示例：用户访问管理员资源
app.delete('/admin/users/:id', auth, (req, res) => {
  if (req.user.role !== 'admin') {
    return res.status(403).json({
      error: 'Forbidden',
      message: 'Admin access required'
    });
  }
  
  // 删除用户
});

5.4 404 Not Found

// 含义：资源不存在
// 场景：URL 错误、资源已删除

HTTP/1.1 404 Not Found

{
  "error": "Not Found",
  "message": "The requested resource was not found"
}

// 常见原因：
// 1. URL 拼写错误
// 2. 资源已被删除
// 3. 路由配置错误

// SPA 应用处理（Vue Router 示例）
const router = createRouter({
  routes: [
    // ...其他路由
    {
      path: '/:pathMatch(.*)*',
      component: NotFound
    }
  ]
});

5.5 405 Method Not Allowed

// 含义：请求方法不允许
// 场景：使用错误的 HTTP 方法

// 客户端请求
POST /users/123  // 用户详情页只支持 GET

// 服务端响应
HTTP/1.1 405 Method Not Allowed
Allow: GET, PUT, DELETE

{
  "error": "Method Not Allowed",
  "message": "POST method is not supported for this resource"
}

5.6 408 Request Timeout

// 含义：请求超时
// 场景：服务器等待请求体超时

HTTP/1.1 408 Request Timeout

{
  "error": "Request Timeout",
  "message": "Server timed out waiting for the request"
}

// 常见原因：
// - 网络问题导致请求发送缓慢
// - 客户端发送大量数据但网络带宽不足

5.7 409 Conflict

// 含义：请求冲突
// 场景：并发修改、重复创建

HTTP/1.1 409 Conflict

{
  "error": "Conflict",
  "message": "Email already exists"
}

// 示例：并发更新检测（乐观锁）
app.put('/articles/:id', (req, res) => {
  const { version } = req.body;
  const article = getArticle(req.params.id);
  
  if (article.version !== version) {
    return res.status(409).json({
      error: 'Conflict',
      message: 'Article has been modified by another user'
    });
  }
  
  // 更新文章
});

5.8 410 Gone

// 含义：资源已永久删除
// 场景：资源曾经存在但已删除

HTTP/1.1 410 Gone

{
  "error": "Gone",
  "message": "This resource is no longer available"
}

// 404 vs 410
// 404: 资源不存在（可能从未存在）
// 410: 资源曾经存在但已永久删除

5.9 413 Payload Too Large

// 含义：请求体过大
// 场景：上传文件超过限制

HTTP/1.1 413 Payload Too Large

{
  "error": "Payload Too Large",
  "message": "Maximum file size is 10MB"
}

// Node.js 示例
app.post('/upload', (req, res) => {
  const contentLength = parseInt(req.headers['content-length']);
  const MAX_SIZE = 10 * 1024 * 1024; // 10MB
  
  if (contentLength > MAX_SIZE) {
    return res.status(413).json({
      error: 'Payload Too Large',
      message: `Maximum file size is ${MAX_SIZE / 1024 / 1024}MB`
    });
  }
  
  // 处理上传
});

5.10 429 Too Many Requests

// 含义：请求过于频繁
// 场景：API 限流

HTTP/1.1 429 Too Many Requests
Retry-After: 60

{
  "error": "Too Many Requests",
  "message": "Rate limit exceeded. Please try again later."
}

// 限流实现示例
const rateLimit = require('express-rate-limit');

const limiter = rateLimit({
  windowMs: 15 * 60 * 1000, // 15 分钟
  max: 100, // 每个 IP 最多 100 次请求
  message: {
    error: 'Too Many Requests',
    message: 'Too many requests from this IP, please try again later'
  }
});

app.use('/api/', limiter);

六、5xx 服务器错误

6.1 500 Internal Server Error

// 含义：服务器内部错误
// 场景：未捕获的异常

HTTP/1.1 500 Internal Server Error

{
  "error": "Internal Server Error",
  "message": "An unexpected error occurred"
}

// 常见原因：
// - 代码异常未捕获
// - 数据库连接失败
// - 配置错误

// 全局错误处理
app.use((err, req, res, next) => {
  console.error(err.stack);
  
  res.status(500).json({
    error: 'Internal Server Error',
    message: process.env.NODE_ENV === 'production' 
      ? 'An unexpected error occurred'
      : err.message
  });
});

6.2 502 Bad Gateway

// 含义：网关错误
// 场景：反向代理无法从上游服务器获取响应

HTTP/1.1 502 Bad Gateway

{
  "error": "Bad Gateway",
  "message": "Received an invalid response from the upstream server"
}

// 常见原因：
// - Nginx 无法连接到 Node.js 服务
// - 上游服务未启动
// - 上游服务响应超时

// Nginx 配置排查
upstream backend {
  server 127.0.0.1:3000;
  keepalive 64;
}

server {
  location /api {
    proxy_pass http://backend;
    proxy_connect_timeout 60s;
    proxy_read_timeout 60s;
  }
}

6.3 503 Service Unavailable

// 含义：服务不可用
// 场景：服务器过载、维护中

HTTP/1.1 503 Service Unavailable
Retry-After: 3600

{
  "error": "Service Unavailable",
  "message": "Server is temporarily unavailable due to maintenance"
}

// 常见原因：
// - 服务器维护
// - 流量过大
// - 资源耗尽

// 维护模式示例
app.use((req, res, next) => {
  if (process.env.MAINTENANCE_MODE === 'true') {
    return res.status(503).json({
      error: 'Service Unavailable',
      message: 'System is under maintenance. Please try again later.',
      retryAfter: 3600
    });
  }
  next();
});

6.4 504 Gateway Timeout

// 含义：网关超时
// 场景：上游服务器响应超时

HTTP/1.1 504 Gateway Timeout

{
  "error": "Gateway Timeout",
  "message": "The upstream server did not respond in time"
}

// 502 vs 504
// 502: 上游服务器返回无效响应
// 504: 上游服务器没有响应（超时）

// Nginx 配置超时时间
location /api {
  proxy_pass http://backend;
  proxy_read_timeout 300s;  # 5 分钟
  proxy_send_timeout 300s;
}

八、最佳实践

8.1 API 设计规范

// RESTful API 状态码使用规范

// 创建资源 → 201 Created
app.post('/users', (req, res) => {
  const user = createUser(req.body);
  res.status(201).json(user);
});

// 获取资源 → 200 OK
app.get('/users/:id', (req, res) => {
  const user = getUser(req.params.id);
  if (!user) {
    return res.status(404).json({ error: 'User not found' });
  }
  res.json(user);
});

// 更新资源 → 200 OK 或 204 No Content
app.put('/users/:id', (req, res) => {
  updateUser(req.params.id, req.body);
  res.status(204).end();
});

// 删除资源 → 204 No Content
app.delete('/users/:id', (req, res) => {
  deleteUser(req.params.id);
  res.status(204).end();
});

8.2 错误响应格式

// 统一的错误响应格式
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Validation failed",
    "details": [
      {
        "field": "email",
        "message": "Invalid email format"
      },
      {
        "field": "password",
        "message": "Password must be at least 8 characters"
      }
    ],
    "requestId": "req-abc123",
    "timestamp": "2025-05-12T10:30:00Z"
  }
}

// 实现
app.use((err, req, res, next) => {
  const statusCode = err.statusCode || 500;
  
  res.status(statusCode).json({
    error: {
      code: err.code || 'INTERNAL_ERROR',
      message: err.message,
      details: err.details || [],
      requestId: req.id,
      timestamp: new Date().toISOString()
    }
  });
});

总结

理解 HTTP 状态码是 Web 开发的基础技能：

  1. 2xx 表示成功，不同场景选择不同状态码（200/201/204）
  2. 3xx 表示重定向，理解 301/302/304 的区别
  3. 4xx 表示客户端错误，正确区分 401/403/404
  4. 5xx 表示服务器错误，做好错误处理和日志记录

选择正确的状态码可以让 API 更规范，帮助客户端更好地处理响应。
