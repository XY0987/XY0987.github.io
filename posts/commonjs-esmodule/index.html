JavaScript模块化详解：从AMD、CMD到CommonJS与ES Module

发表于2024|更新于2026|前端基础
|字数总计:3.5k|阅读时长:14分钟

前言：为什么需要模块化

在 JavaScript 早期,脚本通常通过 <script> 标签引入,这种方式存在诸多问题：

  • 全局作用域污染：所有变量都挂载在 window 对象上,容易造成命名冲突
  • 依赖管理混乱：脚本加载顺序必须严格控制,依赖关系难以维护
  • 代码难以复用：无法清晰地组织代码结构,难以实现按需加载

随着 Web 应用的复杂度不断提升,模块化成为解决这些问题的关键。本文将系统梳理 JavaScript 模块化的发展历程,深入对比各种模块化规范的差异。

一、JavaScript 模块化发展历程

1.1 无模块化时代（1995-2009）

早期的 JavaScript 代码组织方式：

// 全局变量方式
var name = 'module1'
var age = 18

// 命名空间方式（简单对象封装）
var Module1 = {
  name: 'module1',
  age: 18,
  getName: function() {
    return this.name
  }
}

缺点：外部可以直接修改内部数据,不安全。

1.2 IIFE 模式（2003-2010）

立即执行函数（IIFE）通过闭包创建了私有作用域：

var Module = (function() {
  var privateVar = '私有变量'
  
  function privateMethod() {
    console.log('私有方法')
  }
  
  return {
    publicVar: '公共变量',
    publicMethod: function() {
      console.log(privateVar) // 可以访问私有变量
      privateMethod()
    }
  }
})()

Module.publicMethod() // 可以调用
Module.privateMethod() // 报错,无法访问

优点：

  • 实现了数据私有化
  • 避免了全局污染

缺点：

  • 依赖关系不明确
  • 无法实现按需加载
  • 模块间依赖仍需手动管理

1.3 模块化规范涌现（2009-2015）

随着 Node.js 的诞生和前端工程化的发展,出现了多种模块化规范：

规范 年份 应用场景 特点
CommonJS 2009 服务端 同步加载,Node.js 默认规范
AMD 2011 浏览器 异步加载,RequireJS 推广
CMD 2011 浏览器 异步加载,SeaJS 推广
ES Module 2015 通用 静态分析,官方标准

二、CommonJS 规范详解

2.1 基本概念

CommonJS 是 Node.js 采用的模块化规范,主要用于服务端。其核心思想是通过 require 同步加载模块,通过 module.exports 导出模块。

2.2 使用示例

// 导出模块（moduleA.js）
const name = 'ModuleA'
const age = 18

function getName() {
  return name
}

// 方式1：导出对象
module.exports = {
  name,
  age,
  getName
}

// 方式2：导出单个值
module.exports = getName

// 方式3：导出值后追加属性
exports.version = '1.0.0'
// 注意：不能直接使用 exports = { ... }
// exports 是 module.exports 的引用,重新赋值会断开引用关系

// 导入模块
const moduleA = require('./moduleA.js')
console.log(moduleA.name)
console.log(moduleA.getName())

2.3 模块加载机制

CommonJS 的 require 是同步加载,这在服务端不是问题,因为文件都在本地磁盘上。但在浏览器端,同步加载会导致阻塞。

模块缓存机制：

// CommonJS 模块第一次加载会被缓存
// 后续 require 返回的是缓存的副本
const a1 = require('./moduleA')
a1.count = 1

const a2 = require('./moduleA')
console.log(a2.count) // 1,来自缓存

2.4 值拷贝特性

CommonJS 输出的是值的拷贝：

// b.js
let a = 1
let obj = { name: '原始值' }

setTimeout(() => {
  a = 100
  obj.name = '修改值'
}, 300)

module.exports = { a, obj }

// a.js
const data = require('./b')
setTimeout(() => {
  console.log(data.a)     // 1（基本类型：值拷贝）
  console.log(data.obj)   // {name: '修改值'}（引用类型：浅拷贝）
}, 2000)

三、AMD 规范详解

3.1 AMD 简介

AMD（Asynchronous Module Definition,异步模块定义）是专门为浏览器端设计的模块化规范。代表实现是 RequireJS。

核心特点：

  • 异步加载模块,不阻塞页面渲染
  • 依赖前置：在定义模块时就声明所有依赖
  • 提前执行：依赖模块在加载后立即执行

3.2 基本用法

// 定义模块（无依赖）
define(function() {
  return {
    name: 'ModuleA',
    getName: function() {
      return this.name
    }
  }
})

// 定义模块（有依赖）
define(['moduleB', 'moduleC'], function(b, c) {
  // 依赖的模块会提前加载并执行
  return {
    doSomething: function() {
      console.log(b.name, c.version)
    }
  }
})

// 加载模块
require(['moduleA', 'moduleB'], function(a, b) {
  // 回调函数在所有依赖加载完成后执行
  a.getName()
  b.doSomething()
})

3.3 配置 RequireJS

require.config({
  baseUrl: './js',  // 基础路径
  paths: {
    jquery: 'lib/jquery.min',  // 模块路径映射
    underscore: 'lib/underscore.min'
  },
  shim: {
    // 为非 AMD 模块配置导出方式
    backbone: {
      deps: ['underscore', 'jquery'],
      exports: 'Backbone'
    }
  }
})

四、CMD 规范详解

4.1 CMD 简介

CMD（Common Module Definition,通用模块定义）由国内开发者玉伯提出,代表实现是 SeaJS。

核心特点：

  • 异步加载模块
  • 依赖就近：可以在代码中随时引入依赖
  • 延迟执行：依赖模块在真正使用时才执行

4.2 基本用法

// 定义模块
define(function(require, exports, module) {
  // 依赖就近书写
  const a = require('./a')  // 同步加载
  const b = require('./b')
  
  // 异步加载
  require.async('./c', function(c) {
    c.doSomething()
  })
  
  // 导出
  exports.doSomething = function() {
    a.method()
    b.method()
  }
})

4.3 AMD vs CMD 对比

对比项 AMD (RequireJS) CMD (SeaJS)
依赖声明 依赖前置,提前声明 依赖就近,按需声明
执行时机 依赖加载后立即执行 依赖使用时才执行
代码风格 更符合编程直觉 更贴近 CommonJS
性能 初始加载可能较慢 按需加载,性能更好
调试 依赖清晰,易于调试 运行时加载,调试稍难 
// AMD - 依赖前置
define(['a', 'b', 'c'], function(a, b, c) {
  // a, b, c 都已加载并执行
  a.doSomething()
})

// CMD - 依赖就近
define(function(require) {
  // 按需加载
  if (condition) {
    const a = require('a')  // 只在需要时加载
    a.doSomething()
  }
})

五、ES Module 详解

5.1 ES Module 简介

ES Module 是 ES6（2015）引入的官方模块化标准,它是 JavaScript 语言层面的模块化解决方案。

核心特点：

  • 静态结构：import/export 必须在顶层,便于静态分析和 tree-shaking
  • 值的引用：导出的是值的动态引用,而非拷贝
  • 异步加载：适合浏览器环境
  • 严格模式：自动采用严格模式

5.2 基本用法

// 导出方式
// 方式1：命名导出
export const name = 'ModuleA'
export function getName() {
  return name
}

// 方式2：批量导出
const age = 18
const version = '1.0.0'
export { age, version }

// 方式3：默认导出
export default {
  name: 'ModuleA',
  getName: function() { return this.name }
}

// 方式4：混合导出
export const version = '1.0.0'
export default { name: 'ModuleA' }

// 导入方式
// 导入命名导出
import { name, getName } from './moduleA.js'

// 导入默认导出
import moduleA from './moduleA.js'

// 混合导入
import moduleA, { name, getName } from './moduleA.js'

// 重命名
import { name as moduleName } from './moduleA.js'

// 整体导入
import * as moduleA from './moduleA.js'

// 动态导入
import('./moduleA.js').then(module => {
  module.default.getName()
})

5.3 值引用特性

ES Module 导出的是值的动态引用：

// moduleA.js
export let count = 1
export const obj = { name: '初始值' }

setTimeout(() => {
  count = 100
  obj.name = '修改值'
}, 1000)

// moduleB.js
import { count, obj } from './moduleA.js'

console.log(count)  // 1
console.log(obj.name)  // 初始值

setTimeout(() => {
  console.log(count)  // 100（值的实时引用）
  console.log(obj.name)  // 修改值
}, 2000)

注意：ES Module 导出的值是只读的,不能直接修改：

import { count } from './moduleA.js'
count = 10  // SyntaxError: "count" is read-only

// 但可以修改引用类型的内部属性
import { obj } from './moduleA.js'
obj.name = 'new name'  // 允许修改

六、CommonJS vs ES Module 深度对比

6.1 加载机制对比

对比项 CommonJS ES Module
加载方式 运行时加载 编译时静态分析
加载时机 动态加载,可在任何位置 静态导入,必须在顶层（动态导入除外）
加载性质 同步加载 异步加载
输出方式 值的拷贝 值的引用

6.2 动态导入对比

CommonJS：

// 可以在任何地方动态导入
if (condition) {
  const module = require('./module.js')
  module.doSomething()
}

// 循环导入
const modules = ['a', 'b', 'c'].map(name => require(`./${name}.js`))

ES Module：

// 静态导入必须在顶层
import module from './module.js'

// 条件导入会报错
if (condition) {
  import module from './module.js'  // SyntaxError
}

// 使用动态导入（返回 Promise）
if (condition) {
  import('./module.js').then(module => {
    module.default.doSomething()
  })
}

// 或使用 async/await
async function loadModule() {
  const module = await import('./module.js')
  return module.default
}

6.3 循环依赖处理

循环依赖是模块化中的重要问题,两者的处理方式不同：

CommonJS 的循环依赖：

// a.js
const b = require('./b')
console.log('a.js:', b.name)  // b.js: undefined（b 还未执行完成）
exports.name = 'A'

// b.js
const a = require('./a')
console.log('b.js:', a.name)  // undefined（a 未导出完成）
exports.name = 'B'

// 执行 a.js 的输出：
// b.js: undefined
// a.js: B

CommonJS 在遇到循环依赖时,返回的是已经执行部分的值。

ES Module 的循环依赖：

// a.js
import { name } from './b'
console.log('a.js:', name)  // b.js: B
export const name = 'A'

// b.js
import { name } from './a'
console.log('b.js:', name)  // a.js: A
export const name = 'B'

ES Module 通过值的动态引用,可以获取到最终的值。

6.4 this 指向

// CommonJS
console.log(this === module.exports)  // true
console.log(this === exports)         // true

// ES Module
console.log(this)  // undefined（严格模式）

6.5 模块顶层的变量

// CommonJS
console.log(__filename)  // 当前文件路径
console.log(__dirname)   // 当前目录路径
console.log(module)      // 模块对象
console.log(exports)     // 导出对象

// ES Module
// 以上变量都不存在
// 需要通过 import.meta 获取信息
console.log(import.meta.url)  // 当前模块的 URL

七、模块化最佳实践

7.1 如何选择模块化规范

服务端（Node.js）：

  • Node.js v12 之前：使用 CommonJS
  • Node.js v12+：优先使用 ES Module（需配置 package.json 的 “type”: “module”）
  • 支持混合使用：.cjs 文件使用 CommonJS,.mjs 文件使用 ES Module

浏览器端：

  • 现代浏览器：直接使用 ES Module
  • 需要兼容旧浏览器：使用构建工具（Webpack、Vite、Rollup）

同时支持两端：

// package.json
{
  "type": "module",
  "exports": {
    ".": {
      "import": "./dist/index.mjs",
      "require": "./dist/index.cjs"
    }
  }
}

7.2 Tree Shaking 优化

ES Module 的静态特性使其支持 Tree Shaking（摇树优化）,可以删除未使用的代码：

// utils.js
export function funcA() { console.log('A') }
export function funcB() { console.log('B') }
export function funcC() { console.log('C') }

// main.js
import { funcA } from './utils.js'
funcA()

// 打包后,funcB 和 funcC 会被删除

CommonJS 是动态加载,无法静态分析,因此不支持 Tree Shaking。

7.3 模块化设计原则

  1. 单一职责原则：每个模块只负责一个功能
  2. 高内聚低耦合：模块内部高度相关,模块之间依赖最小化
  3. 避免循环依赖：设计良好的模块依赖关系
  4. 合理暴露接口：只导出必要的接口,保持内部实现的私有性
// ❌ 不好的设计：导出所有内容
export default {
  publicMethod,
  privateMethod,  // 不应该暴露
  internalHelper  // 内部辅助函数
}

// ✅ 好的设计：只导出公共接口
export function publicMethod() {
  // 使用内部的私有方法
  privateMethod()
}

function privateMethod() {
  // 内部实现,不导出
}

八、模块化发展总结

8.1 演进历程

全局变量/命名空间 → IIFE → CommonJS/AMD/CMD → ES Module
    (无规范)        (闭包)    (社区规范)      (官方标准)

8.2 各规范适用场景总结

规范 适用场景 状态
IIFE 简单脚本封装 已淘汰
AMD 浏览器端异步加载 逐渐淘汰（被 ES Module 取代）
CMD 浏览器端异步加载 已淘汰
CommonJS Node.js 服务端 仍广泛使用
ES Module 前后端通用 现代标准

8.3 未来趋势

  • ES Module 成为主流：随着浏览器和 Node.js 的支持,ES Module 已成为现代 JavaScript 的标准
  • 构建工具优化：Webpack、Vite 等工具充分利用 ES Module 的静态特性进行优化
  • 兼容性处理：通过构建工具和转译器（Babel）,可以无缝使用 ES Module 同时兼容旧环境

九、实战示例：模块化改造

9.1 从 IIFE 改造为 ES Module

改造前（IIFE）：

var Calculator = (function() {
  var result = 0
  
  function add(a, b) {
    result = a + b
    return result
  }
  
  function subtract(a, b) {
    result = a - b
    return result
  }
  
  return {
    add: add,
    subtract: subtract,
    getResult: function() {
      return result
    }
  }
})()

改造后（ES Module）：

// calculator.js
let result = 0

export function add(a, b) {
  result = a + b
  return result
}

export function subtract(a, b) {
  result = a - b
  return result
}

export function getResult() {
  return result
}

// main.js
import { add, getResult } from './calculator.js'

add(1, 2)
console.log(getResult())  // 3

9.2 CommonJS 改造为 ES Module

改造前（CommonJS）：

// utils.js
const formatDate = (date) => {
  return date.toISOString().split('T')[0]
}

const parseDate = (str) => {
  return new Date(str)
}

module.exports = {
  formatDate,
  parseDate
}

// main.js
const { formatDate } = require('./utils')
console.log(formatDate(new Date()))

改造后（ES Module）：

// utils.js
export const formatDate = (date) => {
  return date.toISOString().split('T')[0]
}

export const parseDate = (str) => {
  return new Date(str)
}

// main.js
import { formatDate } from './utils.js'
console.log(formatDate(new Date()))

十、总结

JavaScript 模块化经历了从无到有、从混乱到统一的发展过程：

  1. 早期阶段：全局变量、命名空间、IIFE 等非标准方案
  2. 社区规范：CommonJS（服务端）、AMD/CMD（浏览器端）
  3. 官方标准：ES Module 成为现代 JavaScript 的统一模块化方案

核心要点：

  • CommonJS 是值的拷贝,ES Module 是值的引用
  • CommonJS 是运行时加载,ES Module 是编译时静态分析
  • CommonJS 是同步加载,ES Module 是异步加载
  • ES Module 支持 Tree Shaking,更适合现代前端工程

在实际开发中,建议优先使用 ES Module,它是 JavaScript 的官方标准,具有更好的静态分析能力、更优的性能表现和更广泛的支持度。对于 Node.js 项目,新版本已经支持 ES Module,可以逐步迁移。
